# Deployment Record — kailash 2.44.1 (2026-06-22)

Single-package core **patch** release `kailash 2.44.0 → 2.44.1`. Closes the three
implementable public-reachable gaps from the `#1406` Production/Stable stub-marker
inventory (PR #1420). Each was a documented feature the code did not actually
perform (the `zero-tolerance.md` Rule 3c class). The fourth public-reachable gap
(anomaly-detector benchmark, `nodes/monitoring/performance_benchmark.py`) genuinely
needs an ML dependency and stays `track`.

## What shipped

### Fixed

- **Edge monitoring `active_count` no longer always equals the total alert count**
  (`kailash.nodes.edge.edge_monitoring_node`). `active_count` in the `get_alerts`
  action used `[a for a in alerts if active_only or True]` — the `or True` made the
  comprehension a no-op, so the field reported the full alert count regardless of
  how many alerts were active. Extracted `EdgeMonitor.is_alert_active()` (the
  cooldown-window predicate, previously inline in `get_alerts`) and computed
  `active_count` from it. `EdgeMonitor.get_alerts(active_only=True)` now also treats
  an alert with no cooldown-history entry as inactive (previously kept) — unifying
  the filter and the count on a single predicate. Unreachable on the normal alert
  path (`_check_thresholds` records the cooldown entry at the instant it appends an
  alert), so no live behaviour change for naturally-generated alerts.
- **`ImportPathValidator.validate_directory(...)` honours `include_tests`**
  (`kailash.runtime.validation`, `kailash.cli.validate_imports`). The
  `--include-tests` CLI flag was parsed but never forwarded to the validator, so
  test files were always skipped (`zero-tolerance.md` Rule 3c — documented kwarg
  with no effect). `validate_directory` now accepts `include_tests: bool = False`;
  the CLI forwards `args.include_tests`.

### Changed

- Removed a stale, misleading commented-out `max_pages` line in
  `kailash.nodes.api.rest._handle_pagination`. The inventory had called the param
  "silently ignored"; ground truth is that the cap **is** read and enforced in the
  remaining-pages fetch loop (`while pages_fetched < max_pages`). Only the dead
  comment implied otherwise. No behaviour change; a regression test now pins the cap.

### Inventory

`STUB-MARKER-INVENTORY.md` + its CI-guard baseline updated in the same commit:
`67 → 64` total markers, `gap 11 → 8`, public-reachable `4 → 1`. Pre-existing unused
imports in the touched files removed.
