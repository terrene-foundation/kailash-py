## What shipped

Single-record SQLite upsert with `conflict_on=["<field>"]` on a model whose field is not
declared unique failed with `ON CONFLICT clause does not match any PRIMARY KEY or UNIQUE
constraint`. The SQLite path already ran a `WHERE`-based existence pre-check to detect
INSERT-vs-UPDATE (SQLite has no PostgreSQL `xmax`), then discarded the result and still emitted
`INSERT ... ON CONFLICT`. The fix uses that pre-check result to emit a plain `INSERT` (row absent)
or `UPDATE ... WHERE` (row present) via a new dialect-agnostic
`SQLDialect.build_precheck_upsert_query`, dropping the `ON CONFLICT` constraint dependency on the
single-record SQLite path. Values stay parameter-bound; identity/PK columns are excluded from
`SET`. PostgreSQL keeps its atomic `ON CONFLICT` + `xmax` form unchanged. Pure DML — no runtime DDL.

Independent of connection type (reproduces on file-backed SQLite); surfaced, not caused, by the
#1502 `:memory:` shared-cache fix which made the table exist so the `ON CONFLICT` was finally reached.

## Bug-origin note (institutional)

#1508/#1518/#1519/#1520 are NOT new regressions — they are pre-existing latent defects in the
DataFlow upsert/SQLite/multi-tenant subsystem (the `conflict_on` feature landed `a96c942d1`
2025-11-02; the ON CONFLICT + tenant-injection machinery predates the 2026-03-11 monorepo move).
They surface now because the subsystem is under active repair: #1498 → #1502 → #1508 are a
layered-masking cascade (each fix made the next bug reachable — #1508's test was red the whole
time behind a `no such table` error until #1502 made the table exist), and red-teaming #1508
surfaced the adjacent untested paths (#1518 tenant, #1519 bulk). Root signal: test-coverage debt
concentrated in the upsert subsystem.
