## Implementation summary

`DurableExecutionEngine._enqueue_for_run` previously enqueued
`Task(workflow_blob=b"", ...)` because the engine had no built-in
serializer for arbitrary workflow objects, leaving cross-process
workers (subscribers to the underlying `SQLTaskQueue` running on a
separate host) unable to reconstruct the workflow without out-of-band
registry access.

Both producer surfaces (`WorkflowScheduler` and
`DurableExecutionEngine`) now route through the new shared helper at
`runtime/_workflow_blob.py::serialize_workflow_to_blob` so they emit
byte-identical JSON-encoded UTF-8 bytes for the same workflow. The
helper owns the discriminator dispatch (`zero-tolerance.md` Rule 3d:
explicit `Workflow` type + class-level `to_dict` protocol; no
duck-type fallback), the producer-boundary size cap
(`MAX_WORKFLOW_BLOB_BYTES = 8 MiB`), and the no-pickle security
rationale (`security.md` § "No arbitrary-code execution on user
input"). Multi-site drift between scheduler and durable-engine
producers is structurally impossible — both go through one helper.

Workers reconstruct via:

```python
from kailash.workflow.graph import Workflow
workflow = Workflow.from_dict(json.loads(blob.decode("utf-8")))
```

**Additive contract**: workers using the prior local-registry
convention keep working (they ignore `workflow_blob`); workers needing
reconstruction-from-blob now have a documented JSON shape to parse.

**Side-benefit**: the W6 `SQLTaskQueueDispatcher(classification_policy=…)`
redaction consumer's `_redact_workflow_blob` path now actually fires
for engine-dispatched tasks. Pre-fix the empty `b""` payload made
`json.loads("")` raise inside the redaction helper (caught non-fatally,
but redaction never ran on engine paths). Post-fix the consumer sees
populated JSON and the documented redaction behavior fires the same
way it already fired for scheduler-dispatched paths.
