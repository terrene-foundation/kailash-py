# 2026-05-06 — kailash-dataflow 2.7.9

## Root cause

Pre-fix, `TransactionManager._get_adapter` returned a `_PoolWrapper` around the long-lived `_connection_manager._adapter`. That adapter's asyncpg pool was created inside `async_safe_run`'s short-lived worker-thread event loop during `DataFlow.__init__`; the worker loop closed at return. Every subsequent `db.transactions.transaction()` call from a different event loop hit `RuntimeError: Event loop is closed` on `pool.acquire()`. The `db.express.*` path was unaffected because it already routed through `AsyncSQLDatabaseNode._get_adapter()`'s per-loop priority chain.

## Fix shape

1. **Phase 1 — `TransactionManager._get_adapter` re-routing (`packages/kailash-dataflow/src/dataflow/features/transactions.py`):**
   - Walks the same priority chain as `db.express.*`: `_get_or_create_async_sql_node(db_type)._get_adapter()` → `_shared_pools` → runtime pool → `_PROCESS_POOL_REGISTRY` (WeakValueDictionary, reaped on loop close) → fallback `connect()` under per-key creation locks.
   - Adapter resolution normalized via `getattr(adapter, "connection_pool", None) or getattr(adapter, "_pool", None)` because the core-SDK `ProductionPostgreSQLAdapter` returned by `_get_adapter()` exposes the pool as `_pool`, not `connection_pool`.
   - `_get_adapter_from_context` (used by `TransactionScopeNode` / `TransactionSavepointNode`) converted to `async`; both callers updated to `await`.
   - 2 unit-test files updated to mock `_get_or_create_async_sql_node` + async `_get_adapter` via `AsyncMock` and `_detect_database_type`.

2. **Phase 2 — `ConnectionManager.initialize_pool` transient reachability (`packages/kailash-dataflow/src/dataflow/utils/connection.py`):**
   - Removed `_connection_manager._adapter` retention.
   - Init-time fail-fast contract (`dataflow-pool.md` Rule 2 — `await adapter.connect()` MUST succeed before `__init__` returns) preserved exactly: opens transient adapter, verifies via `connect()`, `disconnect()`s within one `async_safe_run` call, narrow exception swallow on cleanup-on-failure.
   - Internal callers (`health_check`, `get_connection_stats`, `disconnect`, dialect-detection sites in `engine.py` / `engine.async_methods.py`) migrated to walking `_PROCESS_POOL_REGISTRY` for live pool stats or to `dialect_factory` for type-only queries.
   - `_PoolWrapper` (internal, dead branch) removed.
   - Internal API only — no `__all__` export, no spec coverage; per `zero-tolerance.md` Rule 6a internal-only carve-out, no deprecation shim required.

3. **Phase 3 — Tier-2 regression suite (`packages/kailash-dataflow/tests/regression/test_issue_835_transaction_cross_loop.py`):**
   9 tests against real PostgreSQL (`IntegrationTestSuite`) — cross-loop `begin()`, `asyncio.run()`-twice, `asyncio.gather` across two child loops, nested savepoint/rollback cross-loop, rollback path, `TransactionScope` async-cm, concurrent two-loop `begin()` contention, WeakValueDictionary reaping, pool-cap stress (50 sequential loops not breaching `max_pool_count_per_process=100`). Autouse fixture lowers `idle_timeout=2` for pytest-xdist parallelism; production defaults unchanged.

4. **Phase 4 — Spec + CHANGELOG + version bump:**
   `specs/dataflow-cache.md` §12.1 (loop-affinity contract for `db.transactions.begin()`) + §13.4 (async transaction participation in the per-loop registry) describe post-merge behavior in present tense per `specs-authority.md` Rule 5. CHANGELOG entry under `[2.7.9] — 2026-05-06`. Atomic version bump in `pyproject.toml` + `__init__.py::__version__` per `zero-tolerance.md` Rule 5.
