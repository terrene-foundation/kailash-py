# Deployment Record — kailash 2.41.1 (2026-06-20)

Single-package core patch `kailash 2.41.0 → 2.41.1` shipping two trust-plane
correctness fixes surfaced by a holistic post-multi-wave redteam of the
recently-shipped PACT KSP/Bridge governance epic (#1368–#1386) + trust-plane
surface.

## What shipped

- **#1393 (HIGH, `kailash.trust.pact`)** — YAML envelope
  `confidentiality_clearance` + `max_delegation_depth` were silently dropped on
  the #1386 YAML-apply path: `EnvelopeSpec` had no slot, `_parse_envelopes`
  never read them, `yaml_resolvers.resolve_envelope` never forwarded them into
  `ConstraintEnvelopeConfig`. A YAML-authored `max_delegation_depth: 1` silently
  became `None` (UNLIMITED) at enforcement; an authored confidentiality ceiling
  never applied — no error. Both fields are now parsed (fail-closed on invalid
  level / non-positive-int depth / unhashable list-or-dict values via a shared
  `_is_valid_level` guard routing all four clearance-level sites) and forwarded
  to the runtime config, where they participate in monotonic-tightening
  validation end-to-end (a widening child envelope → `MonotonicTighteningError`).
- **#1394 (MEDIUM, `kailash.trust.revocation.cascade`)** — `cascade_revoke`
  returned `RevocationResult(success=False, revoked_agents=[])` on a partial
  failure even when the best-effort rollback could NOT restore an
  already-soft-deleted chain, so the audit result diverged from store state
  ("no agents revoked" while chains stayed deleted). `_rollback_chains` now
  returns the agents it could not restore (no longer swallows); `revoked_agents`
  reflects store ground truth + a WARN names any chain left revoked. Docstrings
  that over-claimed transaction-backed "atomic chain invalidation" — which the
  code never performed and the InMemory transaction context cannot provide (it
  snapshots only active chains, so a soft-delete cannot be rolled back) — are
  reconciled to the real best-effort-rollback (non-transactional) contract; the
  false "uses transactions" unit test is reconciled + a pin test asserts
  `cascade_revoke` never calls `store.transaction()`.
