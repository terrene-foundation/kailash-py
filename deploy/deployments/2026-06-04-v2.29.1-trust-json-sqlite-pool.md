# Deployment — kailash 2.29.1 (#1243 canonical-JSON NaN + #1245 SQLite store leak + #1248 cross-loop pool disposal)

**Date:** 2026-06-04
**Package:** `kailash` `2.29.0 → 2.29.1` (patch — three independent trust/runtime correctness fixes)
**Issues:** #1243, #1245, #1248 — all closed by merge.
**Follow-ups filed:** #1258 (serialize_for_signing ensure_ascii — out of scope, breaking), #1260.

## What shipped

### 1. Canonical-JSON rejects NaN / Infinity / non-string keys (#1243)

The trust signing layer's `canonical_json_dumps` (delegate / SPEC-09 pre-image)
serialized `float("nan")` / `float("inf")` as the literal tokens `NaN` / `Infinity`
— invalid JSON that no other SDK's parser accepts, breaking cross-SDK byte parity
and producing a signature over a non-round-trippable pre-image. Now passes
`allow_nan=False` (raises `ValueError`) plus `_reject_non_string_keys` with cycle
detection (markers-set add-on-entry / discard-on-exit, raises on circular reference
instead of leaking a `RecursionError` outside the error taxonomy).

`serialize_for_signing` (Ed25519 / HMAC trust-plane pre-image) carried the same
`allow_nan` hole on a structurally separate code path — fixed there too in the
same PR. (The `ensure_ascii` half is correct as-is — it matches its
`trust-plane-canonical.json` fixture; the unicode-normalization change is deferred
to #1258 as breaking.)

### 2. SQLite trust stores close every per-thread connection on `close()` (#1245)

`PostureStore`, `SqliteChainStore`, and `SqliteTrustPlaneStore` each opened a
`threading.local()` connection per accessing thread but `close()` / `_sync_close()`
closed only the _calling_ thread's connection — every other thread's handle leaked
until process exit. Each store now tracks `self._all_conns: set` under
`self._conns_lock`, registers on open, and on close flips the closed-flag FIRST
(so no new connection races in) then closes all tracked connections under the lock,
best-effort. Reviewer surfaced the 3rd store (`SqliteTrustPlaneStore`) + a
flag-first close race during the PR; both fixed in-PR.

### 3. Cross-loop async-SQL pool disposal scoped to the disposing loop (#1248)

`AsyncSQLDatabaseNode.clear_shared_pools` tore down the entire `_shared_pools` map
and `_PROCESS_POOL_REGISTRY` on any loop teardown, so a `LocalRuntime` finishing on
one event loop disposed pools still in use by another loop. Now
`clear_shared_pools(cls, graceful=True, *, loop_id=None)` filters to
`k.startswith(f"{loop_id}|")` when a loop is named, and gates the process-registry
`.clear()` to `loop_id is None`. `LocalRuntime.run_in_thread` / `_cleanup_event_loop`
and `AsyncLocalRuntime` cleanup all pass `loop_id=id(loop)`.
