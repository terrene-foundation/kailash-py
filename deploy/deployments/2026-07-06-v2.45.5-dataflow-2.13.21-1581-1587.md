## What shipped

### kailash 2.45.4 â†’ 2.45.5 (`src/kailash/nodes/data/async_sql.py`)

- **#1581 â€” `AsyncSQLDatabaseNode` is transaction-scope aware.** `_AdapterTransactionScope` gained a `.transaction` property (raw driver handle); `async_run` now reads `inputs.get("transaction")` and threads it through `_execute_with_retry` â†’ `_execute_with_transaction` (and the batch path `execute_many_async` â†’ `_execute_many_with_transaction`). A new leading borrow-don't-own branch runs `adapter.execute(..., transaction=...)` with NO begin/commit/rollback (the enclosing scope owns the lifecycle). Param-threaded, never stored on the shared cached node instance, so concurrent ops can't leak one scope's transaction into another.
- **#1580 â€” uniform `adapter.transaction()` contract + pool-safe commit/rollback** (already on `main` since v2.45.4 tag, unreleased until now): PostgreSQL/MySQL/SQLite adapters expose one consistent `transaction()` async-context contract; commit/rollback return the borrowed connection to the pool exactly once (double-release + begin-orphan closed); quiet SQLite terminal close so it can't mask the driver error.

### kailash-dataflow 2.13.20 â†’ 2.13.21

- **#1581 â€” generated CRUD nodes participate in an enclosing `TransactionScopeNode`.** Pre-fix each CRUD node ran every statement on its own cached `AsyncSQLDatabaseNode` in `transaction_mode="auto"` (per-statement autocommit; the kwarg was inert). Inside a scope a CRUD write committed independently and **survived a later `TransactionRollbackNode`**. A module-level `_run_sql_in_scope` reads `active_transaction` off the workflow context and injects `scope.transaction`; all 10 single-record CRUD sites route through it (read-your-writes inside scope; discarded on rollback). **Fail-closed:** active scope with no `.transaction` handle raises `NodeExecutionError`. The PG `$11`-param-type retry now runs on the POOLED node (joins the scope) instead of a fresh node that escaped it. `TransactionSavepointNode` + `TransactionRollbackToSavepointNode` (defined + exported but never registered) are now registered; savepoint-name validation `re.match(...$)` â†’ `re.fullmatch(...)`. **Bulk CRUD nodes are NOT yet scope-aware** â€” separate 3.3k-LOC subsystem, tracked as **#1585** (an `xfail(strict=True)` pin holds the gap and self-clears when it lands).
- **#1587 â€” Express cache adapter closed on `DataFlow.close()`/`close_async()`.** `DataFlowExpress` eagerly auto-detects a cache backend; when Redis is reachable that is an `AsyncRedisCacheAdapter` owning a `ThreadPoolExecutor`. Neither close path tore it down, leaking the executor's worker threads (`ResourceWarning: AsyncRedisCacheAdapter not closed` at GC) on every documented cleanup path (FastAPI lifespan, async fixture). `AsyncRedisCacheAdapter` gains a sync `close()`; `DataFlowExpress` gains `close()`/`close_async()` closing `_cache_manager`; both `DataFlow` close paths tear down the async Express instance AND the engine-level `_cache_integration` adapter; `close_async` offloads the blocking shutdown via `asyncio.to_thread`. Pre-existing on `main`; invisible on `[dev]`-only CI (no Redis â†’ in-memory fallback â†’ no executor).
