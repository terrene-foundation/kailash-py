## kailash-align 0.7.3 — #1426 (trl 1.x compatibility)

0.7.2 could not run **any** SFT/DPO/KTO/ORPO/Online-DPO fine-tune against **any**
`trl` in its declared `>=1.0,<2.0` range — the `*Config.to_trl_config()` adapters

- trainer setup targeted the trl 0.x API. Ground-truth verified against real
  trl 1.0.0 and trl 1.6.0 (config construction; no GPU).

* **`to_trl_config()` emits trl 1.x kwargs:** SFT `max_length` (was removed
  `max_seq_length`); DPO/KTO drop removed `max_prompt_length`; GRPO/RLOO `beta`
  (was `kl_coef`) + `vllm_gpu_memory_utilization` (was `vllm_gpu_utilization`).
  GRPO/RLOO renames were a same-bug-class fix beyond the issue's enumerated list
  (autonomous-execution Rule 4).
* **`pipeline.py` peft guard:** no longer passes both a `PeftModel` and
  `peft_config` (trl 1.x raises `ValueError`); `peft_config` omitted when the
  model is already a `PeftModel`. LoRA preserved.
* **orpo / online_dpo de-registered** from `METHOD_REGISTRY` (trl >=1.0 removed
  ORPOTrainer/ORPOConfig + OnlineDPOTrainer/OnlineDPOConfig upstream — raised on
  every supported trl). `validate_method_name`/`get_method` reject them →
  redirect to dpo/grpo/sft_then_dpo. The `ORPOConfig`/`OnlineDPOConfig`
  dataclasses retained for back-compat; their `to_trl_config()` raises an
  informative `TrainingError`.
* **Trainer lazy-import guard:** a registered method whose trl trainer class was
  removed (`xpo`/`nash_md`/`ppo`/`cpo`/`bco`) now raises an informative
  `TrainingError` instead of a raw `ImportError`.
* Docs + spec reconciled: align package docs (`docs/00-authority/CLAUDE.md`,
  `docs/01-method-selection-guide.md`) + `specs/alignment-training.md` §2.5/§2.8
  - the `AlignmentConfig` docstring no longer instruct the removed methods.

Gates: reviewer (HIGH registry de-reg + MED sibling-guard, both fixed),
security-reviewer CLEAN, adversarial verification workflow `wf_062a74d9-854`
(1 must-fix = docstring drift, fixed). 246 tests pass (config + registry +
pipeline) vs trl 1.6.0.

## kailash-ml 2.2.2 — #1430 (numba floor)

A fresh (lock-ignoring) `pip install kailash-ml` on Python 3.12/3.13/3.14
backtracked to `numba 0.53.1` → `llvmlite 0.36.0` (no cp312/313/314 wheel →
source-build `RuntimeError`). `numba` is pulled transitively by
`umap-learn`/`pynndescent` (UMAPTrainable). Added `numba>=0.61` floor → forces
`llvmlite>=0.44`. Minimum version floor per `dependencies.md` (NOT a cap).
Verified: 3.12/3.13/3.14 fresh-resolve → llvmlite 0.47.0.
