## What shipped

### kailash-kaizen 2.39.0 → 2.40.0 — BREAKING CHANGE

- **#1892 — four-axis `azure_ai_foundry` wire + legacy provider path removed.** A new preset (`kaizen.llm.presets.azure_ai_foundry_preset` + `LlmDeployment.azure_ai_foundry(endpoint, api_key, model, *, api_version=None)`) targets Azure AI Foundry's unified, model-agnostic model-inference endpoint (`POST {endpoint}/models/chat/completions`) — one fixed URL serves every model deployed to the Foundry project, the model id travels in the wire body, never the URL. Wired into `kaizen.llm.deployment_resolver.resolve_deployment_for`. Authenticates via the same `api-key: <KEY>` header `AzureEntra`'s api-key variant already provides; wire protocol is the same OpenAI-compatible `WireProtocol.OpenAiChat` every other direct-provider preset speaks (tool-calling, streaming, reasoning-model kwarg filtering all apply automatically).
- **Legacy path DELETED**: `kaizen.providers.llm.azure` module, `AzureAIFoundryProvider` (built on `azure-ai-inference`), and `LLMAgentNode._legacy_provider_chat` are all removed. `kaizen.providers.registry.PROVIDERS` is now the empty dict. The `DeprecationWarning` shim for this removal shipped in kaizen 2.39.0 (one full minor cycle) per `zero-tolerance.md` Rule 6a. Migration: only environment variable names changed (`AZURE_AI_INFERENCE_ENDPOINT` → `AZURE_AI_FOUNDRY_ENDPOINT`, `AZURE_AI_INFERENCE_API_KEY` → `AZURE_AI_FOUNDRY_API_KEY`, new `AZURE_AI_FOUNDRY_DEPLOYMENT` / `AZURE_AI_FOUNDRY_API_VERSION`); no `LLMAgentNode`/`BaseAgent` code changes required. Direct standalone use of the removed `AzureAIFoundryProvider` now raises `ModuleNotFoundError`.
- Drops `azure-ai-inference` + `azure-core` from the `providers-azure` extra (sole consumer removed); `azure-identity` remains for `AzureEntra`'s workload-identity/managed-identity variants.
- Closes #1720 — `azure_ai_foundry` was the last unmigrated provider in `providers/llm/`.

### kailash-dataflow 2.18.0 → 2.19.0

- **#1846 — `FieldType.Vector(dim)` embedding column type.** A parameterized field type (`VectorFieldType` dataclass + `FieldType.Vector` staticmethod, since `FieldType` is a bare `Enum` that can't carry a dimension parameter) for embedding/pgvector `@db.model` fields. Per-dialect DDL: PostgreSQL `vector(N)` via pgvector (documented `TEXT` fallback when the extension is off), SQLite `TEXT`, MySQL `JSON`. New `encode_vector`/`decode_vector` codec emits the canonical `[a,b,c]` value literal in **fixed-decimal, never scientific notation** (so embedding magnitudes below `1e-4` round-trip byte-identically) and **fails closed** with `VectorValueError` on non-finite values (any NaN/±Inf spelling), a non-positive/non-int/out-of-range dimension, or an oversized input. New public symbols re-exported from `dataflow.core`: `FieldType.Vector`, `FieldType.VECTOR`, `VectorFieldType`, `VectorValueError`, `encode_vector`, `decode_vector`.
- Byte-pinned cross-SDK regression vectors ship at `tests/fixtures/issue_1846_vector_field_type_vectors.json` (per `cross-sdk-inspection.md` Rule 4 — the Rust SDK owns the equivalent byte-lockstep vector).
