## What shipped

**kailash 2.45.2 — core SQLite adapter RETURNING guard.** `AsyncSQLDatabaseNode`'s
SQLite path (`src/kailash/nodes/data/async_sql.py`, two DML short-circuits at the
transaction branch + `_execute_on_connection`) returned `[{"rows_affected": N}]` for
every `INSERT`/`UPDATE`/`DELETE` on a leading-keyword check, never fetching a
`RETURNING` result set. Added `and "RETURNING" not in query.upper()` to both so
RETURNING queries fall through to the fetch — parity with the PostgreSQL adapter guard
that already existed. Non-RETURNING DML unchanged.

**kailash-dataflow 2.13.5 — upsert dialect update-value binding.**
`build_upsert_query` (SQLite AND PostgreSQL) built `DO UPDATE SET col = EXCLUDED.col`,
but `EXCLUDED` is the INSERT-proposed (`create`) value, not the caller's separate
`update` payload — so on conflict the row was set to the `create` values, silently
ignoring `update`. Now binds the `update` values as parameters, continuing the
`:p<i>` sequence so `AsyncSQLDatabaseNode`'s positional→named rebuild stays aligned.
Combined with the core fix, single-record upsert now returns the upserted row instead
of `{"record": {"rows_affected": 0}}`. Also fixed two pre-existing PostgreSQL upsert
failures, verified against real Postgres (CI DataFlow Infra Regression green).
