## Workstreams shipped

### #788 — `kaizen.llm.testing.mock_preset()` test-only module

Cross-SDK parity with kailash-rs `LlmDeployment::mock()` at
`crates/kailash-kaizen/src/llm/deployment/presets.rs:1183` (gated behind
`#[cfg(any(test, feature = "test-utils"))]`). Python lacks cargo
features → structural defense is **physical module separation**:
`LlmDeployment.mock` does NOT exist on the production class,
`kaizen.llm.presets.mock_preset` does NOT exist, `"mock"` is NOT a
registered preset. Test code MUST `from kaizen.llm.testing import
mock_preset` explicitly — the module path itself is the deliberate
red-flag (`grep -rn 'kaizen.llm.testing' src/` surfaces production-side
leakage). 17 Tier-1 tests assert the structural-defense + functional
contract (PR #798).

### #790 — 7 capability matrix rows for Python-only presets

Adds `together`, `fireworks`, `openrouter`, `deepseek`, `lm_studio`,
`llama_cpp`, `docker_model_runner` to
`kaizen.llm.capabilities._PRESET_CAPABILITIES`. Previously these
fell through to `ALL_FALSE_CAPABILITIES` —
`LlmDeployment.together(...).supports()` reported uncapable for
tools/vision despite Together AI hosting tool-calling and vision-capable
models. Convention follows existing rows (`ollama` / `groq` /
`mistral`): `vision=True` means deployment surface CAN serve
vision-capable models; per-model gating is caller's responsibility.
`deepseek` is the conservative outlier (`vision=False`) because
`api.deepseek.com/v1` exposes only deepseek-chat / deepseek-coder; the
DeepSeek-VL family ships as separate weights, not served by this preset
(PR #800). 8 new tests; `_NON_EMPTY_PRESETS` parametrized sweep extended
(48/48 pass).

**Cross-SDK reconciliation owed**: kailash-rs
`CapabilityMatrix::for_preset` at
`crates/kailash-kaizen/src/llm/deployment/capabilities.rs:120-250` does
NOT yet have rows for these 7 presets; currently falls through to
`Self::all_false()`. Per `rules/upstream-issue-hygiene.md`, no
auto-cross-file — kailash-rs should land equivalent rows in a
coordinated cross-SDK release. Until then, Python `supports()` reports
the canonical matrix; Rust returns all-False for the same preset name.

### #794 — `cohere_preset` default endpoint v1 → v2

Default endpoint advanced from `https://api.cohere.com/v1` (legacy
Generate API) to `https://api.cohere.ai/v2` (modern Chat API) for
byte-equivalent cross-SDK parity with kailash-rs `LlmDeployment::cohere()`
at `crates/kailash-kaizen/src/llm/deployment/presets.rs:386-396`. Wire
tag stays `WireProtocol.CohereGenerate` — Rust comment at
`presets.rs:378-380` confirms v2 delegates through `OpenAiAdapter` under
the same wire tag (the wire tag is the adapter-routing key, NOT the
API-generation indicator). The new `LlmDeployment.cohere_from_env()`
constructor from #791 inherits the v2 default automatically (PR #799).

**Behavior change with migration path**: callers depending on the legacy
v1 Generate request envelope (different shape from v2 Chat) MUST opt in
via explicit overrides:

```python
cohere_preset(api_key="...", model="...",
              base_url="https://api.cohere.com", path_prefix="/v1")
```

Both endpoints currently coexist; Cohere has not announced a v1 sunset
date. CHANGELOG `Migration` section documents the opt-in.
