## What shipped

### Explicit SQLite datetime/date adapters (#1250)

Python 3.12 deprecated the implicit default `datetime`/`date` → SQLite adapters;
every `dataflow` SQLite write of a `datetime` / `date` value emitted a
`DeprecationWarning` (and on 3.13+ the implicit adapters are slated for removal,
which would silently change stored timestamp formatting). `dataflow/adapters/sqlite.py`
now registers explicit adapters at module import:

```python
def _register_sqlite_datetime_adapters() -> None:
    _sqlite3.register_adapter(_datetime.datetime, lambda val: val.isoformat(" "))
    _sqlite3.register_adapter(_datetime.date, lambda val: val.isoformat())
_register_sqlite_datetime_adapters()
```

**Critical format detail:** the `datetime` adapter uses `isoformat(" ")` (space
separator) — NOT the default `isoformat()` (T separator). The legacy implicit
adapter stored the space-separated form; matching it byte-for-byte keeps existing
SQLite databases readable and avoids a silent storage-format change. The regression
test `test_issue_1250_sqlite_datetime_deprecation.py` asserts the space-separator
output, and a follow-up test locks the tz-aware adapter format.
