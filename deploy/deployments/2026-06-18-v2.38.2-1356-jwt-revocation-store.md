# Deployment Record — kailash 2.38.2 (#1356 JWT pluggable revocation store)

**Date:** 2026-06-18
**Package:** kailash (core) — single-package patch
**Version:** 2.38.1 → 2.38.2
**Issue:** #1356 (MEDIUM — security control silently inert in multi-worker deployments)

## What shipped

`JWTAuthManager` token revocation was backed by a per-instance in-memory `set`,
so a token revoked on one worker stayed valid on every other worker in any
multi-worker / multi-pod deployment — the revocation security control silently
failed to take effect. The fix introduces a synchronous, pluggable
`TokenRevocationStore`:

- `verify_token` / `revoke_token` stay **synchronous** (per-request hot path; no
  API break). Revocation check moved post-decode, keyed on `jti`.
- Default `InMemoryTokenRevocationStore` preserves the original process-local
  behavior; documented as such in the class docstring.
- Injectable shared backend via `JWTAuthManager(revocation_store=...)` propagates
  revocation across every worker that shares it (Redis / DB / distributed cache).
- New exports `TokenRevocationStore` + `InMemoryTokenRevocationStore` in
  `kailash.middleware.auth`.
- Decode-failure revocation path is TTL-bounded (capped at the refresh-token
  window) so it cannot grow the store without limit.

10 regression tests (`tests/unit/middleware/test_jwt_revocation_store.py`).
