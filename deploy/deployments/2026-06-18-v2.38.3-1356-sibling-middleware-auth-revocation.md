## What shipped

`MiddlewareAuthManager` — the legacy nodes-based JWT verifier in
`kailash.middleware.auth`, exported as public API — had **no token-revocation
capability**: no `jti` claim, no revocation-store consultation in `verify_token`,
no `revoke_token` method. A token could not be invalidated before natural expiry.
Surfaced by #1356's R1 security-reviewer as the sibling gap (F1).

This release wires the SAME pluggable `TokenRevocationStore` #1356 introduced
(`kailash/middleware/auth/revocation.py`) into `MiddlewareAuthManager`:

- `create_access_token` stamps a `jti` claim (the shared-store revocation key).
- `verify_token` rejects revoked tokens (`is_revoked(jti=, token=)`, both passed)
  with `HTTPException(401, "Token has been revoked")`; `except HTTPException:
raise` propagates deliberate 401s instead of masking them.
- new `async revoke_token(token)` — jti+exp revoke; decode-failure fallback
  revokes by raw token, TTL-capped at `token_expiry_hours` (forged-exp defense).
- `__init__`: `enable_blacklist=True` (default) + injectable `revocation_store`.
  Inject a SHARED backend → revocation propagates across workers; default
  `InMemoryTokenRevocationStore` is process-local.

Purely additive (jti added; existing claims unchanged; pre-2.38.3 tokens still
verify) — hence patch, matching the #1356 precedent.

## Pre-existing bug also fixed (zero-tolerance Rule 1)

Three `security_logger.execute(severity="warning")` sites (1 new + 2 pre-existing
in `verify_token`/`verify_api_key`) passed an invalid `SeverityLevel`
(`CRITICAL`/`HIGH`/`MEDIUM`/`LOW`/`INFO`). `SecurityEventNode` raised `ValueError`,
so **every invalid/expired token escaped `verify_token` as an internal error, not
a clean 401**. Fixed to `MEDIUM` + best-effort `_emit_security_event` helper (a
logging-backend failure can no longer convert a 401 into a 500).
