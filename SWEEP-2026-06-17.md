### [MED] [Sweep 3] Open issue #1356 — JWTAuthManager revocation is process-local

- **Location:** `kailash.middleware.auth.JWTAuthManager.revoke_token()` / `.verify_token()` (core SDK).
- **Disposition:** FILE-ISSUE (already filed by user; **top next-session candidate**).
- **Evidence:** `enable_blacklist=True` by default backs the blacklist with an in-memory
  `self._blacklisted_tokens: set`; a token revoked on one worker stays valid on every
  other worker sharing the signing key until natural expiry. Repro in the issue body.
- **Why it matters:** revocation is a security control that silently no-ops in the default
  multi-worker / multi-pod production topology. MEDIUM.
- **Acceptance (from issue):** pluggable shared revocation store OR explicit docstring +
  `enable_blacklist` documentation that the built-in blacklist is process-local, plus a
  regression test pinning the chosen contract. Cross-SDK: likely mirrors in kailash-rs
  (JWT middleware) — inspect per `cross-sdk-inspection.md` when picked up.
