## 0. Conventions

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, and **OPTIONAL** in this document are to be interpreted as described in RFC 2119 and RFC 8174.

Conformance levels referenced by this spec follow `eatp/07-conformance.md`: **EATP Compatible** (with the **Compatible-Legacy** migration sub-level), **EATP Conformant**, and **EATP Complete**. Where this spec writes "Conformant level" or "Complete level" it means EATP Conformant and EATP Complete respectively; "Compatible-Legacy" is the named sub-conformance beneath EATP Compatible defined by EATP-07 for pre-upgrade records during the suite migration window.

This specification defines an **SDK binding**. It binds an already-shipped cryptographic wrapper (the SLIP-0039 Shamir wrapper realized in `kailash-py` `trust/vault/shamir.py`, Apache 2.0, Foundation-owned) to the Trust Vault surface (`trust/vault/backup.py`). It does **not** redefine the SLIP-0039 cryptography (delegated upstream to SLIP-0039 and to the ritual layer EATP-10) and it does **not** redefine the recovery ritual UX or the post-recovery posture semantics **D6** (owned by `eatp/10-shamir-recovery.md@v1.0`). The boundary is drawn explicitly in §3, mirroring the way EATP-10 §3 draws its boundary against SLIP-0039.

This specification cites the following bindings at fixed versions:

- `binding D1, D2a/b/c, D3 from eatp/08-algorithm-identifier.md@v1.0`, audit-anchor signing (`alg_id` precedence, parent's-algorithm-and-canonical-form, JCS canonical form).
- `binding D4 from eatp/09-tiered-audit-dispatcher.md@v1.0`, recovery-tier dispatch, dispatcher mediation, global-anchor inheritance, witness binding.
- `binding D6 from eatp/10-shamir-recovery.md@v1.0`, post-recovery posture downgrade and 7-day cooling-off, and the trust-anchored-time discipline of EATP-10 §14 (the cooling-off-timestamp bullet), which this binding extends to vault anchor timestamps (fix [16]).
- `binding D5 from eatp/11-posture-store.md@v1.0`, effective **posture** resolution. **Note (per F-AUTHZ-4):** D5 resolves a _posture_ on the autonomy total order; it does **not** resolve capability tokens. Capability gating (§4.2) is a structurally separate substrate axis (`CapabilitySet`). This spec cites D5 only for the constraint-dimension posture consulted by the verification gradient and for the D6 cooling-off consequence (§4.2.1), never as the resolver of `vault:*` tokens.

A spec cites another at a **specific version** (`@v1.0`), never "latest". Any amendment to a cited decision triggers re-validation of this spec per `01-analysis/03-cross-spec-dependencies.md` §9.

## 1. Scope

EATP-10 (`eatp/10-shamir-recovery.md@v1.0`, Published) specifies the **ritual** that wraps SLIP-0039 secret-sharing: paper cards, generation/reconstruction/rotation UX, the social-graph caveat, the `RecoveryRitualRecord` and its `shard_commitments` array, and the post-recovery posture semantics (**D6**). EATP-10 is the human-and-product layer. It deliberately does **not** specify how the Trust Vault SDK surface invokes the shamir wrapper, how the calling agent's authority is checked, what audit anchor a programmatic backup writes, or how a programmatic backup refuses a stale key generation.

This specification fills that SDK-binding gap. It is the **tighter, more stable sibling** of EATP-10, carved out of the same originating work (`mint#8`) but scoped to the code-facing contract that unblocks `kailash-py` issues #606 (the gated `back_up_vault_key` stub) and #630 (the corresponding restore/rotate surface). Where EATP-10 governs what a human does in a room with paper cards, EATP-12 governs what `back_up_vault_key(...)`, `restore_vault_key(...)`, and the `rotate_holders(...)` binding do when an agent calls them.

This specification defines:

1. The **input shape** of the vault key-binding surface (handle-based primary; raw-bytes escape hatch, disabled by default) (§4.1).
2. The **clearance gate** that checks `vault:backup` / `vault:restore` / `vault:rotate` tokens in the caller's bound `RoleScope.capabilities` (`CapabilitySet`), scoped to the vault's tenant/domain (§4.2).
3. The **shard-to-holder binding** that makes each shard attributable to a registered holder identity and (at Complete level) per-holder revocable (§4.3).
4. The **commitment binding** that ties each shard set and the reconstructed KEK to the identity and generation of the vault key the operator intends to restore (§4.4), the cryptographic anti-injection control.
5. The **audit envelope** that every backup/restore writes to the recovery tier, dispatcher-mediated and signed per EATP-08, fail-closed before the key goes active (§4.5).
6. The **failure taxonomy**, typed errors for the binding surface (§4.6).
7. The **rotation trigger** that binds holder departure to the shipped `rotate_holders` wrapper and the KEK-rotation event that advances the generation (§5).
8. The **stale-generation guard**, the production-biting failure mode that refuses a backup older than (or relabelled away from) the current KEK generation, and the compromised-generation denylist (§6).
9. The **cross-SDK conformance vectors** that pin the SLIP-0039 encryption parameters, the KEK-identity commitment, and the audit-envelope canonical form (§7).

This is a binding spec, not a cryptographic spec and not a ritual spec. A SLIP-0039 implementation alone does not satisfy it; the shipped shamir wrapper plus this binding does. The reference realization is `kailash-py` `trust/vault/shamir.py` (shipped, tested, Apache 2.0). The reference realization for the binding itself is the `trust/vault/backup.py` surface that this spec un-gates. Several surfaces this binding requires are **net-new** and do not yet exist in shipped code; these are flagged explicitly in §3.4 and are tracked under `kailash-py#630`/`#1304`. The spec leads the implementation; it does not misdescribe the implementation.

### 1.1 Out of scope

- **The SLIP-0039 cryptography.** The bit-level construction, wordlist, MAC, threshold-parameter semantics, Feistel encryption, and master-secret reconstruction are governed by SLIP-0039 and are out of scope here. This spec MUST NOT redefine them. It pins the _parameters_ it emits (§3.3, per F-XSDK-1) without re-specifying the algorithm. (Mirrors EATP-10 §3.2.)
- **The recovery ritual UX and paper-shard format.** Generation/reconstruction/rotation as a human procedure, the printed-card layout, read-back, distribution, and the social-graph caveat are owned by `eatp/10-shamir-recovery.md@v1.0` (§5-§7, §10). This spec MUST NOT redefine them; it binds the SDK to them.
- **Post-recovery posture semantics (D6).** The forced-SUPERVISED downgrade across all five dimensions and the 7-day cooling-off interval are owned by `eatp/10-shamir-recovery.md@v1.0` §8 (D6, N8-PO-01...04) and inherited by `eatp/11-posture-store.md@v1.0`. This spec MUST NOT redefine D6 or its trigger condition; **any** restore that materializes a KEK triggers D6 by reference (§5.4, corrected per F-BND-3). The capability-side consequence of D6 (§4.2.1) is a binding-local enforcement of D6's intent, not a redefinition of D6.
- **The EATP Capability taxonomy as a closed enum.** This spec names the capability tokens `vault:backup`, `vault:restore`, `vault:rotate`, `vault:restore-stale`, and (when the governance knob is configured) `vault:approve` and `vault:witness`, and specifies their gating semantics. The canonical enumeration of all EATP capability tokens is a separate surface, the kailash-py substrate currently carries spine-declared _string_ tokens on `CapabilitySet` (the EATP `Capability` taxonomy "is not yet surfaced in kailash-py"; `delegate/types.py:511-516`). This spec MUST NOT freeze the full taxonomy; it constrains only the `vault:*` tokens it introduces.
- **Multi-group SLIP-0039 provisioning (Mode B).** The shipped wrapper uses single-group `group_threshold=1, groups=[(threshold, total)]`. Mode B is deferred by `eatp/10-shamir-recovery.md@v1.0` §4.3 (per F32); this spec inherits that deferral and MUST NOT claim Mode B rotation conformance.
- **Trivial (1-of-n) and degenerate (1-of-1) thresholds.** The shipped `ShamirRitual` rejects `threshold==1 with total>1` and its docstring defers an `allow_trivial=True` relaxation to "mint ISS-37". This spec is that governance review and **closes** the deferral: it does **not** lift the floor (§3.3, N12-TH-01). Any future trivial-split relaxation requires a separate, explicitly-scoped amendment with its own governance gate; EATP-12 silence MUST NOT be read as the unlock the wrapper docstring references.
- **Cryptographic key zeroization (the constant-time _property_ of the upstream library).** The shipped `rotate_holders`/`reconstruct` delete the intermediate secret on a best-effort residence-minimization basis (`del` in a `finally` block), not guaranteed cryptographic zeroization, and the upstream reference SLIP-0039 implementation is documented by its authors as not constant-time. This spec inherits that cryptographic limitation and does not strengthen it (the SLIP-0039 boundary of §3.1 is preserved). It does, however, require a `del`-in-`finally` consumption discipline at the binding boundary (§4.1, N12-IN-05, per F-BND-9), recommends a memory-locked process (§8), and, critically, does **not** scope out **disclosing and gating** the non-constant-time hazard (N12-CRY-SC, §3.3; per F-CRYPTO-12). "We do not fix the crypto" is distinct from "we hide the hazard"; only the former is out of scope.
- **Storage backend for backups.** Whether shard sets are printed, held in HSM, or escrowed is deployment policy and out of scope.

## 2. Motivation

The Trust Vault holds the Key-Encryption Key (KEK), the vault master key that wraps the data keys. When the human authority behind that KEK loses access, EATP-10 specifies the ritual to recover it. But the ritual is invoked, in production, through an SDK call. Today that call does not exist: `trust/vault/backup.py` `back_up_vault_key` is a gated stub that raises `NotImplementedError` and points at this spec (issue #606). The cryptography is done and tested; only the **binding** is blocked. Several independent failure modes motivate closing the binding, and the red team converged on a sharper set than the first draft assumed:

1. **An un-gated binding leaks key material across the API.** The current stub signature accepts `vault_key: bytes`, raw KEK bytes crossing the public API surface. Every caller that holds raw KEK bytes to pass them in is a place the KEK can be logged, swapped, or exfiltrated. The binding must resolve the key **internally** from a handle so raw bytes do not cross the boundary by default (decision 1; §4.1).
2. **An un-gated binding has no authority check, and the authority object is easy to get wrong.** SLIP-0039 splits whatever secret it is handed; the wrapper does not know whether the caller is allowed to back up or restore this vault. Without a clearance gate the KEK can be sharded or reconstructed by any code path that reaches the wrapper. Worse, the capability tokens do **not** live on the `DelegateConstraintEnvelope` (which carries only constraint dimensions + `genesis_id` and has no capability field); they live on `RoleScope.capabilities` -> `CapabilitySet`, and they are domain/tenant-scoped. A gate that checks the wrong object, or ignores tenant/domain scope, fails open or authorizes the wrong vault (§4.2; F-AUTHZ-1, F-AUTHZ-2).
3. **An un-gated restore re-installs an attacker-chosen root key.** `reconstruct(shards)` returns _whatever_ secret the presented shards combine to. With no commitment binding the reconstructed secret to the vault identity the operator intends, a caller holding any internally-consistent k-of-n set (foreign, self-generated, or old-generation) can install attacker-known root key material into an authorized vault. SLIP-0039's MAC passes; the wrapper raises nothing. This is the single most dangerous gap in the surface, and the first draft only _cited_ EATP-10's commitment check rather than **binding** it (§4.4; F-CRYPTO-1/2/4/5, F-AUTHZ-3, F-BND-1).
4. **An un-gated restore silently restores a stale or compromised key.** When the KEK has been rotated, an older backup reconstructs to a **superseded** KEK; restoring it rolls the vault back to a key data keys are no longer wrapped under, a data-loss and security-downgrade event at once. A generation _label_ that is not cryptographically bound to the shard set can simply be edited to defeat the guard. The binding must tag every backup with the generation it captured, **bind** that tag into the commitment, refuse stale restores by default, and maintain a compromised-generation denylist (decision 4; §6; F-CRYPTO-3/6).
5. **An authorized restore that is unaudited is an accountability bypass.** A key-recovery binding whose audit anchor can be bypassed (direct storage write), skipped (write fails after the key goes active), or laundered (forced-stale event recorded as ordinary restore) defeats EATP's whole accountability story. The audit anchor must be dispatcher-mediated, fail-closed before the KEK goes active, and tamper-evident as a distinct class (§4.5, §6; F-AUDIT-1/2/3/4).

Envoy Phase 01/02 requires a programmatic, audited, authority-gated, cryptographically-bound Trust Vault key-binding surface. EATP-10 gives the ritual; EATP-12 gives the call. This spec fills the binding gap. It does not invent cryptography and it does not re-specify the ritual.

## 3. Binding boundary

This section draws the boundary explicitly, in the manner of EATP-10 §3. The binding sits between two layers it MUST NOT redefine.

### 3.1 What this spec inherits (and MUST NOT redefine)

- **From SLIP-0039**: the shard construction, wordlist, MAC, threshold semantics, Feistel encryption, and reconstruction algorithm. The binding calls them through the shipped wrapper; it does not reimplement them.
- **From the shipped wrapper** (`kailash-py` `trust/vault/shamir.py`, the reference realization): the exact API surface
  - `ShamirRitual(threshold: int, total_shards: int)`, frozen dataclass; validates `1 <= threshold <= total_shards <= 16`, **rejects `threshold == 1` with `total_shards > 1`**, and (per its docstring) defers an `allow_trivial=True` relaxation to ISS-37. **Editor's correction (per F-XSDK-8):** the wrapper as written still _accepts_ the degenerate `1-of-1` ritual (only `threshold==1 with total>1` is rejected); the EATP-10 floor (N8-T-02: `2 <= k <= n <= 9`, MUST NOT permit `k = 1` or `n = 1`) is stricter, and this spec adopts the EATP-10 floor normatively in §3.3.
  - `generate(secret: bytes, ritual: ShamirRitual, *, passphrase: bytes = b"") -> List[List[str]]`, single-group `group_threshold=1, groups=[(threshold, total)]`.
  - `reconstruct(shards: List[List[str]], *, passphrase: bytes = b"") -> bytes`, returns the raw master secret; **performs no identity check** against any intended target (this is exactly why §4.4 exists). `combine_mnemonics` requires **exactly** `threshold` mnemonics and rejects both under- and over-supply, and rejects mixed-identifier sets. (Note the upstream-erratum cross-reference in §3.5: the Published EATP-10 V1 "any subset of size >= k" claim is false against this wrapper.)
  - `serialize_shard / deserialize_shard`, paper-print form (space-delimited SLIP-0039 mnemonic).
  - `rotate_holders(old_shards, new_ritual, *, passphrase=b"") -> List[List[str]]`, reconstructs then regenerates under a new ritual; intermediate secret `del`-ed best-effort.
  - `EntropySource`, **a reserved alias in `__all__` that no public function accepts** (`generate` sources entropy internally from the library's CSPRNG). Per F-XSDK-9 the binding MUST NOT promote `EntropySource` into the public surface (§3.3, N12-CRY-PIN clause (e)).
    The binding MUST NOT fork or shadow these signatures; it composes them.
- **From `eatp/10-shamir-recovery.md@v1.0`** (the ritual layer): the generation/reconstruction/rotation **ritual**, paper-shard format, threshold defaults (3-of-5, N8-T-01) and floor (N8-T-02/03), the **`RecoveryRitualRecord`** and its **`shard_commitments`** array (SHA-256 of shard ciphertexts, §9.1), the **commitment check (N8-R-01)** with error code **`unknown-shard`**, the **no-plaintext-display rule (N8-R-02)**, the **`insufficient-shards`** code (V2), the **compromise dual-emit MUST (N8-A-02)**, the **override dual-emit MUST (N8-PO-04)**, the **trust-anchored-time discipline of EATP-10 §14 (the cooling-off-timestamp bullet)**, and **D6** (N8-PO-01...03: post-recovery SUPERVISED across five dimensions + 7-day cooling-off). The binding reuses these names and codes; it does not invent parallel ones (corrected per F-BND-4, F-XSDK-4). **Boundary note on `wrong-tier`:** EATP-10 N8-A-01 defines a `wrong-tier` code referencing an `info` tier that is not in EATP-09's reference tier set; because this binding hard-targets `tier="recovery"`, that code is unreachable on this surface and is NOT inherited as a live binding error (§4.6, F-XSDK-11).
- **From `eatp/08-algorithm-identifier.md@v1.0`**: `alg_id` precedence and first-field-under-JCS rule (D3), the parent's-algorithm-and-canonical-form rule (D1) for the audit anchor, and the no-trust-in-mutable-markers discipline (D2c) that informs the generation counter (§5.2). **Scope qualification (per F-AUDIT-8):** D3's literal "first JCS field" form applies where the implementation controls record framing; for the delegate-substrate-framed vault anchor (whose signed pre-image is `content_signing_bytes`, which carries no top-level `alg_id` slot), `alg_id` rides `event_payload.alg_id` per N12-AU-02, and the D3 _parseability intent_ is satisfied by the pinned schema position rather than the literal first-field placement.
- **From `eatp/09-tiered-audit-dispatcher.md@v1.0`**: the `recovery` tier, the **dispatcher-mediation MUST (N9-D-01)** (`AuditDispatcher.dispatch()`; direct storage writes are non-conformant), the **tier-required rule (N9-D-02)** with `unknown-tier`, the **atomicity rule (N9-D-03)**, the **no-cross-tier-chaining rule (N9-T-02)**, the **global anchor (N9-G-01)** and **witness binding (N9-G-03, Complete)**, and the **fail-closed witness pattern (N9-G-04)**. **Substrate-status note (per F-SUBSTRATE-2, Ruling 2):** the EATP-09 named-tier `AuditDispatcher` (with a `tier` parameter, a `recovery` tier, `seal`, `DispatchReceipt`, and the global anchor `G_k`) is a **REQUIRED surface that is net-new** and is present in **neither** shipped SDK today. EATP-09 §12 admits only a compatibility shim exists and that the SDKs "have no `AuditDispatcher`". The only shipped `dispatch()` is the gradient-keyed PACT-08 `TieredAuditDispatcher.dispatch(anchor, level: VerificationLevel)` (`trust/pact/audit.py:520`), whose second argument is `level` (not `tier`), which returns `None` (not a `DispatchReceipt`), and which has no named `recovery` tier, no `seal`, and no `verify_global_anchor`. The `tier="recovery"` / `DispatchReceipt` contract of N12-AU-02/02b therefore depends on the EATP-09 dispatcher landing (tracked in `kailash-py#630`); a conformant deployment MAY alternatively provide a binding adapter mapping recovery-tier enrollment + receipt onto the shipped gradient-keyed surface. This is a substrate-status disclosure, not a relaxation of the dispatcher-mediation requirement.
- **From `eatp/11-posture-store.md@v1.0`**: the **`effective_posture(principal, dimensions) -> TrustPosture`** function (N3-EP-01) used to read the principal's constraint-dimension posture for the verification gradient, **not** to resolve capabilities (D5 returns a posture, not a `CapabilitySet`; F-AUTHZ-4).
- **From the kailash-py delegation substrate** (`delegate/types.py`, `delegate/envelope.py`, `delegate/audit.py`): `CapabilitySet` (frozen tuple of string tokens; `intersect` at `types.py:535` is the privilege-non-escalation _composition_ primitive, see the substrate-accuracy note below), `RoleScope(domain, CapabilitySet)`, the `DelegateConstraintEnvelope` **TIGHTENING-ONLY** composition invariant (the rs **F5 type-state** invariant per kailash-rs M2-02, tracked under issue **#1035**; `envelope.py:6-8`), the closed **`DelegateEventType`** M4-canonical enum and the **C3 audit-visibility classifier** (`_AUDIT_VISIBLE_EVENT_TYPES`, `audit.py:309`), and `content_signing_bytes(event_type, event_payload, signer_delegate_id)` (`audit.py:79`) as the single signed pre-image.

> **Substrate-accuracy note on the capability gate (per F-AUTHZ-10, Ruling 2).** The shipped authorization gate is `DispatchSurface` Invariant 3 (`delegate/dispatch.py:1441`), which checks `connector.requires_capabilities` against a **single** bound role's `role.scope.capabilities.capabilities` (`DispatchSurface.__init__` takes one `role: Role`, `dispatch.py:1267`). `CapabilitySet.intersect` (`types.py:535`) exists, but it has **zero authorization call sites** in shipped code; the only `.intersect(` invocations are on `ConstraintEnvelope`. The Conformant-level capability gate of this spec is therefore membership in the **bound role's** `RoleScope.capabilities`, mirroring Invariant 3. The "composed across bound roles via `CapabilitySet.intersect`" phrasing describes the privilege-**non-escalation composition** primitive (capabilities of a delegate bound to multiple roles are the _intersection_ across roles), which is a **REQUIRED forward-path surface** that applies if and when the substrate exposes multi-role binding to the dispatch surface (tracked under `#1035`); it is **not** a Conformant-level resolution step the current single-role gate performs.

### 3.2 What this spec adds on top

- The **handle-based input shape**, KEK-class type check, and the documented raw-bytes escape hatch (disabled by default) (§4.1).
- The **clearance gate** evaluated against the bound role's `CapabilitySet`, tenant/domain-scoped, with the governance-approver HELD action and the D6 cooling-off capability suspension (§4.2).
- The **shard-to-holder attribution binding** against a deployment holder registry, with OPTIONAL per-holder wrapping for revocation (§4.3).
- The **KEK-identity + generation commitment** that binds a shard set and the reconstructed secret to the vault key the operator intends, the commitment-algorithm agility binding, and the per-deployment `shard_commitments` reuse for foreign-shard rejection (§4.4), the cryptographic anti-injection control the first draft only cited.
- The **vault audit envelope**, the specific anchor payload a programmatic backup/restore/rotation emits, dispatcher-mediated, fail-closed, mapped onto the closed `DelegateEventType` enum via `subtype` (§4.5).
- The **stale-generation guard** and the **compromised-generation denylist** (§6), plus the **KEK-rotation audit anchor** that advances the generation (§5.2).
- The **cross-SDK conformance vectors** pinning encryption parameters, KEK-identity commitment, and audit canonical form (§7), with a materialized **golden fixture** (Appendix B).

The boundary is clean: SLIP-0039 and the shipped wrapper produce shards; EATP-10 governs the ritual, the `shard_commitments`/`RecoveryRitualRecord`, and D6; this spec produces an **authority-gated, identity-and-generation-bound, audited SDK call** that drives them. The relationship to EATP-10 is sibling-and-binding, not parent-and-child: EATP-12 cites EATP-10 for the ritual, the commitment surface, and D6, and EATP-10 is unchanged by this spec (save for the upstream V1 erratum flagged in §3.5, which corrects EATP-10's own text, not this binding).

### 3.3 Pinned ritual and SLIP-0039 parameters

The first draft asserted that the serialized paper-print form is "a published standard and therefore reproducible." That is true **only if both SDKs agree on the SLIP-0039 backup-encryption parameters**, which the wrapper inherits from library defaults and does not pin (per F-XSDK-1, mixing an `extendable=True` shard with an `extendable=False` shard raises "The identifier parameters don't match", and `iteration_exponent` changes the PBKDF2 round count). The binding therefore pins them.

> **N12-CRY-PIN (pinned SLIP-0039 parameters)**: A conformant binding MUST emit shards under the following fixed parameters, and MUST record them in the backup audit envelope (§4.5) so a cross-SDK restore can detect a mismatch:
> (a) **extendable-backup flag** = `true` (matches the shipped wrapper's library default; a deployment MUST NOT vary it per-backup).
> (b) **`iteration_exponent`** = `1` unless a deployment-wide pinned value is recorded in the audit envelope; the value MUST be identical across all conformant SDKs in the deployment.
> (c) **single-group encoding** `group_threshold = 1, groups = [(k, n)]` (Mode A only; Mode B out of scope per §1.1).
> (d) **master-secret length** in {128-bit, 256-bit}. The escape-hatch path (§4.1) MUST reject any other length with `invalid-secret-length` (§4.6) before calling the wrapper. The chosen length MUST be recorded in the audit envelope as `slip39_params.master_secret_bits` (§4.5.1, per F-XSDK-10) so a 128-vs-256 divergence surfaces as `parameter-mismatch` at the cross-SDK reproducibility layer.
> (e) **entropy source**: shard randomness MUST come from a CSPRNG (`os.urandom`/`secrets` equivalent). The binding MUST NOT expose a caller-seedable entropy parameter on the public `back_up_vault_key` path; `EntropySource` is a reserved alias and is NOT part of the frozen binding surface. Any deterministic-entropy facility is a test-only fixture path that MUST NOT be reachable from the conformant public surface (predictable shares are a quorum-bypass; F-XSDK-9).

> **N12-TH-01 (threshold floor; trivial split forbidden)**: `back_up_vault_key` and the rotation binding (§5) MUST reject any ritual that does not satisfy the EATP-10 floor `2 <= k <= n <= 9` (N8-T-02), with the typed error `invalid-ritual` (§4.6), before any key is resolved. In particular a `1-of-n` or `1-of-1` ritual MUST be rejected for vault-KEK backup at **all** conformance levels, the KEK is the highest-value secret and quorum is mandatory. The wrapper's deferred `allow_trivial=True` relaxation MUST NOT be reachable from the conformant vault-KEK surface. The default threshold presented is 3-of-5 (N8-T-01); the strict-majority recommendation N8-T-03 SHOULD be surfaced in any operator UX. (Closes the ISS-37 deferral; F-CRYPTO-10, F-BND-7, F-XSDK-8.)

> **N12-CRY-SC (non-constant-time reconstruction: disclosure + execution gate)**: The reference SLIP-0039 implementation the wrapper calls is documented by its authors as **not constant-time** and "suitable for correctness verification rather than handling of high-value secrets in adversarial settings" (`trust/vault/shamir.py:36-43`). Because EATP-12 reconstructs the **vault KEK** (its own §2 "highest-value secret"), the binding MUST surface this hazard rather than hide it (this is the part §1.1 does NOT scope out):
>
> - (a) **Conformant**: the binding MUST record a `side_channel_hardened` boolean in the backup/restore audit envelope (§4.5.1) and the `BackupReceipt`, defaulting to `false` when KEK reconstruction runs through the non-constant-time reference `combine_mnemonics`. This makes the property testable (V7).
> - (b) **Complete**: KEK reconstruction MUST run in a hardware-backed module (HSM / secure enclave) such that the master secret never enters the non-constant-time userspace routine; `side_channel_hardened: true` is assertable **only** at Complete. NIST SP 800-57 key-management and PKCS#11 wrap practice are the benchmark.
>   This clause is disclosure-and-execution-environment only; it does NOT strengthen the SLIP-0039 cryptography (preserving the §3.1 boundary). (Per F-CRYPTO-12.)

### 3.4 Required net-new substrate (implementation-gap disclosure)

Per Ruling 2 (specs lead implementations; claims about shipped surfaces must be truthful), the following primitives are **REQUIRED surfaces that are net-new**, they exist in **neither** shipped SDK today and are tracked under `kailash-py#630`/`#1304`. The §6 stale-guard and the N12-IN-02 type check depend on them. The §9 red team meticulously audited the _capability_ substrate (`CapabilitySet`, envelope, `DelegateEventType`) against shipped reality but did not initially audit the _key-manager_ substrate; this subsection closes that honesty gap.

- **`key_class in {KEK, DATA}` metadata tag.** Shipped `KeyMetadata` (`trust/key_manager.py:110-118`) carries `key_id, algorithm, created_at, expires_at, is_hardware_backed, hsm_slot, is_revoked, revoked_at, rotated_from`, **no key-class field**. The `TrustPlaneKeyManager` protocol (`plane/key_managers/manager.py:47-93`) is sign/get_public_key/key_id/algorithm only, and `LocalFileKeyManager` is a single Ed25519 **signing** key with no encryption hierarchy. The binding REQUIRES the key metadata to expose a `key_class` tag (net-new).
- **`kek_generation: int` monotonic counter** on the key's authoritative metadata. Shipped metadata carries `rotated_from` (a key-ID lineage string), not a generation integer. The binding REQUIRES a monotonic generation integer advanced only by an audited KEK-rotation event (N12-RT-06) (net-new).
- **KEK/data-key wrapping hierarchy.** No KEK-vs-data-key distinction exists in shipped key management; the binding REQUIRES it so the KEK protects data keys transitively (net-new).

Until these land, conformance requires the metadata extension above. N12-IN-02 and §6 are written in _binding-adds_ framing accordingly: the binding requires the key manager to expose a KEK-class tag and a generation counter; it does not assert they are already present at `trust/plane/key_managers/`.

## 4. The vault key-binding surface

This section specifies the binding contract: the two public entry points `back_up_vault_key` and `restore_vault_key`, their input shape, the clearance gate, the shard-to-holder binding, the commitment binding, the audit envelope, and the failure taxonomy. The signatures below use Python type-hint syntax for clarity; the contract is language-neutral and the cross-SDK reproducibility surface is the serialized paper-print form plus the pinned audit/commitment canonical forms (§7).

### 4.1 Input shape (decision 1)

The binding is **handle-based primary**. The public surface takes a key **handle/ID** and resolves the current vault master key (the KEK, not data keys) internally.

> **N12-IN-01 (handle-based primary input)**: The public `back_up_vault_key` and `restore_vault_key` entry points MUST accept a key handle (an opaque vault key identifier), not raw key bytes, as their primary key argument. The binding MUST resolve the handle to the current vault master key (KEK) **internally**, inside the trusted vault process. Raw key bytes MUST NOT cross the public API boundary by default. An implementation whose default public surface requires the caller to supply raw KEK bytes is non-conforming at Conformant level.

> **N12-IN-02 (KEK only, type-enforced, never data keys)**: The handle MUST resolve to the vault master key (Key-Encryption Key). Handle resolution MUST verify the resolved object is **tagged KEK-class** in the key manager's metadata and MUST reject a data-key, wrapped-key-blob, or any non-KEK handle with the typed error `not-a-kek` (§4.6) **before any sharding**. The `key_class` tag the check reads is a **REQUIRED net-new surface** per §3.4 (it is not present in shipped `KeyMetadata`); until the substrate carries it, conformance requires the §3.4 metadata extension. The binding MUST NOT shard, back up, or restore individual data keys directly; data keys are protected transitively by the KEK. (Per F-CRYPTO-5: the first draft asserted the KEK/data-key boundary but provided no enforcement; the KEK-class type check and `not-a-kek` error make it falsifiable, exercised by V8(b).)

> **N12-IN-03 (raw-bytes escape hatch, disabled by default)**: An implementation MAY retain a low-level raw-bytes form (`secret: bytes`) as a documented escape hatch for migration tooling that already holds key material outside the vault's handle space. The escape hatch:
>
> - MUST be **disabled by default**, gated behind an explicit build/deploy flag (a **MUST**, upgraded from the first draft's SHOULD per F-CRYPTO-7). With the flag absent, any raw-bytes invocation MUST raise `escape-hatch-disabled` (§4.6). An implementation whose default build exposes the raw-bytes restore path is non-conforming.
> - MUST, when enabled, be subject to the same clearance gate (§4.2), holder registry (§4.3), commitment binding (§4.4), audit envelope (§4.5), and stale-generation guard (§6) as the handle-based path. Because a raw-bytes restore supplies the secret directly, the binding MUST still register and verify the KEK-identity commitment for `target_handle` (§4.4) and MUST reject bytes that do not match the target's registered commitment with `kek-commitment-mismatch`; the generation consulted is `target_handle`'s authoritative generation, never a caller-supplied integer.
> - MUST, when enabled and a raw-bytes restore proceeds, require the governance-approver HELD action (N12-CL-03 made mandatory for this path) and emit a distinct loud anchor `subtype = vault_key_restore_raw` dual-emitted to the `recovery` and `safety` tiers.
> - The shipped stub's current `vault_key: bytes` signature (`trust/vault/backup.py`, issue #606) is sanctioned to evolve to the handle-based form under this spec; the stub docstring's note that "signature evolution is in scope for ISS-37" is the warrant for that change.

> **N12-IN-04 (key-identity capture at backup)**: `back_up_vault_key` MUST record the resolved KEK's stable key identity (key-ID) and bind it into the KEK-identity commitment (§4.4, N12-CB-01). This is the anchor that lets restore reject a cross-vault re-install (`key-identity-mismatch`, §4.4/§4.6; F-CRYPTO-5, F-AUTHZ-3).

> **N12-IN-05 (no plaintext crosses the boundary; consume-and-`del`)**: Per inherited N8-R-02, a reconstructed KEK MUST NOT be returned to the caller in plaintext: the restore path re-establishes the KEK inside the trusted vault module (or a fresh signing/wrapping operation) and returns an opaque handle, never the bytes. The reconstructed master-secret bytes MUST be consumed inside the trusted vault module and `del`-ed in a `finally` block before any return; the binding MUST run the KEK-handling region in the memory-locked, swap-disabled process §8 requires. The plaintext KEK MUST appear in no return value, log line, receipt field, or audit payload. (Per F-BND-9: the first draft asserted the no-plaintext outcome but specified no enforcement point and no adversarial vector; N12-IN-05 supplies both, exercised by V7(b).)

```python
# Conformant public surface (handle-based primary)
def back_up_vault_key(key_handle: VaultKeyHandle, ritual: ShamirRitual,
                      clearance: ClearanceContext,
                      holders: list[HolderId]) -> BackupReceipt: ...

def restore_vault_key(shards: list[list[str]], target_handle: VaultKeyHandle,
                      clearance: ClearanceContext,
                      *, force_stale: bool = False,
                      passphrase_ref: PassphraseRef | None = None) -> RestoreReceipt: ...
```

`ClearanceContext` carries the caller's bound `RoleScope`/`CapabilitySet`, the dispatch cascade's bound tenant, and the vault's `(tenant, domain)` as resolved from the handle (§4.2). `passphrase_ref` is a provenance handle, never raw passphrase bytes on the public path (§4.4.1). `BackupReceipt` carries the shard set (or references to it), the audit-anchor reference, the captured `kek_generation` tag (§6), the KEK-identity commitment, its `kek_commitment_alg` (§4.4), the key-check-value (N12-CB-04), the `shard_commitments` (§4.4), the `side_channel_hardened` flag (N12-CRY-SC), and the holder binding (§4.3), never the KEK bytes or passphrase. `RestoreReceipt` carries the re-established key handle and the audit-anchor reference, never the KEK bytes.

### 4.2 Clearance gate (decision 2)

Authorization to back up or restore is the caller's **clearance** plus, on restore, the **physical SLIP-0039 quorum** (k shards) and the **commitment match** (§4.4). All required gates are independent.

> **N12-CL-01 (backup capability required, on the correct substrate object)**: `back_up_vault_key` MUST verify that the calling agent holds the `vault:backup` token in the **bound role's `RoleScope.capabilities`** (the `CapabilitySet` of the role under which the call is dispatched), mirroring the shipped `DispatchSurface` Invariant 3 (`delegate/dispatch.py:1441`, `role.scope.capabilities.capabilities`), before resolving the key handle. **The capability axis is `CapabilitySet`, NOT `DelegateConstraintEnvelope`** (which carries only the five constraint dimensions + `genesis_id` and has no capability field; F-AUTHZ-1). The `DelegateConstraintEnvelope` TIGHTENING-ONLY invariant (the rs F5 type-state, #1035) is the constraint-limit guarantee and is evaluated for the verification-gradient envelope; capability non-escalation is the separate `intersect` composition primitive. Multi-role intersected resolution via `CapabilitySet.intersect` is a REQUIRED forward-path surface (#1035), NOT a Conformant-level resolution step (per F-AUTHZ-10; see §3.1 substrate-accuracy note). A caller lacking `vault:backup` MUST be rejected with `missing-clearance` **before** any key resolution occurs. No backup is produced and no KEK is materialized for an unauthorized caller.

> **N12-CL-02 (restore capability required; independent gates)**: `restore_vault_key` MUST verify that the calling agent holds the `vault:restore` token in the bound role's `RoleScope.capabilities`, evaluated identically to N12-CL-01. A caller lacking `vault:restore` MUST be rejected with `missing-clearance` before any shard is combined. The clearance check, the SLIP-0039 quorum check, the commitment check (§4.4), and the generation check (§6) are **independent**: holding `vault:restore` without k valid shards fails on threshold (`insufficient-shards`, §4.6); presenting k valid shards without `vault:restore` fails on clearance (`missing-clearance`); presenting k valid foreign shards fails on commitment (`unknown-shard` / `kek-commitment-mismatch`, §4.4). Every required gate MUST pass for a restore to proceed.

> **N12-CL-02a (tenant/domain scoping, capability is not flat; binding-owned)**: A `vault:*` token is **not** scope-free. Tenant/domain scoping is a **binding-OWNED** control, not inherited substrate behavior: the shipped capability gate (`dispatch.py:1441-1463`) reads only `CapabilitySet` token membership and is domain-blind, and `RoleScope` (`types.py:579`) carries only `domain`, it has **no tenant field**. Therefore the binding MUST, in addition to the token-membership check:
>
> - (a) independently verify that the dispatch cascade's bound tenant (`TenantScopedCascade.tenant`, resolved from `ClearanceContext`, never read off `RoleScope`) equals the vault's tenant resolved from `key_handle`/`target_handle`; and
> - (b) independently verify that the bound role's `RoleScope.domain` covers the vault's domain (an explicit binding-added check, since the substrate gate is domain-blind).
>   Both checks are **fail-closed** (tenant first, then domain, then token). A `vault:restore` granted in domain A or tenant A MUST fail with `missing-clearance` against a target in domain B / tenant B even with k valid shards. The binding MUST NOT rely on the substrate to perform a tenant/domain capability cascade; the shipped gate does not. (Per F-AUTHZ-2 and the substrate-accuracy correction F-AUTHZ-11: a flat token would let a low-value dev-vault grant authorize a cross-tenant production-KEK recovery.) Exercised by V2(c).

> **N12-CL-03 (governance-approver HELD action, Complete-level knob, fully specified)**: A deployment MAY require, in addition to the caller's capability, an independent governance-approver authorization realized as a **HELD** action in the EATP verification gradient. This knob is **OPTIONAL** and MUST NOT be mandatory for Conformant-level conformance; an implementation that offers it MAY claim it as a **Complete-level** enhancement, and it is mandatory only for the high-risk paths this spec names (raw-bytes restore N12-IN-03, forced-stale restore N12-SG-03, cooling-off restore N12-CL-04). When configured, the binding MUST:
>
> - (a) require the approver to hold a distinct **`vault:approve`** token in the vault's tenant/domain;
> - (b) **forbid approver == requester**, a distinct `delegate_id` and distinct credential; self-approval is non-conforming;
> - (c) include the approver's identity and a signed approval token **inside the audit-envelope `event_payload`**, which is covered by the `content_signing_bytes` pre-image (§4.5, N12-AU-03), so a restore with a missing or forged approval is cryptographically detectable;
> - (d) **fail-closed**: a restore that proceeds without a valid, in-pre-image approval MUST be rejected with `missing-clearance` (not merely "recorded").
>   (Per F-AUTHZ-7: the first draft left self-approval possible, the approver capability undefined, and the approval outside the signed bytes. Note that `content_signing_bytes` signs `event_type + event_payload + signer_delegate_id`; placing the approval token in `event_payload` puts it inside the signed pre-image even though the engine-assigned chain-link fields are excluded by construction.) Exercised by V8(d).

> **N12-CL-05 (Complete-level backup ceremony witness)**: The moment of maximum KEK exposure is `back_up_vault_key` **generation**: the master secret is materialized in one process to be split, and a single authorized caller with `vault:backup` runs it alone (the N12-SH-01 registry limits WHO holds shards, not that generation is single-actor). At **Complete** level, on the generation/backup path, the binding MUST require an independent witness, a distinct `vault:witness`-holding principal, `!= requester` and `!= any configured approver`, whose identity and a signed witness token are bound into the `vault_key_backup` audit envelope's signed `event_payload` (covered by `content_signing_bytes` per N12-AU-03), mirroring N12-CL-03's structure but applied to backup rather than restore. This is **OPTIONAL** at Conformant (which already inherits EATP-10 §6.1's human operator+witness read-back ritual when run through the product path) and **REQUIRED at Complete**, framed as defense-in-depth parity with EATP-10 §6.1 witnessing and root-key-ceremony practice (NIST SP 800-57, Trezor SLIP-0039 ceremony guidance), NOT as the attacker-holder exfil fix (already closed by F-AUTHZ-6). A backup lacking the in-pre-image witness token MUST be rejected at Complete, and a self-witness (witness == requester) MUST fail. (Per F-AUTHZ-12.) Exercised by V8(f).

#### 4.2.1 Post-recovery capability suspension (D6 enforcement, not redefinition)

> **N12-CL-04 (cooling-off capability suspension; trust-anchored clock)**: D6 downgrades the recovered principal's _posture_ to SUPERVISED (N8-PO-01) but the substrate posture axis does **not** remove a `vault:*` token from a `CapabilitySet`, posture and capability are orthogonal (F-AUTHZ-4). To enforce D6's intent at the vault layer, during the 7-day cooling-off window (N8-PO-02) the binding MUST treat the recovered principal's `vault:restore`, `vault:backup`, and `vault:rotate` tokens as **suspended**: a second restore/backup/rotation by that principal MUST require the governance-approver HELD action (N12-CL-03) regardless of deployment level, or be rejected with `missing-clearance` if no approver is configured. The determination of whether a principal is within the 7-day window MUST use the **trust-anchored clock** per `binding D6 from eatp/10-shamir-recovery.md@v1.0` §14 (the same trust-anchor that signs the recovery anchor's timestamp), never a locally-mutable wall clock; a clock roll-forward MUST NOT lift the suspension, and on inability to consult the trust-anchored time the suspension MUST remain in force (fail-closed). This is a binding-local consequence of D6, not a redefinition of D6. (Per F-AUTHZ-4 and the trust-clock hardening F-TEMP-2.) Exercised by V8(e).

The clearance gate is the SDK analogue of EATP-10's human recovery-requester verification (EATP-10 §6.3 step 1): EATP-10 verifies a human; EATP-12 verifies an agent's bound `CapabilitySet`. Both gates can apply in a deployment that runs the ritual through the SDK.

### 4.3 Shard-to-holder binding (decision 3)

Each shard MUST be attributable to a **registered** holder identity. Physical custody (the default) and optional cryptographic per-holder wrapping are the two levels.

> **N12-SH-01 (holder attribution against a registry)**: Every shard produced by `back_up_vault_key` MUST be bound to a holder identity drawn from a **deployment-controlled holder registry** (named, attestable principals); the binding MUST reject any holder identifier not in the registry with `unregistered-holder` (§4.6). The audit envelope (§4.5) MUST record which registry holder identities the shards were distributed to, by stable identifier, so that "shard k was held by holder H" is reconstructable from the audit record. Attribution MUST NOT record shard contents (§4.5). Physical custody, the holder physically possesses the paper shard per EATP-10 §5, is the default attribution mechanism and is sufficient at **Conformant** level. (Per F-AUTHZ-6: caller-arbitrary holder identifiers turn backup-to-attacker-holders into a sanctioned exfiltration channel; the registry closes it.) Exercised by V8(c).

> **N12-SH-02 (per-holder wrapping, Complete-level revocation; SHOULD for-cause)**: An implementation MAY wrap each shard under a per-holder public key or per-holder passphrase before distribution, so that a departed holder's shard can be **revoked** without re-running the full ritual against every holder. Per-holder wrapping is **OPTIONAL** (a MAY) at Conformant level and is a **Complete-level** enhancement; for a holder revoked **for cause** (suspected compromise) an implementation SHOULD use per-holder wrapping AND MUST advance the generation per N12-SH-04. The HMAC sign/verify primitives at `delegate/audit.py` are the available wrapping primitives for the passphrase variant; the public-key variant is implementation-defined and MUST be reproducible across implementations only at the unwrapped serialized-shard form (§7). When per-holder wrapping is used, revocation of a departed holder MUST be recorded in the audit envelope as a holder-revocation event and MUST cause subsequent reconstruction attempts that present the revoked holder's (now-unwrappable) shard to fail with `revoked-holder` (§4.6).

> **N12-SH-03 (revocation does not weaken threshold below k)**: Revoking a departed holder's shard under N12-SH-02 reduces the available holder set. An implementation MUST NOT allow revocation to silently drop the effective reconstruction set below the ritual threshold `k`. If revocation would leave fewer than `k` un-revoked holders, the implementation MUST require a rotation (§5) and MUST surface this to the operator rather than leaving the vault unrecoverable.

> **N12-SH-04 (for-cause revocation forces a generation-advancing rotation)**: At **Conformant** level (physical custody, no wrapping) a departed holder who retained a paper card holds a cryptographically valid, **current-generation** shard; neither an "identifier check" nor attribution can cryptographically distinguish it from a legitimate shard, and a holder rotation does **not** advance the generation (§5.2). Therefore, when a holder is revoked **for cause**, the binding MUST treat the revocation as a **KEK-affecting event** and perform a full KEK-rotation (§5.2) that advances `kek_generation`, so the departed holder's retained shards become **stale** (refused by N12-SG-02) in addition to any commitment supersession. The for-cause revocation MUST emit a single `vault_kek_rotation` anchor (N12-RT-06) carrying **`for_cause = true`** AND the new (post-revocation) re-shard distribution (`holders`/`shard_count`/`shard_commitments`/the new `kek_identity_commitment`); the `for_cause` flag the audit envelope MUST record therefore lands on the same anchor that advances the generation and carries the new distribution, with no two-anchor split and no flag stranded on an anchor whose generation is unchanged (fix [4], third cycle). Because the `vault_kek_rotation` anchor now carries `shard_commitments`, N12-CB-03's foreign-shard check has a `g+1` source immediately, without waiting for a subsequent backup. (Per F-AUTHZ-5, F-CRYPTO-8: the first draft over-claimed that a revoked current-generation shard "fails a commitment/identifier check", at Conformant that check cannot distinguish it; generation-supersession is the honest defense.) Exercised by V4(c).

Physical custody (Conformant) gives attribution but not cryptographic revocation: a departed holder who kept a paper card still holds a cryptographically valid shard of the unchanged master secret until generation-supersession (N12-SH-04) or per-holder unwrapping-failure (Complete) revokes it. Both levels bind each shard to a registry holder identity for attribution; they differ in whether revocation is cryptographic, procedural, or generation-driven.

### 4.4 Commitment binding (the anti-injection control)

`reconstruct(shards)` returns whatever the shards combine to, with no notion of which vault they belong to. The first draft only _cited_ EATP-10's N8-R-01 commitment check; this section **binds** it and adds a cross-SDK-reproducible KEK-identity commitment. This is the load-bearing fix for F-CRYPTO-1/2/4/5, F-AUTHZ-3, and F-BND-1.

The commitment pre-image MUST be an unambiguous, length-delimited canonical encoding so that no two distinct input tuples share a pre-image (a bare `H(a || b || ...)` concatenation lets `(vault_id='ab', salt='c')` and `(vault_id='a', salt='bc')` collide regardless of secret entropy; F-CRYPTO-13). The binding reuses the substrate's canonical JSON form (RFC 8785 / JCS, `canonical_json_dumps`, `delegate/audit.py`) that the rest of the suite already pins for every other hashed pre-image.

> **N12-CB-01 (KEK-identity + captured-generation commitment registered at backup)**: Every backup MUST register, in the `BackupReceipt`, the audit envelope (§4.5), and against the vault resolved from `key_handle` (equivalently the resolved KEK's `vault_id` per N12-IN-04; `target_handle` is the restore-side parameter and is NOT present on the backup surface, fix [15]), a **KEK-identity commitment** that commits to the reconstructed master secret's _identity_ without encoding the KEK bytes. The commitment binds the **captured `kek_generation`**, the generation value the KEK carried at backup time. This captured value, not the vault's later current generation, is what the restore-side verifier recomputes over (N12-CB-02(b)/(c)) and what is bound per N12-SG-01(b). The commitment is:
>
> ```
> kek_identity_commitment = H_alg( canonical_json_dumps({
>     "domain_sep": "EATP-12/kek-identity-commitment/v1",
>     "vault_id": <string>,
>     "kek_generation": <int>,
>     "master_secret": <lowercase-hex>,
>     "passphrase_provenance": <string-tag, §4.4.1>
> }) )
> ```
>
> where `H_alg` is the hash function identified by the recorded `kek_commitment_alg` (N12-CB-04). The single **canonical input tuple** is exactly `{domain_sep, vault_id, kek_generation, master_secret, passphrase_provenance}`, where `kek_generation` is the **captured generation** (the value the backup tagged at backup time, N12-SG-01), and MUST be enumerated identically, and over the same **captured-generation** value semantics, in N12-CB-01, N12-CB-02(b)/(c), §8, and the Appendix B golden fixture (resolving the prior draft's three divergent input lists and the captured-vs-current generation-value confusion; F-CRYPTO-13, fix [1]/[14]). Notes:
>
> - `domain_sep` is the **fixed ASCII constant** `"EATP-12/kek-identity-commitment/v1"`, pinned and reproduced in the Appendix B golden fixture.
> - **`vault_salt` is DELETED** from the construction: `domain_sep` provides domain separation, the construction commits to `master_secret` directly, and an undefined-and-unrecorded per-vault salt made the V6 byte-identical guarantee unsatisfiable (F-CRYPTO-14). There is no `vault_salt` field.
> - The commitment MUST bind `vault_id`, `kek_generation`, and the passphrase provenance (§4.4.1) so that a relabelled generation, a wrong passphrase, or a cross-vault target yields a mismatch rather than a silent success.
> - Because it commits to the **reconstructed secret** (deterministic across SDKs and across re-generations for a fixed `(vault_id, generation, secret, passphrase_provenance)`), it is the cross-SDK and cross-generation anti-injection anchor, unlike per-shard ciphertext hashes, which are non-deterministic (F-XSDK-9, F-CRYPTO-9). It is a one-way commitment, not the KEK bytes, so N12-AU-01 contents-exclusion still holds.
>   A keyed construction (`HMAC-H(K = HKDF-Extract(domain_sep), msg = canonical-encoding(tuple))`) giving hiding independent of master-secret entropy is a RECOMMENDED hardening, not the core requirement; the load-bearing requirement is the unambiguous canonical encoding above.

> **N12-CB-04 (commitment-algorithm agility + standalone key-check-value)**: The commitment hash function MUST be drawn from a commitment-suite identified by an **EATP-08 §3.3 registry token** (the compound signature-and-hash identifier such as `eatp-v1`, NOT the bare hash-column value `sha-256`), mirroring EATP-11's `hash_anchor` precedent (`<alg_id>:<hex>` with `alg_id = eatp-v1`) which EATP-12's most security-critical hash previously failed to follow (F-CRYPTO-11, fix [11]). EATP-08 §3.3 registers no standalone bare-hash token; the registry identifier is the compound token and the hash primitive (SHA-256 for `eatp-v1`) is resolved from that row's Hash column:
>
> - (a) A first-class **`kek_commitment_alg`** field (a fixed string) MUST be recorded in the `BackupReceipt`, the audit `event_payload` (§4.5.1 pinned schema), and any `RecoveryRitualRecord`. Pin `kek_commitment_alg = "eatp-v1"` explicitly (the EATP-08 §3.3 registry token whose Hash column resolves to SHA-256) until a PQC-secure commitment suite is registered. The value MUST be identical across all conformant SDKs in a deployment.
> - (b) **Recompute-under-recorded-alg (a D1 analogue), against the per-algorithm registry.** At restore, the verifier MUST recompute the commitment under the backup's **recorded** `kek_commitment_alg`, NEVER the verifier's current-suite hash, mirroring `binding D1 from eatp/08-algorithm-identifier.md@v1.0`'s parent's-algorithm principle (noting the commitment is not itself a chain hash). It MUST compare the recompute to the registered commitment for `target_handle` at the captured generation **under that same recorded algorithm**, selected from the per-(`target_handle`, generation) registry of commitments-keyed-by-algorithm that N12-CB-04(c) maintains, NOT against the latest or current registered algorithm. A decades-old paper backup computed under `eatp-v1` (SHA-256) MUST still verify after the deployment's current suite advances (e.g. to `eatp-v1.1` SHA-512/256), because the `eatp-v1` commitment remains registered, rather than failing with the same `kek-commitment-mismatch` code used for injection (F-XSDK-10). `commitment-alg-mismatch` (§4.6) fires **only** when NO registered commitment exists for `target_handle` at the captured generation under the backup's recorded algorithm (an algorithm never registered and never recommitted for that vault/generation), NOT merely when the recorded algorithm differs from the current or latest one. A registered-but-superseded algorithm still verifies; an unregistered algorithm fails distinctly, not with the misleading `kek-commitment-mismatch` (fix [2]). A registered-but-**retired** algorithm (its entry marked non-verifiable by a `vault_kek_retire` anchor per (e)) is distinct from both: it fails with **`retired-commitment-alg`** (§4.6), never silently verifying against the retired entry and never collapsing into `kek-commitment-mismatch` or `commitment-alg-mismatch` (fix [5], third cycle).
> - (c) **Hash-sunset migration is ADDITIVE.** The verifier maintains, per (`target_handle`, `kek_generation`), a **registry of KEK-identity commitments keyed by `kek_commitment_alg`**. A legitimate backup MAY have its commitment algorithm advanced by a signed, dispatcher-mediated `recovery`-tier re-commitment anchor (`subtype = vault_kek_recommit`, §4.5) that **ADDS** the new-algorithm commitment `C_Y` **without deleting** the prior-algorithm commitment `C_X`; both `C_X` and `C_Y` stay live until the prior-algorithm entry is **explicitly retired** via the retirement operation of (e) below (mirroring a hash-sunset migration window in which both hashes remain valid during the window, after which the weak hash is sunset). The recommit MUST NOT alter `kek_generation` or `vault_id`; it binds the new commitment to the **same** reconstructed secret and the same `(vault_id, kek_generation)` as the prior commitment, and a chain of `vault_kek_recommit` anchors forms the migration history. This additive registry lets a long-lived corpus survive a hash sunset without disabling the check and without stranding old-algorithm escrow backups (fix [2]). The `vault_kek_recommit` anchor records the from-to pair (`prior_kek_commitment_alg`, `prior_kek_identity_commitment`, `new_kek_commitment_alg`, `new_kek_identity_commitment`) per the N12-AU-04 recommit schema (fix [12]). An operator-surfaced **growth metric** MUST be exposed over the per-(`target_handle`, `kek_generation`) commitment registry so the additive accretion of recommit entries is observable and bounded operationally; a deployment MAY additionally impose a normative cap on live entries per (`target_handle`, `kek_generation`), enforced by requiring a retirement (e) before a further recommit once the cap is reached (fix [5], third cycle).
> - (d) **Standalone integrity / key-check-value (KCV).** The `BackupReceipt` MUST carry a key-free **KCV**, a short, truncated, domain-separated one-way hash of the reconstructed KEK, `KCV = truncate_N( H_alg( canonical_json_dumps({"domain_sep":"EATP-12/kcv/v1","vault_id":..., "kek_generation":<int>, "master_secret":<hex>}) ) )` per NIST SP 800-130, with **`N = 8` bytes (the leading 8 bytes of the digest), encoded as 16 lowercase-hex characters**. `N` is a fixed constant across all conformant SDKs in a deployment, independent of digest size or master-secret length, sized so it leaks no usable secret (fix [10], closing the prior circular deferral to §4.5.1). This lets an operator confirm offline "this blob reconstructs the key I expect" against an out-of-band expected KCV **without** a live-vault round-trip (the N12-CB-02 commitment check is live-vault-dependent; a blob in escrow for years is otherwise integrity-naked, unlike PKCS#11 C_WrapKey / RFC 9180 HPKE / NIST SP 800-130 practice; F-CRYPTO-15). A relabelled/tampered blob MUST fail KCV standalone with `kcv-mismatch` (§4.6). A deployment MAY additionally carry, at **Complete** level, a keyed AEAD/MAC tag over the blob's associated data (`kek_generation`, `vault_id`, `slip39_params`, holder distribution, the commitment, the `shard_commitments`) under a deployment backup-integrity key, OPTIONAL, with the explicit caveat that the integrity key's own custody is itself a recovery-scenario dependency (if lost with the vault, the keyed tag is unverifiable), which is precisely why the key-free KCV is the mandatory floor and the keyed AEAD the enhancement.
> - (e) **Algorithm retirement (weak-algorithm sunset).** The additive registry of (c) keeps every registered commitment live indefinitely, which means a weakened or compromised commitment algorithm would otherwise remain a valid verification path forever (a restore presenting a backup recorded under the broken algorithm would still verify against the still-live entry). The binding therefore defines a normative **retirement** operation so a specific `(kek_commitment_alg -> kek_identity_commitment)` registry entry can be marked **non-verifiable**:
>   1. Retirement is performed by a signed, dispatcher-mediated, fail-closed `recovery`-tier anchor `subtype = vault_kek_retire` (§4.5, N12-AU-04) recording `vault_id`, `kek_generation`, the `retired_kek_commitment_alg`, the `retired_kek_identity_commitment`, the principal, and the trust-anchored timestamp. The effect (marking that specific entry non-verifiable) is derivable from the audited anchor chain, so a verifier learns retirement from the chain, never from a locally-mutable store.
>   2. Retiring an algorithm requires either a distinct **`vault:retire-alg`** token in the vault's tenant/domain OR the governance-approver HELD action (N12-CL-03); it is a high-consequence operation and MUST NOT be performed on the ordinary `vault:restore`/`vault:backup` capability.
>   3. A restore presenting a backup recorded under a **retired** algorithm MUST fail with the distinct error **`retired-commitment-alg`** (§4.6), NEVER silently succeed and NEVER collapse into `kek-commitment-mismatch` (injection) or `commitment-alg-mismatch` (never-registered). The recompute-under-recorded-alg rule of (b) is evaluated only against **live, non-retired** entries.
>   4. **Recoverability safety.** A retirement that would strand legitimate escrow backups MUST first require re-commitment of the affected corpus under a live, non-retired strong algorithm: the N12-AU-04 `vault_kek_retire` row mandates that the retired entry already be superseded by a live `C_Y` (a prior `vault_kek_recommit`) for the same `(vault_id, kek_generation)`, so retirement closes the weak path without destroying recoverability.
>   5. **EATP-08 reconciliation.** When an EATP-08 §3.3 registry row for a `kek_commitment_alg` is deprecated/sunset upstream, the deployment MUST retire the corresponding per-(`target_handle`, `kek_generation`) commitment entries via (e)(1) after first re-committing per (e)(4). The EATP-08 §3.3 sunset dependency is added to the §8 / §11.1 cross-citation re-validation list so an upstream algorithm sunset triggers re-validation of this spec. (Per fix [5], third cycle.)

> **N12-CB-02 (restore MUST verify the commitment before re-establishing any key)**: `restore_vault_key` MUST, **before** re-establishing any key:
>
> - (a) reconstruct the candidate secret via `reconstruct(shards, passphrase)`;
> - (b) recompute the KEK-identity commitment over the **canonical input tuple** `{domain_sep, target_handle's vault_id, the backup's CAPTURED kek_generation (the value carried in the backup blob and bound into the commitment per N12-SG-01), candidate master_secret, passphrase provenance}`, using the hash identified by the backup's recorded `kek_commitment_alg` (N12-CB-04(b)). The recompute generation is the **captured** generation, **never** the vault's authoritative current generation. This commitment authentication is **N12-FT-02 step 7** (the earlier step), which authenticates the captured generation before the ordinal comparison runs. The current-vs-captured ordinal comparison is a separate, **later** job, handled by the staleness gate (N12-SG-02 / N12-FT-02 step 8), which sources the current generation from the audited rotation chain (N12-RT-06). This separation is what makes the forced-stale path (N12-SG-03 / V3(b)) coherent: `force_stale` overrides only the ordinal staleness step (step 8), while this commitment step (step 7) then PASSES for a legitimate superseded backup because it recomputes over the captured generation the commitment was registered against (fix [1]/[14]);
> - (c) compare it to the commitment registered for `target_handle` at the backup's **captured** generation, under the **same** recorded `kek_commitment_alg` the backup recorded (selected from the per-algorithm registry of N12-CB-04(c)), and **reject with `kek-commitment-mismatch`** if it does not equal (or **`commitment-alg-mismatch`** if NO registered commitment exists for `target_handle` at the captured generation under the recorded algorithm, per N12-CB-04(b));
> - (d) **reject with `key-identity-mismatch`** if `target_handle`'s vault identity differs from the identity the shard set captured (cross-vault re-install).
>   A restore that re-establishes a key without passing (b)-(d) is **non-conforming at Conformant level**. This closes arbitrary key-injection (F-CRYPTO-1), the attacker-controlled-passphrase substitution channel (F-CRYPTO-4, since a wrong passphrase yields a different secret and thus a commitment mismatch), and cross-vault re-install (F-CRYPTO-5, F-AUTHZ-3). Exercised by V3(g) (generation-integer tamper whose ciphertexts ARE in the consulted distribution, so step 6 passes and this step-7 commitment check fires -> `kek-commitment-mismatch`, sub-clause c) and V8(b) (cross-vault -> `key-identity-mismatch`, sub-clause d). (Citation corrected per F-DOC-1: the prior draft mis-cited V6(a), which tests positive cross-SDK reproducibility of N12-CB-01, not the rejection MUST of N12-CB-02; the V3(c) citation is repointed to V3(g) per fix [19], because a genuine-old relabelled set on the default path is now refused at step 6 with `unknown-shard`, and the step-7 `kek-commitment-mismatch` defense is exercised by the V3(g) generation-integer tamper whose ciphertexts reach the consulted distribution.)

> **N12-CB-03 (per-deployment foreign-shard check via `shard_commitments`)**: Within a single deployment and generation (single issuing SDK, identical ciphertexts), the binding MUST also maintain and check the EATP-10 **`shard_commitments`** array (SHA-256 of shard ciphertexts). On the SDK-only path this array is sourced from **the valid `vault_key_backup` / `vault_holder_rotation` / `vault_kek_rotation` distribution anchor for `target_handle` in the `recovery` tier that established the generation being restored** (the `shard_commitments` carried in `event_payload` per N12-AU-01/AU-04), not from a `RecoveryRitualRecord` the SDK path never emits; all three of these anchor subtypes carry a `shard_commitments` set (the `vault_kek_rotation` row re-shards and carries its own distribution, fix [3]), and `vault_kek_recommit`/`vault_kek_retire` are deliberately EXCLUDED as sources because they re-shard nothing. When the vault was provisioned via the EATP-10 human ritual, the equivalent `RecoveryRitualRecord(kind=generation)` is the source. For an **ordinary** (in-generation) restore the binding reads the record that established the **current** generation's shard set; for a **forced-stale** restore (N12-SG-03) the foreign-shard check (N12-FT-02 step 6) sources `shard_commitments` from the **CAPTURED (restored, old-generation) distribution** (the anchor that established `restored_generation`), so the genuine old-generation shard set reaches reconstruction rather than failing at `unknown-shard` against the current-generation distribution (fix [2], third cycle). A presented shard whose ciphertext hash is **not** in the applicable (current-generation, or captured-generation under `force_stale`) array MUST be rejected with **`unknown-shard`** (the EATP-10 N8-R-01 code) **before** reconstruction is attempted, including a homogeneous, internally-consistent self-generated foreign set that shares its own SLIP-0039 identifier and therefore never trips the wrapper's mixed-identifier path (F-CRYPTO-2). The `shard_commitments` ciphertext check is the **within-deployment** foreign-shard defense; the KEK-identity commitment (N12-CB-01) is the **cross-SDK / cross-generation** defense. The two are complementary, and the cross-SDK guarantee (§7, V6) rests on the KEK-identity commitment, never on ciphertext hashes (F-XSDK-9). Exercised by V5(c)/(f) and the forced-stale reachability sub-assertion V3(b).

#### 4.4.1 Passphrase provenance

> **N12-PP-01 (passphrase provenance defined and bound)**: The SLIP-0039 passphrase is a key-stretching input to reconstruction, **not** an authenticator, `combine_mnemonics` returns a different MAC-valid secret for every distinct passphrase. The binding MUST define passphrase provenance and MUST bind it into the KEK-identity commitment (N12-CB-01):
>
> - **RECOMMENDED**: derive the passphrase deterministically inside the trusted vault from `(vault_id, kek_generation)` so it is not an attacker-supplied free parameter; the public surface then takes no passphrase.
> - If operator-supplied, the passphrase MUST be referenced by an opaque `passphrase_ref` on the public surface (never raw bytes across the API), MUST be treated as secret, and MUST be excluded from the audit envelope (N12-AU-01). The restore signature MUST accept the provenance handle so passphrase-protected backups are restorable (F-CRYPTO-4).
> - The binding MUST validate passphrase printability (SLIP-0039 requires printable ASCII, code points 32-126) **before** calling the wrapper, surfacing `invalid-passphrase` (§4.6) deterministically rather than as a mapped library `ValueError` (F-XSDK-6).
>   §8 states explicitly that the passphrase is NOT an integrity check and MUST NOT be relied on as authorization.

### 4.5 Audit envelope (decision: audited binding)

Every backup, restore, rotation, KEK-rotation, re-commitment, commitment-retirement, and **clearance-denied attempt** MUST write an EATP audit anchor, dispatcher-mediated, fail-closed before the key goes active. The anchor records ritual parameters, distribution metadata, the commitments, and the generation, **never** shard contents, KEK bytes, data-key bytes, or passphrase.

> **N12-AU-01 (audit anchor required, contents excluded; outcome vs denial schemas separated)**: Every `back_up_vault_key` and `restore_vault_key` invocation that **passes the clearance gate** (N12-CL) MUST write exactly one **outcome anchor** (success or typed-failure). Every invocation **rejected at the clearance gate** MUST write a distinct **denial anchor**. The two payloads are explicitly different schemas (resolving the prior draft's self-contradiction where a denial was told to carry "NO key-ID resolution required" yet also a `vault_id`/commitment field list that does not exist before handle/shard resolution; F-AUDIT-9):
>
> - **Outcome-anchor `event_payload`** (`subtype in {vault_key_backup, vault_key_restore, vault_key_restore_forced_stale, vault_key_restore_raw, vault_kek_rotation, vault_holder_rotation, vault_kek_recommit, vault_kek_retire}`) MUST include the fields listed **for that subtype's row in N12-AU-04**, not one uniform set across all outcome subtypes (fix [5]): a `vault_key_backup` carries the full set (`alg_id`, `{k,n}`, holder distribution per N12-SH-01, shard count, `vault_id`, captured `kek_generation`, the KEK-identity commitment + `kek_commitment_alg` per N12-CB-04, the KCV, `shard_commitments`, the pinned SLIP-0039 parameters incl. `master_secret_bits`, `side_channel_hardened`, principal, and, when configured, approver and Complete-level witness tokens); a `vault_kek_rotation` carries generation-advance AND re-shard distribution fields (holders/shard_count/shard_commitments/the new commitment/slip39_params/`for_cause`, fix [3]/[4]); a `vault_kek_recommit` carries the from-to commitment pair (no re-shard fields); a `vault_kek_retire` carries the retired `(algorithm -> commitment)` pair (no re-shard fields); a restore carries the re-established handle ref and verified commitment (no new holder distribution of its own). The exact required/forbidden set per subtype is pinned in N12-AU-04.
> - **Denial-anchor `event_payload`** (`subtype in {vault_key_backup_denied, vault_key_restore_denied}`) MUST include EXACTLY `{subtype, principal, missing_capability_or_scope, target_handle_ref (opaque, unresolved), timestamp, time_attested}`. `vault_id`, `kek_generation`, the commitments, the KCV, and ritual `{k,n}` are **explicitly OMITTED** (not null-filled, to keep the canonical pre-image deterministic), because a restore denied before handle/shard resolution (V2) has no such values.
> - **Denial-summary `event_payload`** (`subtype = vault_denial_summary`, fix [6]) is the coalesced windowed summary emitted under a denial flood (below). Its schema is pinned in N12-AU-04 and it lands in the `safety` tier alongside the denial anchors it summarizes.
>   No payload may include shard contents, shard mnemonics, KEK bytes, data-key bytes, the passphrase, or any per-holder wrapping secret. This is the binding-surface analogue of EATP-10 §9.1 and inherits the contents-exclusion discipline of `rules/observability.md` MUST Rule 4 cited in `trust/vault/backup.py`. (Per F-AUDIT-3 and F-AUDIT-9.)
>   **Tier routing (normative):** outcome anchors MUST dispatch to the `recovery` tier; **denial anchors and the `vault_denial_summary` MUST dispatch to the `safety` tier** (EATP-09 §4.1's class for constraint violations / BLOCKED verdicts), because unauthorized-access probing is a security-relevant access-control event, not a recovery-ritual record, and routing denials into the indefinitely-sealed `recovery` tier turns a near-unprivileged probe into a seal-flood DoS. Under a denial flood the binding MUST emit a **`vault_denial_summary`** record (N12-AU-04) in lieu of N per-attempt anchors, preserving every distinct probing principal (in-line up to a cap `M`, and completely beyond `M` via the `principal_set_root` digest, fix [8]) and the per-tier `safety` chain link; denial anchors MUST NOT be dropped (V2-denial-flood; F-AUDIT-10). **This is a DISPATCH-TIME aggregation (fix [7]):** when the binding detects a flood it emits ONE summary covering the window at the point of dispatch, in place of N per-attempt anchors; it is NOT the post-hoc, retention-driven EATP-09 §4.1 compaction operation and does NOT rewrite already-stored `safety` anchors. It therefore does not violate EATP-09 §4.1's "Not compactable" classification of `safety`, which forbids collapsing already-stored anchors, not bounding emission rate at dispatch. The summary record CHAINS AFTER the last stored `safety` anchor (its dispatcher-assigned `previous_anchor_hash` is computed over that prior anchor like any other `safety` record), so the per-tier chain link is unambiguous.

> **N12-AU-01a (anchor is a well-formed EATP Audit Anchor; chains within tier)**: The anchor MUST be a well-formed EATP Audit Anchor per `eatp/02-trust-lineage-chain.md` Element 5, carrying `transaction_id`, `lineage_hash`, `outcome`, `log_ref`, and the dispatcher-assigned `previous_anchor_hash` that chains it **within its dispatch tier** (outcome anchors in `recovery`, denial anchors in `safety` per N12-AU-01; per N9-T-02 there is NO cross-tier hash chaining; cross-tier ordering is the global anchor). For an **outcome** anchor, the `vault_id` and `kek_generation` are carried in `event_payload` and bound into `lineage_hash`, so a forced-stale rollback (§6) is chain-ordered against the KEK-generation history and an investigator can prove "this rollback happened after generation g+1 was committed" (F-AUDIT-5). For a **denial** anchor (which has no resolved `vault_id`/`kek_generation`), `lineage_hash` binds the `principal` and the opaque `target_handle_ref` instead (F-AUDIT-9).

> **N12-AU-02 (dispatcher-mediated, recovery-tier, signed per EATP-08, fail-closed; alg_id rides payload)**: Every **outcome** audit anchor MUST be enrolled into the `recovery` tier, and every **denial** anchor into the `safety` tier (N12-AU-01), via **`AuditDispatcher.dispatch(anchor, tier=...)`** per the inherited dispatcher-mediation rule (**N9-D-01**); a direct write to the audit store that bypasses `dispatch()` is **non-conforming** (a non-chained record never reaches a global anchor and is invisible to cross-tier tamper detection, F-AUDIT-1). The named-tier dispatcher is a REQUIRED net-new surface (§3.1 substrate-status note; tracked `kailash-py#630`); a deployment MAY provide an adapter mapping recovery-tier enrollment + `DispatchReceipt` onto the shipped gradient-keyed `dispatch(anchor, level)` until it lands. The `dispatch()` call MUST set `previous_anchor_hash` per D1 so the anchor chains in the recovery tier and is committed by the next global anchor `G_k`.
>
> The anchor MUST be signed under the deployment's current `alg_id`. **Because the substrate pre-image is `content_signing_bytes(event_type, event_payload, signer_delegate_id)` (N12-AU-03) and carries no top-level `alg_id` slot, the deployment `alg_id` MUST be carried as a fixed `event_payload.alg_id` field (pinned first in the §4.5.1 / N12-AU-04 schema) so it is covered by the signed pre-image.** The D3 parseability intent (`alg_id` locatable without scanning the full record) is satisfied by the pinned schema position; the literal JCS-first-field form of D3 does not apply to this substrate-framed surface (F-AUDIT-8). This is the same pattern by which the subtype (N12-AU-03) and the governance-approver token (N12-CL-03(c)) ride `event_payload`.
>
> The `recovery` tier's witness-binding requirement follows D4: dispatcher signature is sufficient at **Conformant**; external witness binding is REQUIRED at **Complete** (N9-G-03). A `dispatch()` with a missing/unrecognized tier is rejected with `unknown-tier` (N9-D-02). Because this binding hard-targets `tier="recovery"`, the EATP-10 `wrong-tier` code is unreachable on this surface and is NOT a live binding error (F-XSDK-11; see §4.6).

> **N12-AU-02a (recovery-tier dispatch-acceptance interlock, no fail-closed brick)**: N12-AU-02b makes a successful dispatch a hard precondition for the KEK going active. EATP-09 §3.4 permits a deployment to restrict `dispatch()` against a sealed tier ("unless the tier policy further restricts"), and the `recovery` tier is sealed indefinitely (EATP-09 §4.1). To prevent a conformant EATP-09 deployment from permanently bricking every restore (dispatch restricted -> N12-AU-02b aborts forever), the `recovery` tier MUST accept `dispatch()` of vault key-binding anchors **despite** its indefinite seal, i.e., the EATP-09 §3.4 "further restricts" escape MUST NOT be exercised against the `recovery` tier for vault-binding anchors. The indefinite seal blocks `rotate()`/compaction, **not** `dispatch()` (append-only is not append-prohibited). (Per F-LIVE-1; cross-referenced to the EATP-09 erratum flag in §10.) Exercised by V7(d).
>
> **`safety`-tier extension (fix [9]).** The finalization routed multiple MUST-write paths into the `safety` tier: denial anchors and the `vault_denial_summary` (N12-AU-01), forced-stale dual-emit (N12-SG-03), denylist dual-emit (N12-SG-05), and raw-restore dual-emit (N12-IN-03). EATP-09 §4.1 classifies `safety` as "Append-only, sealed at rotation," and §3.4 grants the same per-deployment "further restricts" right against any sealed tier. The EATP-09 §3.4 "further restricts" escape MUST NOT be exercised against the `safety` tier for these vault-binding anchors either. Because `safety` is "sealed at rotation" (transient) rather than "sealed indefinitely" like `recovery`, the `safety` case is framed as **bounded-degradation**: a `safety`-seal-induced dispatch failure on a required vault path MUST be treated as a **retryable operational incident** per N12-AU-02b (surfaced to the operator as the cause), NOT as a permanent brick and NOT as a silent drop. The `safety` dual-emit of N12-SG-03/N12-SG-05/N12-IN-03 is a fail-closed precondition for the KEK going active (it shares N12-AU-02b's fail-closed ordering); a `safety`-seal failure on that path triggers the same retryable bounded-degradation, never a silent gap and never a permanent brick. The N12-AU-01 "denials MUST NOT be dropped" guarantee likewise survives a `safety` restriction as a retryable incident, not a drop. Exercised by V7(f).

> **N12-AU-02b (audit is a precondition for the KEK going active, fail-closed ordering)**: `restore_vault_key` MUST NOT return an active key handle, and `back_up_vault_key` MUST NOT return shards, until the corresponding audit anchor has been **successfully dispatched** (a `DispatchReceipt` is returned) per N12-AU-02. If the dispatch fails, the operation MUST abort: for restore, the reconstructed KEK MUST NOT be installed as active and MUST be `del`-ed/discarded per N12-IN-05; for backup, the shard set MUST NOT be released. An audit-dispatch failure is a fail-closed condition (analogous to N9-G-04). There is no "key recovered, no anchor" outcome. **Audit-outage bounded degradation (per F-LIVE-2):** an audit-tier _outage_ (the dispatcher transiently unavailable) is an operational incident, not a permanent vault brick, the restore aborts safely (fail-closed) and MAY be retried once the tier is available; the vault is not destroyed, and the operator MUST be surfaced the outage as the cause rather than a cryptographic failure. The fail-closed guarantee is "no key without an anchor," not "one dispatch failure bricks the vault forever." Exercised by V7(c).

> **N12-AU-03 (canonical pre-image, mapped onto the closed event-type enum; subtype namespace reserved)**: The audit-anchor signed pre-image MUST be the canonical UTF-8 bytes produced by the substrate's `content_signing_bytes(event_type, event_payload, signer_delegate_id)` routine (`delegate/audit.py:79`), covering `event_type` + `event_payload` + `signer_delegate_id` under the substrate canonical form (`canonical_json_dumps`, RFC 8785 / JCS-equivalent) per inherited D3. **The substrate `DelegateEventType` is a closed, founder-ratified, rs-canonical M4 enum (5 variants) gated by `_AUDIT_VISIBLE_EVENT_TYPES`; `AuditChainEngine.emit_event` raises `AuditChainEmissionError` for any unknown event_type. A spec MUST NOT mint new top-level event-type strings.** Therefore the vault events MUST be carried as the canonical variant **`EXTERNAL_SIDE_EFFECT`** with the discriminator in `event_payload["subtype"]` (the documented M4 migration pattern, `audit.py:222-240`):
>
> | `subtype`                                              | Meaning                                                                                 |
> | ------------------------------------------------------ | --------------------------------------------------------------------------------------- |
> | `vault_key_backup`                                     | a backup                                                                                |
> | `vault_key_restore`                                    | an ordinary restore                                                                     |
> | `vault_key_restore_forced_stale`                       | a forced stale restore (§6)                                                             |
> | `vault_key_restore_raw`                                | a raw-bytes-escape-hatch restore (§4.1)                                                 |
> | `vault_kek_rotation`                                   | a KEK rotation advancing the generation (§5.2)                                          |
> | `vault_kek_recommit`                                   | a commitment-algorithm advance (N12-CB-04(c)), generation unchanged                     |
> | `vault_kek_retire`                                     | a commitment-algorithm retirement (N12-CB-04(e)), generation unchanged                  |
> | `vault_holder_rotation`                                | a holder rotation (§5.1)                                                                |
> | `vault_key_backup_denied` / `vault_key_restore_denied` | a clearance-gate denial (N12-AU-01)                                                     |
> | `vault_denial_summary`                                 | a coalesced windowed summary of denial anchors under flood (N12-AU-01, V2-denial-flood) |
>
> **Subtype-namespace reservation (per F-XSDK-12).** `EXTERNAL_SIDE_EFFECT` carries a _shared_ subtype discriminator, and the substrate already reserves the namespace `{dispatch_invocation, cascade_emission, lifecycle_transition, posture_ratchet, sovereign_handover}` (`delegate/audit.py:228-236` migration map) for connector-dispatch and M4 lifecycle events. EATP-12 reserves the **`vault_` prefix** as a disjoint namespace: every vault subtype MUST begin `vault_` and MUST NOT equal any substrate-reserved subtype. Because `AuditChainEngine.emit_event` validates ONLY `event_type` (against `_AUDIT_VISIBLE_EVENT_TYPES`, `audit.py:888-895`) and performs **no `subtype` validation**, the **binding** (not the engine) MUST validate `event_payload["subtype"]` against the closed `vault_*` set before emit, raising a typed error on an unrecognized or colliding subtype, so the V6 golden-fixture byte-identical pre-image is defined over a reserved, validated discriminator rather than an unguarded field. A conformant verifier selecting vault records MUST match on `event_type == EXTERNAL_SIDE_EFFECT` AND `subtype in {vault_*}`; subtype filtering without the `vault_` namespace contract is not a sound vault-record selector.
>
> Because `EXTERNAL_SIDE_EFFECT` is in `_AUDIT_VISIBLE_EVENT_TYPES`, the C3 audit-visibility classifier retains these events; they are not silently dropped (F-AUDIT-7 resolved by binding to the real classifier rather than asserting membership in a phantom set). The canonical form of each `event_payload` is pinned by §4.5.1 and the cross-SDK vectors in §7 so two conformant implementations produce byte-identical signed pre-images for identical inputs. **Alternative path**: a deployment MAY instead coordinate a ratified amendment adding dedicated `DelegateEventType` variants on **both** SDKs plus both frozensets; until such an amendment lands, the `subtype` mapping is the only conformant path and a binding that mints bare unsanctioned event-type strings is non-conforming (F-AUTHZ-8).

#### 4.5.1 Canonical payload schema (cross-SDK byte-identical)

JCS canonicalizes a _given_ object's key order and number form; it does not reconcile two differently-_shaped_ objects (F-XSDK-3). The binding therefore pins the `event_payload` shape.

> **N12-AU-04 (pinned per-subtype payload schema + golden fixture)**: The `event_payload` MUST use exactly the field names, JSON types, and encodings listed **for that subtype's row below**, not one uniform "exactly these" set across all subtypes (fix [5]; the prior uniform schema was unsatisfiable for read-only restores, re-commitments, retirements, and denials, which carry different fields than a backup; a KEK rotation carries a backup-like re-shard distribution PLUS generation-advance fields per the third-cycle fix [3]). The shared field encodings are pinned once here, and each subtype's row then names the required and forbidden set. Every payload carries `subtype` (a `vault_*` string, N12-AU-03) and the two-state `timestamp` + `time_attested` grammar below.
>
> **Shared field encodings (where a field appears, it is encoded this way):**
>
> - `alg_id` as a fixed string (the deployment algorithm, pinned first per N12-AU-02);
> - `{"k": <int>, "n": <int>}` threshold (matching EATP-10 §9.1);
> - `holders` as an ordered JSON array of stable holder-ID **strings** (registry IDs; `name_hint`/`contact_hash` excluded, attribution is by registry ID only);
> - `shard_count` as a JSON integer;
> - `kek_generation` as a JSON **integer** (strike "or equivalent total order"; an integer is the only portably-comparable form, F-XSDK-5);
> - `vault_id` as a fixed string;
> - `kek_identity_commitment` and `shard_commitments[]` as lowercase-hex strings;
> - `kek_commitment_alg` as a fixed string (N12-CB-04, the registry token, default `"eatp-v1"`, fix [11]);
> - `kcv` as a lowercase-hex string of **length 16 (8 bytes), per N12-CB-04(d)** (fix [10]);
> - `slip39_params` as `{"extendable": true, "iteration_exponent": <int>, "group_threshold": 1, "master_secret_bits": <128|256>}` (F-XSDK-10; `master_secret_bits` added so a 128-vs-256 divergence surfaces as `parameter-mismatch`; `(k,n)` is NOT duplicated here, it lives in the sibling `{"k","n"}` field);
> - `side_channel_hardened` as a JSON boolean (N12-CRY-SC);
> - `timestamp` + `time_attested` as a **two-state grammar** (fix [13]): `time_attested` is ALWAYS-present JSON boolean (`true` when sourced from the trust-anchored time source of EATP-10 §14 per N12-AU-04a, `false` when unavailable). When `time_attested` is `true`, `timestamp` is RFC 3339 UTC to **second** precision with trailing `Z`; when `time_attested` is `false`, `timestamp` is the single fixed sentinel string `"unverified"` (NOT omitted, NOT null). A verifier MUST treat `timestamp` as either an RFC3339-Z value or the sentinel per `time_attested`;
> - principal/approver/witness as fixed string IDs.
>
> **Per-subtype required/forbidden field sets:**
>
> - **`vault_key_backup`** (full set): `alg_id`, `{k,n}`, `holders`, `shard_count`, `vault_id`, `kek_generation`, `kek_identity_commitment`, `kek_commitment_alg`, `kcv`, `shard_commitments`, `slip39_params`, `side_channel_hardened`, `principal`, `timestamp`+`time_attested`, and (when N12-CL-03 configured) `approver` + signed approval token, and (at Complete per N12-CL-05) `witness` + signed witness token.
> - **`vault_key_restore` / `vault_key_restore_raw`** (ordinary read operations; fix [18], fix [6]): `alg_id`, `re_established_handle_ref` (the opaque re-established handle, never KEK bytes), `vault_id`, `kek_generation`, `generation_checked` (the generation the restore verified against; on the ordinary path this equals `kek_generation`, because an in-generation restore verifies against the same generation it re-establishes, and the two fields are kept distinct because the forced-stale row below makes them diverge, fix [7]), `kek_identity_commitment`, `kek_commitment_alg`, `principal`, `timestamp`+`time_attested`. `holders`, `shard_count`, and `shard_commitments` **MUST ALWAYS be present and MUST be copied from the current-generation distribution recorded in the latest valid `vault_key_backup`/`vault_holder_rotation`/`vault_kek_rotation` distribution anchor for `target_handle`** (the same record N12-CB-03 reads at restore time), NOT the presenting k-shard subset, so the restore anchor is deterministic and verifier-unambiguous across SDKs (the unconditioned "when present" hedge is removed so the rule and the §12.8 fixture cannot diverge, fix [6]). All three of `vault_key_backup`, `vault_holder_rotation`, and `vault_kek_rotation` are valid distribution sources because each now carries a `shard_commitments` set (the rotation row below carries its own re-shard distribution, fix [3]). A restore anchor generates NO new holder distribution of its own.
> - **`vault_key_restore_forced_stale`** (forced-stale rollback; its OWN row, separated from the ordinary read operations, fixes [2]/[7]): `alg_id`, `re_established_handle_ref`, `vault_id`, `kek_generation`, `generation_checked`, `restored_generation` (int), `overridden_current_generation` (int), `kek_identity_commitment`, `kek_commitment_alg`, `holders`, `shard_count`, `shard_commitments`, `principal`, `timestamp`+`time_attested`. The two **net-new, distinctly named** generation fields `restored_generation` (the captured/materialized OLD generation actually installed by the rollback) and `overridden_current_generation` (the current generation the rollback overrode) are carried IN ADDITION to (not a reuse of) `kek_generation`/`generation_checked` (fix [7]). The canonical mapping is exactly one: `kek_generation == generation_checked == restored_generation` (the commitment authenticates over the captured/restored generation, N12-CB-02(b), and the key re-established is that same captured generation), and `overridden_current_generation > restored_generation` (the rollback went backward). `holders`, `shard_count`, and `shard_commitments` MUST be present and MUST be copied from the **CAPTURED (restored, old-generation) distribution** actually used, sourced from the `vault_key_backup`/`vault_holder_rotation`/`vault_kek_rotation` distribution anchor that established `restored_generation`, NOT the current-generation distribution, so the forced-stale anchor records the shard set the restore actually reconstructed from (fix [2]). The materialized fixture is Appendix B §12.11.
> - **`vault_kek_rotation`** (generation advance + re-shard, §5.2 / N12-RT-06; fixes [3]/[4]): `alg_id`, `prior_kek_generation` (int), `kek_generation` (the new generation, int), `vault_id`, the new `{"k":<int>,"n":<int>}` ritual, the new `holders` distribution (ordered registry-ID array), `shard_count`, the new `shard_commitments`, the new `kek_identity_commitment` + `kek_commitment_alg` (the re-shard registers the new generation's KEK-identity commitment), the pinned `slip39_params`, `side_channel_hardened`, `for_cause` (boolean; `true` when the rotation was escalated from a for-cause holder revocation per N12-SH-04, `false` for an ordinary KEK rotation), `principal`, `timestamp`+`time_attested`. A KEK rotation advances the generation AND re-shards under a fresh ritual, so it legitimately establishes the new generation's distribution; this is exactly why N12-CB-03 names it as a `shard_commitments` source and why a restore at `g+1` can find a distribution of record immediately (fix [3]). **No** `kcv` on this subtype unless re-derived. (The prior draft's "Explicitly NO holders/shard_count/shard_commitments ... recorded by the subsequent backup" exclusion is struck: a KEK rotation does carry its own re-shard distribution, fix [3].)
> - **`vault_holder_rotation`** (§5.1 / N12-RT-02): `alg_id`, old and new `{k,n}` (encoded `{"old":{"k","n"},"new":{"k","n"}}`), `departing_holder` (string ID), the new `holders` distribution, `vault_id`, `kek_generation` (unchanged by amicable rotation, N12-RT-03), the new `shard_commitments`, `for_cause` (boolean; `false` for the amicable rotation this row records; a for-cause departure is NOT recorded as an amicable holder rotation but is escalated to a generation-advancing `vault_kek_rotation` carrying `for_cause=true` per N12-SH-04, so the for-cause flag lands on the anchor that actually advances the generation, fix [4]), `principal`, `timestamp`+`time_attested`. **No** `kcv`/`side_channel_hardened` unless re-derived (align with N12-RT-02, fix [5]).
> - **`vault_kek_recommit`** (commitment-algorithm advance, N12-CB-04(c); fix [12]): its own schema carrying the from-to pair, EXACTLY `alg_id`, `vault_id`, `kek_generation` (stated **unchanged**), `prior_kek_commitment_alg`, `prior_kek_identity_commitment`, `new_kek_commitment_alg`, `new_kek_identity_commitment`, `principal`, `timestamp`+`time_attested`. **Explicitly NO** `holders`, `shard_count`, `shard_commitments`, `slip39_params`, `kcv` (a recommit re-shards nothing; this is why N12-CB-03 deliberately EXCLUDES `vault_kek_recommit` as a `shard_commitments` source). The two-commitment from-to pair is intrinsic to a re-commitment and cannot be expressed by a single `kek_identity_commitment` field; the recommit row carries both.
> - **`vault_kek_retire`** (commitment-algorithm retirement, N12-CB-04(e); fix [5, third cycle]): its own schema marking a specific `(algorithm -> commitment)` registry entry non-verifiable, EXACTLY `alg_id`, `vault_id`, `kek_generation` (the generation whose commitment entry is retired, stated **unchanged** by the retirement), `retired_kek_commitment_alg` (the registry token being retired), `retired_kek_identity_commitment` (the specific commitment hex being retired), `principal`, `timestamp`+`time_attested`. **Explicitly NO** `holders`, `shard_count`, `shard_commitments`, `slip39_params`, `kcv` (a retirement re-shards nothing). The retired entry MUST already be superseded by a live, non-retired strong-algorithm commitment for the same `(vault_id, kek_generation)` (a prior `vault_kek_recommit` to `C_Y`) so the retirement closes the weak path without stranding recoverability (N12-CB-04(e)(4)).
> - **`vault_denial_summary`** (windowed denial summary under flood, N12-AU-01 / V2-denial-flood; fix [6]): EXACTLY `{"subtype": "vault_denial_summary", "window_start": <RFC3339-Z>, "window_end": <RFC3339-Z>, "distinct_principals": <ordered JSON array of stable principal-ID strings, sorted ascending, capped at M, fix [8]>, "distinct_missing_capabilities": <ordered JSON array of strings, sorted ascending>, "principal_set_root": <lowercase-hex sorted-hash/Merkle digest over the FULL distinct-principal set>, "coalesced_count": <int>, "timestamp"+"time_attested"}`. Ordering (principals and capabilities sorted ascending) is pinned so the JCS canonical form is byte-deterministic across SDKs. Beyond the in-line cap `M`, complete attribution is preserved by `principal_set_root` (any claimed principal is provable present/absent against the root) so the record size is bounded `O(M)` regardless of distinct-principal cardinality (fix [8]).
>
> **Denial subtypes** (`vault_key_backup_denied`/`vault_key_restore_denied`): EXACTLY `{"subtype": <string>, "principal": <string>, "missing_capability_or_scope": <string>, "target_handle_ref": <opaque string>, "timestamp"+"time_attested" (two-state grammar above)}`, no `vault_id`, `kek_generation`, commitments, KCV, or ritual fields (N12-AU-01). The golden fixture (Appendix B) ships a denial-subtype pre-image (both the attested and the `unverified` form) so the "byte-identical for every subtype" claim is testable (F-AUDIT-9, fix [13]).
>
> §7's V6 ships a **materialized golden fixture (Appendix B)**: the expected canonical pre-image bytes (hex) for a fixed input, which both SDKs MUST reproduce exactly. Appendix B materializes the `vault_key_backup`, `vault_key_restore`, `vault_kek_rotation` (now carrying its re-shard distribution, fix [3]), `vault_kek_recommit`, `vault_key_restore_forced_stale` (§12.11, fixes [2]/[7]), `vault_denial_summary`, and denial (attested + `unverified`) pre-images so the byte-identical claim is testable for the finalization-introduced and third-cycle divergent subtypes, not just backup (fix [5]/[12]/[13]/[18] and third-cycle [2]/[3]/[7]). Without a golden fixture the "byte-identical" claim is untestable; Appendix B supplies it.

> **N12-AU-04a (trust-anchored anchor timestamps for incident-response anchors)**: The anchor `timestamp` of the forced-stale anchor (N12-SG-03) and the compromised-generation denylist anchor (N12-SG-05), the two incident-response-critical events whose absolute time an investigator relies on, MUST be sourced from the trust-anchored time source EATP-10 §14 mandates for the cooling-off comparison (the same trust-anchor that signs the recovery-path timestamp), which this binding extends to vault anchor timestamps, not the caller's or binding host's local wall clock. The hash chain (`previous_anchor_hash`, N12-AU-01a) already protects _ordering_; this clause protects the absolute _value_ so a backdated/forward-dated rollback cannot misdirect incident response. Where trust-anchored time is unavailable, the anchor MUST set `time_attested:false` and record `timestamp` as the fixed sentinel `"unverified"` per the N12-AU-04 two-state grammar (fix [13]), rather than a forgeable host time. (Per F-TEMP-3, with the §14 attribution corrected per fix [16].) Exercised by V7(e).

The binding surface MUST surface typed errors, not generic cryptographic failures, and MUST reuse the Published EATP-10 codes where the condition is the same (no parallel vocabulary; F-BND-4, F-XSDK-4).

> **N12-FT-01 (typed failure taxonomy)**: The binding MUST distinguish at least the following typed error conditions, each with a stable, distinct error identifier:

> **Note on `wrong-tier` (per F-XSDK-11):** the EATP-10 N8-A-01 `wrong-tier` code references a non-existent `info` tier and is a behavior the EATP-09 `dispatch()` contract has no machinery to produce. Because this binding **hard-targets `tier="recovery"`**, `wrong-tier` is **unreachable on this surface** and is therefore NOT a live binding error; only `unknown-tier` (from misconfiguration) can fire. The dangling upstream defect is tracked for an EATP-09/EATP-10 erratum (§10).
>
> An implementation MUST NOT collapse these into a single generic error. `insufficient-shards` (quorum), `corrupted-shard` (integrity), `unknown-shard` (foreign/old-generation), `kek-commitment-mismatch` (injection), `commitment-alg-mismatch` (hash migration), `stale-generation` (policy), and `revoked-generation` (denylist) have different operational responses and MUST be distinguishable by the caller. In particular, foreign-shard rejection MUST surface as `unknown-shard` (the commitment-check code), **not** as `corrupted-shard`, a MAC-valid foreign shard is not corrupted (F-XSDK-4, F-BND-4). Mapping these typed errors onto the wrapper's underlying `ValueError`/`TypeError`/`RuntimeError` is the binding's responsibility; the wrapper's exception text MUST NOT be the only signal.

> **N12-FT-02 (over-supply is deterministic, not SDK-dependent; pre-reconstruction gate ordering pinned)**: The shipped `reconstruct` (via `combine_mnemonics`) requires **exactly** `k` shards and raises on over-supply. The binding MUST make over-supply deterministic across SDKs: it MUST EITHER (a) reject `> k` shards with `too-many-shards`, OR (b) canonically trim to a deterministic k-subset (first-k by holder-ID ascending) before calling `reconstruct`. The choice MUST be stated in deployment conformance metadata so py and rs cannot diverge (one erroring, one silently trimming; F-XSDK-2).
>
> Because two pre-reconstruction checks (`too-many-shards` count and `unknown-shard` foreign-set) could each fire first on an over-supplied foreign set and yield different typed codes across SDKs (F-XSDK-13), the binding MUST apply this **canonical first-failing gate order** so two conformant SDKs return the same first code: (1) clearance; (2) handle-type (`not-a-kek`); (3) shard-count (`insufficient-shards`/`too-many-shards`) **or** canonical trim; (4) parameter check (`parameter-mismatch`); (5) mixed-identifier (`mixed-shard-set`); (6) commitment/foreign-shard check (`unknown-shard`); (7) commitment/identity authentication (`kek-commitment-mismatch`/`commitment-alg-mismatch`/`retired-commitment-alg`/`key-identity-mismatch`); (8) ordinal generation/denylist (`stale-generation`/`revoked-generation`). Under the reject branch a k+1 foreign set fails at step (3) with `too-many-shards`; under the trim branch it trims at (3) then fails at (6) with `unknown-shard`. The parallel V5(f) `mixed-shard-set`/`unknown-shard` ambiguity resolves under the same ordering (step 5 before step 6). The first draft's V1 claim that "restore from any 3, 4, or 5 shards succeeds" is **false** against the reference wrapper and is corrected in V1.
>
> **Ordering note on steps (7) and (8) (commitment authenticates the captured generation FIRST, then ordinal staleness; fix [1]/[14], reconciled third cycle).** The numeric order and the prose now agree literally: authentication precedes ordinal comparison. Step (7), the KEK-identity commitment/identity check, recomputes the commitment over the backup's **captured** generation (N12-CB-02(b)) and so **authenticates** the generation label for any set that reaches it: a generation-integer tamper whose ciphertexts ARE in the consulted distribution (so it passed the step-6 foreign-shard check, e.g. a genuine current-generation blob relabelled to another generation) surfaces here as `kek-commitment-mismatch`, never as a silent pass, and a retired/unregistered algorithm surfaces as `retired-commitment-alg`/`commitment-alg-mismatch`. A genuine-old-generation relabelled set does NOT reach step (7) on the default path: its ciphertexts are not in the current-generation distribution the step-6 foreign-shard check consults, so it is refused earlier at step (6) with `unknown-shard` (N12-SG-02 / N12-CB-03; fix [19]). Step (8), the ordinal generation/denylist comparison, then operates on the **already-authenticated** captured generation, comparing it against the vault's current generation (N12-RT-06) and surfacing `stale-generation`/`revoked-generation`. The two checks are distinct concerns over distinct generation values: step (7) over the **captured** generation (integrity of the label and secret), step (8) over **captured-vs-current** (ordinal currency). Because authentication is the earlier step, "authenticate the captured generation first, then ordinal-compare" holds literally, and N12-SG-02's "MUST NOT trust a caller-supplied generation in isolation" is satisfied at the step-6/step-7 authentication steps. `force_stale` (N12-SG-03) overrides **only** step (8); steps (6) and (7) are never overridden by it, though for a forced-stale rollback the step-6 foreign-shard source is the CAPTURED (old-generation) distribution rather than the current one (N12-CB-03), so a legitimate superseded backup PASSES step (6) against its captured distribution and PASSES step (7) (it recomputes over the captured generation the commitment was registered against) and the forced rollback proceeds, while a relabelled or injected blob whose ciphertexts reach step (7) still fails there and deterministically returns `kek-commitment-mismatch` across all SDKs. An implementation MUST NOT collapse these into one check or recompute the commitment over the current generation; doing so reproduces the circular dependency this note forbids.

> **N12-FT-03 (registry-write-path first-failing gate order; fix [8], third cycle)**: N12-FT-02 pins the order for the **restore** path only. The registry-writing operations the spec adds (`vault_kek_recommit`, `vault_kek_retire`, and the for-cause-escalation `vault_kek_rotation`/amicable `vault_holder_rotation`) perform their own validations and can fail on multiple independent conditions, so the binding MUST pin a canonical first-failing gate order for the write paths too, or two conformant SDKs could surface different first errors for the same malformed write (the F-XSDK-13 determinism class, now closed on the write paths):
>
> - **`vault_kek_recommit`**: (1) clearance/capability + tenant-domain (N12-CL-01/CL-02a) + (if cooling-off) N12-CL-04; (2) generation/`vault_id`-unchanged invariant (`recommit-generation-altered` if `kek_generation` or `vault_id` is altered, N12-CB-04(c)); (3) prior-commitment-exists-in-registry-under-`prior_kek_commitment_alg` (`unknown-prior-commitment` if no live entry matches); (4) new-commitment-binds-the-true-secret-and-same-`(vault_id, kek_generation)` (`recommit-binding-mismatch` otherwise).
> - **`vault_kek_retire`**: (1) clearance/capability (`vault:retire-alg` or the governance-approver HELD action, N12-CB-04(e)(2)) + tenant-domain; (2) generation/`vault_id`-unchanged invariant (`recommit-generation-altered`); (3) retired-entry-exists-in-registry (`unknown-prior-commitment` if the named `(retired_kek_commitment_alg -> retired_kek_identity_commitment)` matches no live entry); (4) a live, non-retired strong-algorithm commitment already exists for the same `(vault_id, kek_generation)` per N12-CB-04(e)(4) (else the retirement MUST be refused with `missing-clearance`-class rejection, never stranding recoverability).
> - **`vault_kek_rotation`** (for-cause or ordinary) and **`vault_holder_rotation`**: reuse the N12-CL-01 clearance/tenant-domain gate and the N12-TH-01 floor gate, in that order (clearance first, then floor), then write the new distribution; these are mostly cross-references since those gates already exist for the rotation surface (N12-RT-01).
>
> Two conformant SDKs MUST therefore return the same first typed error for the same malformed write. (Per fix [8], third cycle.)

## 5. Rotation trigger

When a holder departs, the vault rotates the shard set under a new ritual using the **shipped** `rotate_holders` wrapper. This section binds that wrapper to the authority and audit surface; it does not redefine the wrapper's behavior. It also specifies the **KEK-rotation** event that advances the generation, which the stale-generation guard (§6) depends on.

### 5.1 Holder-rotation binding

> **N12-RT-01 (rotation binds the shipped wrapper)**: On holder departure, the vault MUST perform holder rotation by calling the shipped `shamir.rotate_holders(old_shards, new_ritual, *, passphrase=...)`, which reconstructs the master secret from `old_shards` and regenerates `new_ritual.total_shards` shards under the new `(k, n)`. The binding MUST NOT reimplement reconstruction-then-regeneration; it composes the wrapper. The intermediate secret is `del`-ed best-effort by the wrapper (residence minimization, not cryptographic zeroization, §8). Rotation MUST require the `vault:rotate` token in the caller's bound `RoleScope.capabilities`, evaluated per N12-CL-01/CL-02a (substituting `vault:rotate`, tenant/domain-scoped); a caller lacking it MUST be rejected with `missing-clearance`. The new ritual MUST satisfy the N12-TH-01 floor.

> **N12-RT-02 (holder-rotation audit)**: An **amicable** holder rotation MUST write an audit anchor with `subtype = vault_holder_rotation` to the `recovery` tier per N12-AU-02 (dispatcher-mediated, fail-closed per N12-AU-02b), recording the old and new ritual parameters `{"k","n"}`, the departing holder identity, the new holder distribution, the `vault_id`, the `kek_generation` (unchanged by amicable holder rotation, N12-RT-03), the new `shard_commitments`, and the `for_cause` boolean (set **`false`** for this amicable-rotation audit). The anchor MUST NOT include shard contents or the master secret. A **for-cause** departure is NOT recorded as a `vault_holder_rotation`: per N12-SH-04 it is escalated to a generation-advancing `vault_kek_rotation` carrying `for_cause = true` and the new distribution, so the `for_cause = true` flag rides the anchor that actually advances the generation (fix [4], third cycle).

> **N12-RT-03 (old-ritual shards no longer reconstruct the active set)**: After a holder rotation the shipped wrapper regenerates shards under a fresh SLIP-0039 generation. The binding MUST treat the new shard set as the only valid set for subsequent restores: a restore presenting shards from the **old** ritual MUST be rejected via the `shard_commitments` check (N12-CB-03), surfaced as **`unknown-shard`** (old-generation shard not in the current commitments), or `revoked-holder` when the rejection is attributable to a departed holder's wrapped shard, or `mixed-shard-set` when an old-ritual shard is mixed with new-ritual shards. The binding MUST NOT allow an old-ritual shard set to reconstruct the active KEK after rotation **on the default (non-forced) path**. The single sanctioned exception is an explicit forced-stale rollback (N12-SG-03), where the operator-authorized restore of a superseded generation sources the `shard_commitments` foreign-shard check from the **CAPTURED** old-generation distribution rather than the current one (N12-CB-03), so the genuine old set reaches reconstruction; this is the deliberate, loud, higher-privilege path, not an unguarded bypass (fix [2], third cycle). **Boundary note (per F-BND-6):** the _outcome_ is normative; the SLIP-0039 mechanism by which a fresh identifier is emitted is an upstream-library property and is recorded as a non-normative note, not an EATP-12 MUST.

The `kek_generation` is **not** advanced by an amicable holder rotation (the master secret is unchanged; only the shard distribution changes), **except** when the rotation is triggered by a for-cause departure, which N12-SH-04 escalates to a generation-advancing KEK-rotation. Generation otherwise advances only when the **KEK itself** is rotated (§5.2).

### 5.2 KEK rotation (generation advance) and its audit

> **N12-RT-06 (KEK-rotation event advances the generation and MUST be audited)**: Advancing the vault `kek_generation` counter from `g` to `g+1` is a distinct operation (a KEK rotation, or a for-cause holder revocation per N12-SH-04). Every advance MUST emit an audit anchor with `subtype = vault_kek_rotation` to the `recovery` tier per N12-AU-02 (dispatcher-mediated, fail-closed, chained within tier), recording the prior generation, the new generation, the principal, the trust-anchored timestamp, AND the new generation's re-shard distribution (the new `{k,n}` ritual, `holders`, `shard_count`, `shard_commitments`, the new `kek_identity_commitment` + `kek_commitment_alg`, the pinned `slip39_params`, `side_channel_hardened`), plus the `for_cause` boolean (`true` only when the rotation was escalated from a for-cause holder revocation per N12-SH-04, `false` for an ordinary KEK rotation), per the N12-AU-04 `vault_kek_rotation` row. **A `vault_kek_rotation` re-shards under a fresh ritual and so establishes the new generation's distribution directly on this anchor (fix [3]); a restore at the new generation MUST find this distribution-of-record before it is permitted, and a restore at a generation whose `vault_kek_rotation` (or `vault_key_backup`/`vault_holder_rotation`) distribution anchor is absent MUST fail closed with `unknown-shard` rather than silently skipping the foreign-shard gate (N12-CB-03).** **The current-generation value consulted by the stale-generation guard (N12-SG-02) MUST be derivable from / verifiable against this signed, dispatcher-mediated anchor chain, not from a locally-mutable counter an attacker can edit.** Mirroring EATP-08 D2c (no trust in mutable adoption markers) and EATP-10 §14 (trust-anchored cooling-off clock), a generation advance with no `vault_kek_rotation` anchor is non-conforming, and a generation-rollback attempt (lowering the current generation to make a stale backup look current) MUST fail closed. The `kek_generation` integer is the REQUIRED net-new key-manager surface of §3.4 (it is not present in shipped `KeyMetadata`, which carries only the `rotated_from` lineage string); tracked under `kailash-py#630`. (Per F-AUDIT-6, F-BND-5, and the substrate disclosure F-SUBSTRATE-1.) Exercised by V3(d).

### 5.3 Mode A only

> **N12-RT-04 (Mode A rotation only; Mode B deferred)**: Rotation under this spec is single-group (Mode A) full re-shard, consistent with the shipped wrapper's `group_threshold=1, groups=[(k, n)]` configuration. Per-shard granular rotation (Mode B / multi-group) is **deferred** by `eatp/10-shamir-recovery.md@v1.0` §4.3 (per F32) and inherited here. An implementation MUST NOT claim Mode B rotation conformance on the basis of the shipped wrapper; the wrapper does not support multi-group, and the reserved multi-group extension point in the wrapper's module docstring is explicitly out of scope for this spec.

### 5.4 Restore-as-recovery triggers D6 by reference (no carve-out)

> **N12-RT-05 (any KEK-materializing restore triggers D6)**: **Any** `restore_vault_key` that materializes/reconstructs the KEK triggers the post-recovery posture consequences **D6** by reference to `binding D6 from eatp/10-shamir-recovery.md@v1.0` §8 (N8-PO-01...03): the principal's effective posture is downgraded to SUPERVISED across all five dimensions and the 7-day cooling-off interval starts. **The first draft's carve-out, "a restore that only re-wraps data keys without re-establishing signing authority does not trigger D6", is removed (F-BND-3, F-CRYPTO-6).** EATP-10 ties the downgrade to a successful _reconstruction_, not to whether signing authority was re-established; a data-key-only carve-out would let an adversary frame an induced recovery as a "re-wrap" to dodge the downgrade. If a genuinely benign exemption is ever needed, it belongs in EATP-10 §8 (which already carries the N8-PO-04 override path with its own dual-emit audit), cited, not minted in this binding spec. This spec MUST NOT redefine D6; it triggers it. Implementations claiming Conformant level MUST drive the D6 write path via the PostureStore (`eatp/11-posture-store.md@v1.0`; cross-tested by `eatp/11-posture-store.md@v1.0 V8 (post-recovery-write-path)`), and MUST apply the capability-side cooling-off suspension of N12-CL-04.

## 6. Stale-generation guard (decision 4)

This is the production-biting failure mode. Every backup is tagged with the KEK generation it captured; the tag is **bound** into the commitment so a relabel cannot defeat it; a restore refuses a stale, relabelled, or denylisted generation. The `kek_generation` counter and its KEK-class metadata are REQUIRED net-new key-manager surfaces (§3.4), not yet present in shipped code.

> **N12-SG-01 (generation tagging, bound not merely co-located)**: Every backup produced by `back_up_vault_key` MUST be tagged with the `kek_generation` of the KEK it captured, a **monotonic unsigned integer** advanced only by an audited KEK-rotation event (N12-RT-06). The tag MUST be (a) recorded in the backup's audit envelope (§4.5) and embedded in the `BackupReceipt`'s serialized backup blob under the fixed field `kek_generation:int` so a backup carried between SDKs is self-describing (F-XSDK-5), AND (b) **bound into the KEK-identity commitment** (N12-CB-01) so the _true_ generation a shard set captured is recoverable from the cryptographic binding, not merely from a co-located label. The tag MUST NOT encode the KEK bytes.

> **N12-SG-02 (stale restore refused by default; relabel-proof; foreign-shard precedence, then commitment authentication, then ordinal staleness)**: `restore_vault_key` MUST compare the presented backup's **captured** generation against the vault's current `kek_generation` (sourced from the audited rotation chain per N12-RT-06). The binding MUST NOT trust a caller-supplied generation integer in isolation, where "in isolation" means: the binding never re-establishes a key on a caller-claimed generation without first passing the foreign-shard check (N12-FT-02 **step 6**) and the commitment authentication (N12-FT-02 **step 7**). It MUST confirm the captured generation via the commitment binding (N12-CB-02), which recomputes over the backup's claimed/recorded generation and checks it against the registered commitment.
>
> **Two distinct defenses, in step order, on the default (non-forced) path.** A relabelled blob is refused by one of two gates depending on whether its ciphertexts are in the consulted distribution:
>
> - **Step 6 (`unknown-shard`), for a GENUINE-OLD-generation set.** On the default path the step-6 foreign-shard check (N12-CB-03) sources `shard_commitments` from the **CURRENT** generation's distribution. A KEK rotation re-shards, so a genuine old-generation blob's ciphertexts (true `g`) are NOT in the current `g+1` distribution and are refused at step 6 with **`unknown-shard`** BEFORE the step-7 commitment check runs, regardless of the relabel direction (claimed `g+1` or `g-1`). The first code for a genuine-old relabelled set on the default path is therefore `unknown-shard`, NOT `kek-commitment-mismatch` (fix [19], correcting the prior over-claim). This is the honest precedence: the foreign-shard gate consults the current distribution and the genuine-old ciphertexts are absent from it.
> - **Step 7 (`kek-commitment-mismatch`), when the ciphertexts ARE in the consulted distribution.** The commitment check is the relabel defense only for a set whose ciphertexts reach step 7, i.e. a genuine **current-generation** blob with only its generation integer tampered (its ciphertexts ARE the current distribution, so step 6 passes), or a separately **registered/injected** set recorded in the current distribution. In that case the verifier recomputes the commitment over the blob's claimed (tampered) generation and finds **no registered commitment** over `(secret, claimed_generation)`, surfacing **`kek-commitment-mismatch`** rather than a silent success (N12-CB-02, F-CRYPTO-3). A generation-integer tamper that survives the foreign-shard gate is caught here.
>
> The order is literal and matches the N12-FT-02 numeric order: step 6 (foreign-shard) precedes step 7 (commitment authentication), which precedes step 8 (ordinal staleness). For a set that passes step 6, the commitment binding **authenticates the captured generation** at step 7; only an authenticated-but-older generation then surfaces, at the **later** step 8, as **ordinal** `stale-generation`. The commitment step recomputes over the captured generation (N12-CB-02(b)); it does NOT recompute over the vault's current generation, which is the separate job of this ordinal staleness comparison. If the authenticated captured generation is **older** than the current generation, the restore MUST be refused with `stale-generation` by default. No KEK is re-established from a stale or relabelled backup on the default path. This protects against silently rolling the vault back to a superseded KEK under which current data keys are no longer wrapped. Exercised by V3(a) (ordinal `stale-generation`), V3(c)/(f) (genuine-old relabel refused at step 6 with `unknown-shard`), and V3(g) (generation-integer tamper whose ciphertexts are in the consulted distribution, refused at step 7 with `kek-commitment-mismatch`).

> **N12-SG-03 (force-override is a higher privilege, loud, dual-emitted, and audited)**: An implementation MAY provide an explicit `force_stale` override flag on `restore_vault_key` that permits a deliberate, operator-authorized stale-generation rollback. The forced path is **strictly more dangerous** than an ordinary restore (it re-installs a superseded KEK), so:
>
> - it MUST require the **distinct higher capability `vault:restore-stale`** (or `vault:override`) in addition to `vault:restore`; a caller holding only `vault:restore` with `force_stale=True` MUST be rejected with `missing-clearance` (F-AUTHZ-9);
> - when the governance-approver knob (N12-CL-03) is configured, the HELD approval is **mandatory** for the forced path (F-AUDIT-4);
> - `force_stale` overrides **only** the ordinal staleness step (N12-FT-02 **step 8**); it does NOT override the commitment authentication (step 7), the foreign-shard check (step 6), or any other gate. For the foreign-shard check to pass on a genuine old-generation set, the binding MUST source `shard_commitments` (N12-CB-03 / step 6) from the **CAPTURED** generation's distribution (the anchor that established `restored_generation`), NOT the current generation's, so the genuine old set reaches reconstruction instead of failing at `unknown-shard` (fix [2], third cycle);
> - when it proceeds, the binding MUST emit a **loud, distinct** anchor `subtype = vault_key_restore_forced_stale` recording, under the named N12-AU-04 forced-stale fields, `restored_generation` (the captured/installed old generation, equal to `kek_generation` and `generation_checked` on this anchor), `overridden_current_generation` (the current generation it overrode), the CAPTURED-generation `holders`/`shard_count`/`shard_commitments` actually used, the principal, and the **trust-anchored timestamp** (N12-AU-04a); this anchor MUST be **dual-emitted to the `safety` tier** (a **MUST**, mirroring the in-protocol dual-emit of EATP-10 N8-A-02 / N8-PO-04, F-AUDIT-4) so the forced rollback surfaces on the incident-response path; the materialized fixture is Appendix B §12.11;
> - at **Complete** level the `G_k` covering the forced-stale anchor MUST be witness-bound (N9-G-03) **before** the KEK is installed;
> - recording a forced-stale event under the ordinary `vault_key_restore` subtype is a **non-conformance of severity HIGH**;
> - `force_stale` MUST default to `False`; an implementation whose default permits stale restores is non-conforming.

> **N12-SG-05 (compromised-generation denylist)**: The stale guard is purely ordinal and does not catch re-installing a **current-generation but known-compromised** KEK before a rotation has advanced the counter (F-CRYPTO-6). The binding MUST therefore support marking a generation as **REVOKED/compromised** independent of ordinal staleness. A restore MUST refuse any backup whose captured generation is on the denylist with the distinct error `revoked-generation` and a loud anchor (with **trust-anchored timestamp**, N12-AU-04a) dual-emitted to the `recovery` and `safety` tiers, **even when the generation equals current**. Operationally, a compromise report MUST advance/rotate the current generation (N12-RT-06) so the compromised one becomes stale **and** denylisted. The denylist state MUST be derivable from the audited rotation/denylist anchor chain, not a locally-mutable store. Exercised by V3(e).

The stale-generation, denylist, threshold, clearance, and commitment gates are independent: a stale or denylisted backup with `k` valid shards and a fully-cleared caller still fails (on `stale-generation` / `revoked-generation`) unless `force_stale` is set with the `vault:restore-stale` capability and (where configured) the HELD approval. A restore proceeds only when all gates pass.

## 7. Conformance vectors

A conformant implementation MUST pass V1-V7 at the **Conformant** level. V8 is REQUIRED at the **Complete** level only. All vectors are deterministic given fixture inputs (Appendix B materializes the golden fixture V6 and V7 reference). Cross-SDK vectors (V6) are framed implementation-agnostically: any two conformant implementations MUST agree, and no single SDK is special-cased.

### V1, default-3of5-backup-restore-roundtrip

**Inputs**: A `back_up_vault_key(key_handle, ShamirRitual(threshold=3, total_shards=5), clearance, holders)` call where `clearance` carries `vault:backup` in the vault's tenant/domain and `holders` are five registry IDs, against a vault whose current `kek_generation` is `g`. Then a `restore_vault_key(shards, target_handle, clearance')` call where `clearance'` carries `vault:restore` and exactly 3 of the 5 shards are presented, against the same vault at generation `g`.

**Expected**: The backup resolves the handle internally (no raw KEK bytes cross the API, N12-IN-01; KEK-class verified, N12-IN-02), produces 5 SLIP-0039 shards via `shamir.generate` under the pinned parameters (N12-CRY-PIN), registers the KEK-identity commitment with `kek_commitment_alg="eatp-v1"` (the registry token whose Hash column resolves to SHA-256, fix [11]), the KCV, and `shard_commitments` (N12-CB-01/03/04), and, only after a successful dispatcher-mediated `recovery`-tier dispatch (N12-AU-02b), returns a `BackupReceipt`. The `EXTERNAL_SIDE_EFFECT` anchor with `subtype=vault_key_backup` carries `alg_id` (first), `{"k":3,"n":5}`, 5 registry holder IDs, shard_count 5, `vault_id`, `kek_generation=g`, the commitment + `kek_commitment_alg` + KCV, the pinned `slip39_params` incl. `master_secret_bits`, `side_channel_hardened`, principal, and trust-anchored timestamp, and **never** shard contents or KEK bytes (N12-AU-01). The restore passes clearance (N12-CL-02), threshold (exactly 3 = k), commitment (N12-CB-02), and generation (`g == g`) gates; reconstructs the KEK via `shamir.reconstruct`; re-establishes it inside the trusted module returning an opaque handle (no plaintext KEK returned, N12-IN-05); triggers D6 (N12-RT-05); and emits a `subtype=vault_key_restore` anchor. **Restore succeeds from EXACTLY k=3 shards; the reconstructed KEK MUST byte-equal the original KEK.** Presenting 4 or 5 shards is governed by N12-FT-02 (either `too-many-shards` or deterministic canonical trim to k), it MUST NOT be assumed to "just succeed" (corrected from the first draft per F-XSDK-2). See V5(g).

### V2, clearance-denied (with denial anchor and scope check)

**Inputs**: (a) `back_up_vault_key(...)` by a caller whose bound `CapabilitySet` lacks `vault:backup`. (b) `restore_vault_key(...)` with 3 valid shards by a caller whose `CapabilitySet` lacks `vault:restore`. (c) `restore_vault_key(...)` with 3 valid shards by a caller holding `vault:restore` **in domain A / tenant A** against a `target_handle` whose vault is in **domain B / tenant B**.

**Expected**: (a) Rejected with `missing-clearance` **before** the key handle is resolved (no KEK materialized). (b) Rejected with `missing-clearance` **before** any shard is combined, even though 3 valid shards were presented, demonstrating the clearance and quorum gates are independent (N12-CL-02). (c) Rejected with `missing-clearance` on tenant/domain scope (N12-CL-02a), fail-closed (tenant first via the cascade's bound tenant, then domain via `RoleScope.domain`). In all three cases a **denial anchor** (`subtype=vault_key_backup_denied` / `vault_key_restore_denied`) recording the principal, the missing capability/scope, and an opaque unresolved target-handle reference (NOT a resolved `vault_id`/commitment field set, N12-AU-01) is dispatched to the `safety` tier (the security-event class for access-control denials, N12-AU-01), resolving the first draft's N12-AU-01-vs-V2 contradiction (F-AUDIT-3/9). The capability check evaluates the bound role's `RoleScope.capabilities`, NOT `DelegateConstraintEnvelope`: a caller whose constraint envelope is fully permissive but whose `CapabilitySet` lacks the token MUST still be rejected (F-AUTHZ-1).

### V2-denial-flood, bounded growth, no lost principal attribution

**Inputs**: A caller who reaches the restore/backup entry point **without** the `vault:*` token (denial happens before key resolution, needing almost no privilege) sprays a high volume of denied attempts, each producing a denial anchor routed (per N12-AU-01) to the non-compactable `safety` tier.

**Expected**: Denial anchors MUST NOT be dropped (the prior draft's rate-limit/dedup knob that silently dropped records is struck; F-AUDIT-10). Under flood, the binding MUST emit, **at dispatch time**, a signed **`vault_denial_summary`** record (N12-AU-04, fix [6]) in lieu of N per-attempt anchors, covering the window and preserving the per-tier `safety` chain link AND complete distinct-principal attribution: distinct principals/missing-capabilities are enumerated in-line (sorted ascending) up to a fixed cap `M`, and complete attribution beyond `M` is preserved by `principal_set_root` (a sorted-hash/Merkle digest over the full distinct-principal set), so the summary **record size is bounded `O(M)` regardless of distinct-principal cardinality** and a principal-rotating attacker cannot inflate it (fix [8]). This is a **dispatch-time aggregation, NOT the EATP-09 §4.1 post-hoc compaction** the `safety` tier forbids (fix [7]): it bounds emission rate at the point of dispatch and does not rewrite already-stored `safety` anchors, so it does not violate `safety`'s "Not compactable" classification. Denial anchors and the summary (a security-relevant access-control class, not a recovery-ritual event) MUST be routed to the **`safety`** tier (EATP-09 §4.1's class for constraint violations / BLOCKED verdicts). The summary record MUST be verifiable and MUST account for every distinct probing principal in the window (in-line or provable against `principal_set_root`). Outcome anchors (success/typed-failure) remain in `recovery` unchanged. (Per F-AUDIT-10; resolves F-BND-8; second-cycle fixes [6]/[7]/[8].)

### V3, stale / relabelled / denylisted generation, and the rotation anchor

**Inputs**: A backup captured at `kek_generation = g`. The vault's KEK is rotated, emitting a `vault_kek_rotation` anchor and advancing the current generation to `g+1`. Then: (a) `restore_vault_key(stale_shards, target, clearance)` with `vault:restore`, 3 valid shards, `force_stale=False`. (b) the same call with `force_stale=True` and only `vault:restore`, then with `vault:restore-stale` present. (c) a genuine-old-generation backup whose true captured generation is `g`, whose blob is **relabelled** to `g+1`, restored at current `g+1` on the default path (`force_stale=False`). (d) inspection of the audit chain for the generation advance. (e) a backup at the **current** generation `g+1` whose generation has been placed on the compromised denylist. (f) the relabel-to-**OLDER**-generation precedence case: a genuine backup whose true captured generation is `g`, whose blob is relabelled to `g-1`, with current generation `g+1`, restored on the default path (`force_stale=False`). (g) the generation-integer-tamper case where the presented ciphertexts ARE in the consulted distribution: a genuine **current-generation** (`g+1`) backup whose ONLY tamper is its generation integer (relabelled to `g`), restored at current `g+1` on the default path (`force_stale=False`).

**Expected**: (a) Refused with `stale-generation` (N12-SG-02), clearance and threshold notwithstanding; no KEK re-established. (b) Refused with `missing-clearance`, `force_stale` requires the higher `vault:restore-stale` capability (N12-SG-03, F-AUTHZ-9). With `vault:restore-stale` present (and, where configured, the HELD approval), the restore proceeds: the genuine old-generation shard set **reaches reconstruction** (the N12-FT-02 step-6 foreign-shard check sources `shard_commitments` from the CAPTURED `g` distribution, NOT the current `g+1` distribution, so it does NOT fail at `unknown-shard`, fix [2]), the commitment authentication (step 7) PASSES over the captured `g`, and `force_stale` overrides only the ordinal staleness step (step 8). The binding emits a distinct `subtype=vault_key_restore_forced_stale` anchor to the `recovery` tier **dual-emitted to `safety`**, with `restored_generation=g`, `overridden_current_generation=g+1`, `kek_generation=g`, `generation_checked=g`, the CAPTURED-generation `holders`/`shard_count`/`shard_commitments`, principal, and trust-anchored timestamp (N12-AU-04 forced-stale row; materialized in Appendix B §12.11); recording it under the ordinary `vault_key_restore` subtype is a HIGH non-conformance (F-AUDIT-4). (c) Refused with **`unknown-shard`** at **step 6** (the foreign-shard check), because on the default path the step-6 check sources `shard_commitments` from the CURRENT `g+1` distribution, and the genuine-old-generation blob's ciphertexts (true `g`) are NOT in that `g+1` distribution (a KEK rotation re-shards, so the old ciphertexts never appear in the new generation's commitments). Step 6 therefore rejects before the step-7 commitment check ever runs; the first code is `unknown-shard`, NOT `kek-commitment-mismatch` (fix [19]: a genuine old-generation set is refused at step 6 on the default path because its ciphertexts are not in the consulted current distribution). The loud forced-stale anchor is **not** emitted and `force_stale` is **not** silently satisfied. (d) The `vault_kek_rotation` anchor is present, dispatcher-mediated, chained within the `recovery` tier, recording `g -> g+1` and carrying the new generation's re-shard distribution (`holders`/`shard_count`/`shard_commitments`/new commitment, fix [3]); a generation advance with no such anchor is non-conforming (N12-RT-06, F-AUDIT-6/F-BND-5). (e) Refused with `revoked-generation` even though the generation equals current (N12-SG-05, F-CRYPTO-6). (f) Refused with **`unknown-shard`** at **step 6**, for the same precedence reason as (c): the genuine backup's true-`g` ciphertexts are not in the consulted current `g+1` distribution, so the foreign-shard check fires before the step-7 commitment authentication and before the step-8 ordinal comparison ever runs. The relabel direction (to `g-1` versus `g+1`) is immaterial on the default path because step 6 consults the current distribution regardless of the claimed label, and the genuine-old ciphertexts are absent from it either way; the single canonical first code is `unknown-shard` (fix [19], correcting the prior over-claim that this case returned `kek-commitment-mismatch`). (g) Refused with **`kek-commitment-mismatch`** at **step 7**: here the presented ciphertexts ARE the genuine current-`g+1` ciphertexts and so ARE in the consulted current `g+1` distribution, so step 6 PASSES; step 7 then recomputes the commitment over the blob's claimed (tampered) generation `g` and finds no registered commitment over `(secret, g)` at this vault, surfacing `kek-commitment-mismatch` (N12-CB-02, F-CRYPTO-3). This is the case the commitment check is the defense for: the ciphertexts reach the consulted distribution, so the generation-integer tamper is caught at commitment authentication rather than at the foreign-shard gate.

### V4, holder-rotation, for-cause revocation, and old-shard rejection

**Inputs**: A (k=3, n=5) backup at generation `g`. (a)/(b): one holder departs **amicably**; the vault calls `rotate_holders(old_shards, ShamirRitual(3,5), ...)` with `vault:rotate`, producing a new shard set under a fresh SLIP-0039 identifier; then restores at `g` using (a) 3 shards from the **new** set, (b) 3 shards from the **old** set. (c): a holder is revoked **for cause**.

**Expected**: For the amicable cases, a `subtype=vault_holder_rotation` anchor records old/new `{"k","n"}`, departing holder, new distribution, `vault_id`, `kek_generation=g` (unchanged for amicable rotation, N12-RT-03), the new `shard_commitments`, and `for_cause=false`. (a) New-set restore succeeds and reconstructs the KEK. (b) Old-set restore MUST fail, old shards are not in the current `shard_commitments`, surfaced as **`unknown-shard`** (foreign/old-generation; **not** `corrupted-shard`, F-XSDK-4/F-BND-4), or `mixed-shard-set` if mixed with new shards, even though the old shards remain SLIP-0039-valid. (c) For-cause revocation MUST trigger a generation-advancing KEK-rotation (N12-SH-04): a **single** `subtype=vault_kek_rotation` anchor is emitted carrying **`for_cause=true`**, `prior_kek_generation=g`, `kek_generation=g+1`, AND the new (post-revocation) re-shard distribution (`{k,n}`, `holders`, `shard_count`, `shard_commitments`, the new `kek_identity_commitment`), so the for-cause flag, the generation advance, and the new distribution all land on one anchor with no two-anchor split and no flag stranded on a generation-unchanged anchor (fix [4], third cycle). `kek_generation` advances to `g+1`, the new generation's `shard_commitments` are immediately available as N12-CB-03's `g+1` distribution-of-record, and the departed holder's retained current-generation shards become **stale** (refused by N12-SG-02), closing the Conformant-level retained-shard window the first draft left open (F-AUTHZ-5, F-CRYPTO-8).

### V5, failure-taxonomy (distinct codes, correct stages)

**Inputs**: Restore attempts against a (k=3, n=5) backup at the current generation, with `vault:restore`: (a) only 2 valid shards; (b) 3 shards, one with a corrupted SLIP-0039 MAC; (c) a **homogeneous self-generated foreign** 3-of-5 set (single consistent identifier, no mixing); (d) 3 shards including one from a revoked (wrapped) holder; (e) a fully-valid 3-shard restore by a caller lacking `vault:restore`; (f) 3 shards mixing one foreign-ritual shard with two valid; (g) k+1 = 4 valid shards; (h) k+1 = 4 **foreign** shards (over-supplied foreign set, exercising the N12-FT-02 gate ordering).

**Expected**: Each MUST fail with the specific typed error and MUST NOT collapse to a generic error (N12-FT-01): (a) `insufficient-shards` (the EATP-10 code, not `insufficient-threshold`); (b) `corrupted-shard` (integrity only); (c) **`unknown-shard`**, the homogeneous foreign set is caught by the `shard_commitments` check **before** reconstruction (N12-CB-03), closing the gap the first draft's "mixed-identifier" heuristic missed (F-CRYPTO-2); (d) `revoked-holder`; (e) `missing-clearance`; (f) `mixed-shard-set` (step 5 fires before the step-6 commitment check), distinct from `corrupted-shard`; (g) per the deployment's N12-FT-02 choice, either `too-many-shards` or a deterministic-trim success, pinned, not SDK-dependent (F-XSDK-2); (h) under the **reject** branch `too-many-shards` (step 3 before step 6); under the **trim** branch, trim to k then `unknown-shard` (step 6), pinned by the N12-FT-02 canonical gate order so two SDKs return the same first code (F-XSDK-13). The codes MUST be distinguishable by the caller from the typed error alone, and foreign-shard rejection MUST surface at the commitment-check stage, not as a SLIP-0039 decode error.

### V6, cross-sdk reproducibility (parameters, KEK-commitment, audit canonical form)

**Inputs**: A fixed `(secret, ritual, passphrase, extendable, iteration_exponent, master_secret_bits)` tuple, the Appendix B fixed KEK byte string, `ShamirRitual(3,5)`, fixed passphrase, `extendable=true`, `iteration_exponent=1`, `master_secret_bits=128` (matching the materialized Appendix B fixture; 256-bit is the other legal value per §4.1's `{128,256}` set, fix [3]/[4]), supplied independently to two **distinct conformant implementations** of the binding (implementation-agnostic; the reference realization is `kailash-py trust/vault/shamir.py`, no implementation special-cased). Each serializes shards to the paper-print form (`serialize_shard`). Sub-vectors: (b) implementation A emits `extendable=true`, implementation B emits `extendable=false` (or divergent `iteration_exponent`/`master_secret_bits`); (c) a restore of a backup recording a `kek_commitment_alg` for which **no commitment is registered** at the captured generation (never registered, never recommitted); (d) a legitimate backup recorded under algorithm X, verified after the deployment's current suite advanced to algorithm Y via a `vault_kek_recommit`.

**Expected**: (a) For the **same** pinned tuple, any 3 of the 5 serialized shards produced by A MUST reconstruct, in B, to a secret byte-equal to the original, and vice versa, reproducibility holds **only because the encryption parameters are pinned** (N12-CRY-PIN). The **KEK-identity commitment** (N12-CB-01) MUST be byte-identical across A and B for the same `(domain_sep, vault_id, kek_generation, secret, passphrase_provenance)` under the recorded `kek_commitment_alg`, this is the cross-SDK anti-injection anchor; per-shard ciphertext hashes are NOT required to match across SDKs (F-XSDK-9, F-CRYPTO-9). The KCV (N12-CB-04(d)) MUST also be byte-identical across A and B for the same `(vault_id, generation, secret)`. The **canonical audit-envelope pre-image** (the `content_signing_bytes` pre-image of the `vault_key_backup` payload per N12-AU-03/AU-04) MUST be byte-identical across A and B, verified against the **materialized golden fixture (Appendix B, hex)**. Note: SLIP-0039 emits a fresh random identifier per `generate()`, so the **mnemonic words** of A and B differ; the reproducibility guarantee is at the `(secret, ritual)` -> reconstructable-secret level, the KEK-commitment level, the KCV level, and the audit canonical-form level, not at the raw-mnemonic-string level. (b) An implementation emitting a **different** `extendable`/`iteration_exponent`/`master_secret_bits` MUST fail reproducibility, surfaced as a mapped **`parameter-mismatch`** (now a defined N12-FT-01 code; the SLIP-0039 "identifier parameters don't match" condition), proving the vector exercises the divergence rather than masking it (F-XSDK-1, F-XSDK-10). (c) MUST fail with **`commitment-alg-mismatch`** (N12-CB-04(b); fires because no registered commitment exists for the recorded algorithm at the captured generation), not a silent success and not the injection code `kek-commitment-mismatch` (F-CRYPTO-11). (d) The legitimate cross-hash restore MUST **succeed** (commitment recomputed under the backup's recorded `kek_commitment_alg`, the D1 analogue, and compared to the registered commitment under that same recorded algorithm), while a relabel/injection still fails, a decades-old corpus survives a hash sunset without disabling the check (F-XSDK-10). **A backup recorded under algorithm X MUST still verify after a `vault_kek_recommit` to algorithm Y, because the additive registry (N12-CB-04(c)) keeps both `C_X` and `C_Y` live** (fix [2]). An implementation MUST NOT special-case any single SDK to pass this vector.

### V6(e), recommit additive-registry (old-algorithm backup survives a sunset)

**Inputs**: A vault at generation `g` with a registered commitment `C_X` under `kek_commitment_alg = eatp-v1`. Perform a `vault_kek_recommit` advancing the commitment algorithm to `eatp-v1.1` (adding `C_Y`). Then (i) restore an **old `eatp-v1`-recorded** backup, and (ii) restore a backup recording an algorithm `eatp-v2.slh-dsa` for which NO commitment was ever registered or recommitted at `g`.

**Expected**: (i) MUST **succeed**: the verifier recomputes under the backup's recorded `eatp-v1` and compares to `C_X`, which remains live in the additive per-(`target_handle`, `g`) registry after the recommit (fix [2]). (ii) MUST fail with **`commitment-alg-mismatch`** (no registered commitment exists for the recorded algorithm at `g`), not `kek-commitment-mismatch` and not a silent success.

### V7, audit envelope (excludes secrets, mediated, fail-closed, recovery-acceptance, trusted-time)

**Inputs**: (a) A successful backup and restore, each emitting an anchor; a fixture inspects the full serialized payloads. (b) An adversarial probe that inspects every return value, log sink, and receipt field for the plaintext KEK. (c) A fault-injection run where `AuditDispatcher.dispatch()` fails after reconstruction but before the key is installed. (d) A restore against an **indefinitely-sealed** `recovery` tier. (e) A host-clock-skew run that backdates/forward-dates the binding host's wall clock during a forced-stale restore. (f) A denial flood AND a forced-stale restore against a `safety` tier whose `dispatch()` is restricted under EATP-09 §3.4 (sealed-at-rotation).

**Expected**: (a) Both anchors are present in the `recovery` tier (N12-AU-02), carry `alg_id` as the pinned-first `event_payload.alg_id` field covered by the `content_signing_bytes` pre-image (N12-AU-02/AU-03/AU-04; the literal JCS-first-field form of D3 does not apply to this substrate-framed surface, F-AUDIT-8), and carry the required `event_payload` fields per N12-AU-04. The fixture MUST confirm that **no** shard mnemonic, **no** KEK byte, **no** data-key byte, **no** passphrase, and **no** per-holder wrapping secret appears anywhere; any appearance is a non-conformance of severity HIGH. Both anchors are enrolled via `dispatch()` (a non-`dispatch()` direct write is detected as non-chained and not covered by any `G_k`, N9-D-01/F-AUDIT-1) and are committed by the next global anchor (N9-G-01). (b) The plaintext KEK MUST appear in **no** return value, log line, or receipt field (N12-IN-05, F-BND-9). (c) The restore MUST **abort**: the reconstructed KEK MUST NOT be installed and MUST be `del`-ed; no active handle is returned (N12-AU-02b fail-closed, F-AUDIT-2). The outage MUST be surfaced as an operational incident (retryable), not as a permanent vault brick (N12-AU-02b bounded degradation, F-LIVE-2). (d) The restore MUST **succeed**, the indefinite seal blocks `rotate()`/compaction, not the security-critical `dispatch()`; the EATP-09 §3.4 "further restricts" escape MUST NOT be exercised against `recovery` for vault anchors (N12-AU-02a, F-LIVE-1). (e) A host-clock fast-forward/backdate MUST NOT alter the recorded forced-stale anchor `timestamp`, which is trust-anchored per N12-AU-04a (F-TEMP-3); where trust-anchored time is unavailable the field reads `unverified` (with `time_attested:false`). (f) Under a restricted `safety` tier (fix [9]), denial anchors and the `vault_denial_summary` MUST NOT be dropped, and the forced-stale dual-emit MUST either land or yield a **retryable operational incident** surfaced to the operator (never a silent gap, never a permanent brick), per the N12-AU-02a `safety`-tier extension; the EATP-09 §3.4 "further restricts" escape MUST NOT be exercised against `safety` for these vault anchors.

### V8, complete-level (witness, revocation, type-check, approver, cooling-off, ceremony)

**Inputs**: (Complete only) A Complete-level deployment with (i) external witness binding on the `recovery` tier's global anchors, (ii) per-holder wrapping (N12-SH-02), (iii) the governance-approver HELD action (N12-CL-03), (iv) KEK-class metadata on handles, and (v) the backup-ceremony witness knob (N12-CL-05). The fixture: (a) backs up under per-holder wrapping, revokes a departed holder, attempts a restore presenting the revoked holder's wrapped shard; (b) attempts a backup of a **data-key** handle, and a restore of vault A's backup against vault B's `target_handle`; (c) attempts a backup naming an **unregistered** holder; (d) attempts a restore without the governance-approver HELD approval, and a restore where approver == requester; (e) a principal within the D6 cooling-off window attempts a second unsupervised restore, including a local-clock fast-forward attempt; (f) a backup lacking the in-pre-image ceremony witness token, and a backup where witness == requester.

**Expected**: (Complete only) Each `recovery`-tier `G_k` covering the vault anchors MUST be deposited with an external witness and be independently retrievable (N9-G-03; cross-reference `eatp/09-tiered-audit-dispatcher.md@v1.0 V4 (global-anchor-survives-dispatcher-compromise)`). (a) The revoked-holder restore MUST fail with `revoked-holder` (N12-SH-02), and revocation MUST NOT have dropped the un-revoked set below `k` (N12-SH-03). (b) The data-key backup MUST fail with `not-a-kek` (N12-IN-02); the cross-vault restore MUST fail with `key-identity-mismatch` (N12-CB-02). (c) The unregistered-holder backup MUST fail with `unregistered-holder` (N12-SH-01, F-AUTHZ-6). (d) The restore lacking the HELD approval MUST fail with `missing-clearance`, the approval token being absent from the signed `event_payload`; a self-approval (approver == requester) MUST also fail with `missing-clearance` (N12-CL-03, F-AUTHZ-7). (e) The in-cooling-off restore MUST require the HELD approver or be rejected with `missing-clearance` (N12-CL-04, F-AUTHZ-4); a local-clock fast-forward MUST NOT lift the suspension (trust-anchored window, F-TEMP-2). (f) The backup lacking the in-pre-image witness token MUST be rejected at Complete, and a self-witness (witness == requester) MUST fail (N12-CL-05, F-AUTHZ-12). A Conformant-level (non-Complete) implementation is **not** required to pass V8.

## 8. Security considerations

- **Commitment binding is the anti-injection keystone (N12-CB-01/02).** Without it, `restore_vault_key` is a root-key re-install primitive: any internally-consistent k-of-n set installs attacker-known key material into an authorized vault, passing clearance, threshold, and (with a relabel) generation gates. The KEK-identity commitment over the **canonical input tuple** `{domain_sep, vault_id, captured kek_generation, master_secret, passphrase_provenance}`, where `kek_generation` is the **captured** generation (the value the backup tagged at backup time, recomputed identically at restore, N12-CB-01/02(b)), is the single control that turns "reconstruct whatever combines" into "reconstruct only the key bound to this vault and captured generation." It is computed via an unambiguous length-delimited canonical encoding (no concat-collision class) over a one-way hash, never the KEK bytes, so it does not violate the audit contents-exclusion rule. The commitment authentication is the **earlier** gate (N12-FT-02 step 7) and recomputes over the **captured** generation; the separate, **later** ordinal staleness gate (N12-SG-02 / N12-FT-02 step 8) compares that authenticated captured generation against the current generation, which is why a legitimate forced-stale rollback (N12-SG-03) passes the commitment check (step 7) even though it is ordinally stale (step 8, the only step `force_stale` overrides).
- **Commitment crypto-agility (N12-CB-04).** The commitment hash is bound to the EATP-08 §3.3 registry via a recorded `kek_commitment_alg` (the registry token `eatp-v1`, whose Hash column resolves to SHA-256), and restore recomputes under the backup's recorded algorithm (a D1 analogue), never the verifier's current suite, comparing against the registered commitment under that same recorded algorithm drawn from the per-(`target_handle`, generation) **additive** registry. A `vault_kek_recommit` ADDS the new-algorithm commitment without deleting the old, so an old-algorithm escrow backup still verifies after a sunset. This closes the self-inflicted denial-of-recovery a hash sunset would otherwise cause for the highest-value secret, while keeping injection caught by `kek-commitment-mismatch` and an unregistered algorithm distinguished as `commitment-alg-mismatch` (which fires only when no registered commitment exists for the recorded algorithm at that generation). A `vault_kek_recommit` anchor migrates a legitimate long-lived corpus across a hash sunset.
- **Standalone integrity / KCV (N12-CB-04(d)).** The N12-CB-01 commitment verifies only against the live target vault; a blob in escrow for years is otherwise integrity-naked. The mandatory key-free KCV (NIST SP 800-130) lets an operator confirm offline that a blob reconstructs the expected key without a live-vault round-trip and without an extra key, surfacing tamper as `kcv-mismatch`. The OPTIONAL Complete-level keyed AEAD adds standalone metadata-tamper detection, with the explicit caveat that the integrity key's own custody is a recovery-scenario dependency, which is exactly why the key-free KCV is the floor.
- **Raw-bytes escape hatch (N12-IN-03)** is the single most dangerous surface in this spec, it reintroduces raw KEK bytes in caller memory and bypasses the handle's generation/identity binding. It is therefore **disabled by default** (a MUST), gated behind a build flag AND the mandatory governance-approver HELD action, still subject to the commitment check, and dual-emitted to the `safety` tier. Its documentation MUST mark it as such.
- **Clearance is two orthogonal axes.** Capability tokens live on `CapabilitySet` (`RoleScope.capabilities`); the Conformant gate is membership in the **bound role's** `CapabilitySet`, mirroring the shipped `DispatchSurface` Invariant 3 (`dispatch.py:1441`). `CapabilitySet.intersect` (`types.py:535`) is the privilege-non-escalation _composition_ primitive for multi-role binding (a REQUIRED forward-path surface, #1035), not a Conformant-level resolution step the single-role gate performs (F-AUTHZ-10). The `DelegateConstraintEnvelope` carries only constraint dimensions + `genesis_id` and is governed by the TIGHTENING-ONLY invariant (the rs F5 type-state, #1035), it is **not** the capability axis. The effective-posture function (D5) returns a _posture_ on the autonomy total order, **not** a `CapabilitySet`; a principal forced to SUPERVISED post-recovery still holds `vault:restore` in the substrate, which is exactly why N12-CL-04 suspends the token at the binding layer during cooling-off. Conflating these axes is the F-AUTHZ-1/4 failure mode.
- **Tenant/domain scoping (N12-CL-02a) is binding-owned.** A `vault:*` token is meaningful only within the vault's `(tenant, domain)`, but the shipped capability gate is domain-blind and `RoleScope` has no tenant field (F-AUTHZ-11). The binding therefore independently verifies the dispatch cascade's bound tenant against the vault's tenant and the bound role's `RoleScope.domain` against the vault's domain, fail-closed (tenant first, then domain, then token); it does NOT rely on a substrate cascade that does not exist. A flat token would let a low-value dev-vault grant recover a production KEK across tenants.
- **Generation integrity (N12-RT-06, N12-SG-01/02).** The stale guard is only as strong as the generation counter. The counter is a monotonic integer (a REQUIRED net-new key-manager surface, §3.4) advanced **only** by an audited, dispatcher-mediated `vault_kek_rotation` anchor, and the current value consulted at restore is sourced from that signed chain, not a mutable local store, mirroring EATP-08 D2c and EATP-10 §14. The generation is bound into the commitment so a relabel yields a mismatch, not a silent rollback.
- **Forced-stale and compromised-generation (N12-SG-03/05).** The forced-stale path is a deliberate security downgrade; it requires a distinct higher capability (`vault:restore-stale`), a mandatory HELD approval where configured, a loud anchor with trust-anchored timestamp dual-emitted to `safety`, and (at Complete) witness binding before install. The denylist closes the current-but-compromised re-install the ordinal guard misses; a compromise report MUST advance the generation so the compromised one becomes both stale and denylisted. A forced stale restore restores a KEK current data keys may not be wrapped under; the operational consequence (data keys unreadable, or re-wrap required) MUST be understood before the override is exercised.
- **Non-constant-time reconstruction (N12-CRY-SC).** The reference SLIP-0039 implementation is documented by its authors as not constant-time (`trust/vault/shamir.py:36-43`); a co-resident timing/cache adversary during `combine_mnemonics` is the residual threat the memory-locked, network-isolated process only partially mitigates. The spec does not strengthen the upstream crypto (the §3.1 boundary holds), but it does NOT hide the hazard: the `side_channel_hardened` flag surfaces the property (default `false`), and Complete level requires hardware-backed reconstruction so the master secret never enters the non-constant-time userspace routine (NIST SP 800-57, PKCS#11 benchmark).
- **Audit is a precondition, not a side effect (N12-AU-02/02a/02b).** Anchors are dispatcher-mediated (N9-D-01; a direct storage write is invisible to cross-tier tamper detection) and fail-closed: the KEK does not go active until the anchor is durably dispatched. The `recovery` tier MUST accept `dispatch()` despite its indefinite seal (N12-AU-02a) so the fail-closed ordering cannot be turned into a permanent brick; an audit-tier outage is a retryable operational incident, not vault destruction. There is no "key recovered, no anchor" outcome. Forced-stale, raw, and denylist events dual-emit to `safety` so the incident-response path fires.
- **Trust-anchored anchor time (N12-AU-04a).** The hash chain protects ordering, but the absolute timestamp value on the incident-response anchors (forced-stale, denylist) is trust-anchored per EATP-10 §14 so a backdated/forward-dated rollback cannot misdirect incident response about when a KEK rollback occurred.
- **Best-effort secret residence, not zeroization.** The shipped wrapper deletes the intermediate master secret on a best-effort basis (`del` in `finally`); the secret may persist in memory, swap, or GC-reachable buffers until overwritten. This spec inherits the limitation, requires a `del`-in-`finally` consumption discipline at the binding boundary (N12-IN-05), and requires (MUST, upgraded from SHOULD) the KEK-handling region to run in a memory-locked, swap-disabled, network-isolated process; deployments SHOULD prefer hardware-backed custody where the KEK never leaves the module in plaintext.
- **No plaintext crosses the boundary (N12-IN-05).** The reconstructed KEK is consumed inside the trusted module and `del`-ed before return; an opaque handle is returned, never bytes. V7(b) is the adversarial probe that an implementation does not leak the bytes into a return value, log, or receipt while nominally returning a handle.
- **Holder revocation window (N12-SH-02/03/04).** Under physical custody (Conformant), a departed holder who retained a paper card holds a cryptographically valid current-generation shard; no commitment or identifier check can distinguish it. The honest defense is generation-supersession: for-cause revocation MUST advance the generation (N12-SH-04). Per-holder wrapping (Complete) additionally closes the window for the wrapped form but not for any unwrapped copy the holder made. Neither level recovers a shard a holder physically exfiltrated; the threshold `k` (k-1 compromised holders tolerated) is the structural defense, exactly as in EATP-10 §14.
- **Backup-ceremony single-actor exposure (N12-CL-05).** The maximum-exposure moment is programmatic generation, where a single `vault:backup` credential materializes and splits the full secret alone. At Complete, an independent in-pre-image `vault:witness` principal (`!= requester`, `!= approver`) is required, defense-in-depth parity with EATP-10 §6.1 human witnessing and root-key-ceremony practice. The registry (N12-SH-01) already bounds WHO holds shards; this closes the remaining single-actor ceremony gap on the generation surface.
- **Passphrase is not authorization (N12-PP-01).** The SLIP-0039 passphrase selects among MAC-valid secrets; it is not an integrity check and MUST NOT be relied on as authorization. It is bound into the commitment so a wrong passphrase yields a mismatch, and its provenance is defined (RECOMMENDED vault-derived) so it is not an attacker-supplied free parameter.
- **Threshold floor (N12-TH-01).** 1-of-n and 1-of-1 KEK rituals are forbidden at all levels. The wrapper's deferred `allow_trivial` unlock is **not** granted by this spec; any future relaxation requires a separate governance-gated amendment.
- **Long-lived backup artifacts: harvest-now-decrypt-later and forward secrecy.** EATP-12 backups are paper SLIP-0039 shards intended to persist for the vault's full lifetime (EATP-10 §5.4 mandates archival-quality media), and the reconstructed secret is the KEK that wraps all data keys. SLIP-0039 shares are information-theoretically secure (the _secret_ is PQC-safe), but the audit-anchor signatures and any per-holder wrapping are the harvest-now-decrypt-later exposure; when SLIP-0039 publishes a PQC-secure construction, deployments MUST re-shard high-value KEKs under it within a deployment-stated migration window (cross-reference EATP-10 N8-P-03 archival media and §13 SLIP-0039 PQC-inheritance path, and the EATP-08 reserved PQC suites `eatp-v2.ml-dsa`/`eatp-v2.slh-dsa`), WITHOUT redefining the SLIP-0039 crypto (§1.1 defers it). On forward secrecy: because N12-RT-03 leaves the master secret unchanged across amicable holder rotations, a single historical shard-set compromise is permanent until a KEK rotation (§5.2), so deployments SHOULD perform periodic KEK rotation (N12-RT-06) on a deployment-defined cadence to bound the validity window of any historical compromise. The re-shard-on-PQC event MUST be carried by the audited `vault_kek_rotation` anchor so the migration is itself accountable. (Per F-TEMP-1.)
- **Cross-SDK asymmetry.** The Rust SDK (`kailash-rs`, proprietary) does not expose an equivalent vault/shamir surface in the Foundation-accessible repository at finalization time. Cross-SDK parity (V6) is verified at the SLIP-0039 serialized paper-print form (with pinned parameters), the KEK-identity commitment, the KCV, and the audit canonical form, all published, implementation-agnostic surfaces, not by inspecting any proprietary implementation. The Foundation cites only the Apache 2.0, Foundation-owned `kailash-py` wrapper as the reference realization; `kailash-rs` is not cited as a Foundation reference.
- **Event-type vocabulary is closed; subtype namespace reserved.** The vault events ride `EXTERNAL_SIDE_EFFECT + event_payload["subtype"]` because `DelegateEventType` is a founder-ratified, rs-canonical, closed 5-variant enum gated by the C3 audit-visibility classifier; minting new top-level event strings would raise `AuditChainEmissionError` and break cross-SDK parity. The `vault_` subtype prefix is reserved disjoint from the substrate's `{dispatch_invocation, cascade_emission, lifecycle_transition, posture_ratchet, sovereign_handover}` namespace, and the binding (not the engine, which validates only `event_type`) MUST validate the subtype before emit (F-XSDK-12). The governance-approver and ceremony-witness tokens are placed inside `event_payload` precisely because `content_signing_bytes` signs the payload (the engine-assigned chain-link fields are excluded by construction, so payload-resident tokens are the only signed location available); for the same reason `alg_id` rides `event_payload.alg_id` (F-AUDIT-8).
- **Cross-spec citation versioning.** Per `01-analysis/03-cross-spec-dependencies.md` §9, this spec cites EATP-08 at v1.0 (D1, D2a/b/c, D3, and the §3.3 commitment-suite registry + its sunset/deprecation machinery, on which the N12-CB-04(e) algorithm-retirement path depends, so an EATP-08 §3.3 algorithm sunset MUST trigger re-validation and corresponding `vault_kek_retire` retirements per N12-CB-04(e)(5)), EATP-09 at v1.0 (D4, N9-D-01/02/03, N9-T-02, N9-G-01/03/04), EATP-10 at v1.0 (ritual + commitment surface + codes + D6 + §14 clock), and EATP-11 at v1.0 (D5/N3-EP-01). Amendments to any cited decision MUST emit a follow-up amendment-issue listing this spec for re-validation. The EATP-09 §3.4/§4.1 recovery-tier dispatch-acceptance clarification and the EATP-10 V1 over-supply erratum (§10) are flagged as upstream errata this spec's correctness depends on.

### 11.3 Numbering reconciliation (N37 -> N12, completed at v1.0)

The workshop draft used the `N37-*` requirement prefix tied to the stable ISS-37 identifier the code cited. On promotion to **Published v1.0** the prefix reconciled to the EATP-12 spec number per the suite convention: a mechanical rename `N37- -> N12-` across all requirements (`N37-IN-* -> N12-IN-*`, `N37-CL-* -> N12-CL-*`, `N37-SH-* -> N12-SH-*`, `N37-CB-* -> N12-CB-*`, `N37-PP-* -> N12-PP-*`, `N37-AU-* -> N12-AU-*`, `N37-FT-* -> N12-FT-*`, `N37-RT-* -> N12-RT-*`, `N37-SG-* -> N12-SG-*`, `N37-CRY-PIN -> N12-CRY-PIN`, `N37-CRY-SC -> N12-CRY-SC`, `N37-TH-* -> N12-TH-*`), with no requirement-text change. The `Tracking` line resolved from `mint ISS-37` to `terrene-foundation/mint#15`, and `Status` advanced from `Draft` to `Published`. Area codes used: **IN** (Input), **CL** (CLearance), **SH** (SHard/holder), **CB** (Commitment Binding), **PP** (PassPhrase), **AU** (AUdit), **FT** (Failure Taxonomy), **RT** (RoTation), **SG** (Stale Generation), **CRY-PIN**/**CRY-SC** (CRYpto parameter pinning / Side-Channel), **TH** (THreshold floor). Code that still cites `N37-*` is reading a pre-promotion reference and should migrate to `N12-*`.

## 12. Appendix B, Golden fixture (normative)

This appendix materializes the cross-SDK byte-identical diff target that N12-AU-04 and V6 reference. Two conformant implementations MUST reproduce each canonical form and each hash below **exactly** for the fixed inputs. The fixture uses JCS (RFC 8785): UTF-8, sorted keys, no insignificant whitespace, integers as integers. All hex is lowercase. The KEK-identity commitment and KCV are computed under `kek_commitment_alg = "eatp-v1"` (the EATP-08 §3.3 registry token whose Hash column resolves to SHA-256; fix [11]), so the recorded algorithm string is `"eatp-v1"` while the hash primitive is SHA-256.

**Top-level wrapper canonicalization (fix [17]).** `content_signing_bytes(event_type, event_payload, signer_delegate_id)` JCS-canonicalizes the **entire** top-level wrapper object, re-sorting its keys to `event_payload`, `event_type`, `signer_delegate_id`, not only the inner `event_payload`. The positional argument order of `content_signing_bytes` is NOT the byte order; the authoritative byte order is the JCS-sorted one shown in each prose object literal below and in each hex pre-image.
