## Special section — agents.py:2287 (the slice/str mismatch)

```python
# agents.py:2272-2287
try:
    # Execute single round
    round_result = self.execute(**round_input)
    ...
    # Update state for next round if memory enabled
    if memory and state_key in round_result:
        current_state = round_result[state_key]   # ← line 2287
```

`Agent.execute` (line 375) signature:

```python
def execute(self, workflow=None, **kwargs):
    """...Returns: Dictionary with structured outputs based on signature, or
       tuple (results, run_id) for workflows..."""
```

The docstring documents the dual return shape but no `-> Union[...]` annotation. Pyright infers the union; `state_key in round_result` narrows to "object supports `__contains__`" (both `dict` and `tuple` do via iteration); `round_result[state_key]` then complains because `tuple[Any, str | None].__getitem__` only accepts `int | slice`, not `str`.

The code is structurally fine for the signature-mode branch (where `round_result` IS a dict) but type-unsafe in the workflow-mode branch — though `execute_multi_round` is documented as signature-mode-only and asserts `self.signature is None` raises at line 2237. The fix is one of:

1. **Add return annotation** `def execute(...) -> Union[Dict[str, Any], Tuple[Dict[str, Any], str]]` AND narrow at the call site: `assert isinstance(round_result, dict)` before line 2286.
2. **Split execute** into `execute_with_signature` and `execute_with_workflow` — cleaner long-term but multi-shard work.
3. **Cast at call site**: `round_result = cast(Dict[str, Any], self.execute(**round_input))`.

Option 1 is the minimal fix and aligns with framework.py's existing TYPE_CHECKING pattern.

## Caller-impact summary (what breaks if we delete)

- **`Kaizen.mcp_registry` property** (deletion candidate): 2 references in one script (`scripts/comprehensive_implementation_tester.py`) — both already produce `None` due to silent ImportError. Deletion is safe; the script is a tester, not production. Per Rule 6a, this is a public attribute removal — would need a deprecation cycle UNLESS we treat it as "removing a method that has never worked", which is structurally never-worked (always returns None). Recommend: delete + delete the script reference + add CHANGELOG entry stating "deprecated mcp_registry property removed (always returned None due to absent registry module)".
- **`Kaizen.discover_mcp_tools` external branch** (partial deletion): 13+ test references but ALL test the local-tools path (which works); deleting the unreachable external-discovery branch is invisible to all callers.
- **`Agent.expose_as_mcp_server` / `expose_as_mcp_tool` / `connect_to_mcp_servers` / `_discover_servers` / `call_mcp_tool`** (full method deletion candidates): 20+ test references + 9 doc/ADR references. Deleting requires: (a) deleting / porting these tests, (b) deleting / annotating the docs, (c) updating the public API to point at `kaizen-agents` BaseAgent equivalents (`discover_mcp_tools` exists there). This is **the larger-than-5-line scope** that pushes Cluster C beyond a typing-only fix.

The minimum-viable Cluster-C shard for issue #822 closure (pyright clean) is:

1. Add `TYPE_CHECKING` block import for `Signature` — 1 line.
2. Add return-type annotation on `Agent.execute` + isinstance narrow — 3 lines.
3. Either (a) `# type: ignore[import-not-found]` on the 6 dead-import lines, OR (b) delete the 6 dead branches + 5 dead methods.

Path (b) is correct per `zero-tolerance.md` Rules 2 + 4 (no fake integration, no workarounds). Path (a) is what pyright wants but would be a Rule-2 violation in disguise. **Recommendation: ship (b) but in a separate shard that includes test sweep + CHANGELOG migration, since it's a public-API removal.** For pyright closure ALONE in this shard, ship (a) with explicit `# type: ignore` comments naming the orphan tracking issue (to be filed separately under #814 cleanup banner).
