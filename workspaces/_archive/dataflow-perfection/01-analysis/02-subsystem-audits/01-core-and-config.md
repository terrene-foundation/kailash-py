# Core + Config Subsystem Audit

**Scope**: `packages/kailash-dataflow/src/dataflow/core/` (34 files, 24,447 lines) + `configuration/` (662 lines) + `database/` (1,626 lines) + `features/` (6,305 lines, depended on by core) + top-level `dataflow/__init__.py` + `dataflow/engine.py` (the `DataFlowEngine` wrapper).

## Summary

Core and config are far worse than the issue reports suggested. I found **7 CRITICAL**, **18 HIGH**, **22 MEDIUM**, and **12 LOW** findings in scope — including two live SQL-injection primitives, a multi-tenant Express cache leak with zero tenant dimension in cache keys, a fake `TransactionManager` that ships as the "transactional safety" mechanism for model registry, a fake `MultiTenantManager` that stores tenants in memory, fake tenant encryption (`f"encrypted_{key}_{data}"`), the confirmed `self._redis_url` dead-state from #354, the confirmed #352 sync-in-async pattern in 13 sites across `model_registry.py`, the entire `ProgressiveConfiguration` family exposed as public API but never consumed, and a version mismatch (pyproject 1.8.0 vs `__version__` 1.7.1). The `DataFlowEngine` wrapper adds almost no value beyond method delegation. Startup validation is advisory-only in direct violation of `rules/dataflow-pool.md` Rule 2. Per-operation Express cache has zero tenant awareness: any `multi_tenant=True` deployment ships a cross-tenant data leak primitive **today**, not just when a Redis fix lands.

## CRITICAL findings

### C1 — Express cache key has no tenant dimension (live data leak)

- **File**: `packages/kailash-dataflow/src/dataflow/cache/key_generator.py:97-135`, consumed at `packages/kailash-dataflow/src/dataflow/features/express.py:964, 988`
- **Symptom**: `generate_express_key(model_name, operation, params)` produces keys like `dataflow:v1:User:list:a1b2c3d4` with no `tenant_id` segment. `express.py` never consults any tenant context (`grep tenant` on `features/express.py` → zero matches). When `multi_tenant=True`, `db.express.list("User", {"active": True})` issued by tenant A is cached and SERVED to tenant B's identical call.
- **Classification**: Live cross-tenant data leak. Worse than fabric #354 because (a) express is in the hot path for every single-record CRUD, (b) the multi-tenant flag is already shipping without a tenant partition, and (c) the `CacheBackend.auto_detect` path already reaches Redis when `REDIS_URL` is set, so the leak is cross-process today as soon as a user follows the docstring.
- **Fix outline**: `generate_express_key` MUST take a required `tenant_id` argument when the parent DataFlow has `multi_tenant=True`, and `ExpressDataFlow.__init__` MUST capture a reference to the DataFlow's tenant-context resolver (`get_current_tenant_id` from `core/tenant_context.py`). Keys become `dataflow:v1:<tenant_id>:User:list:a1b2c3d4`. Cache writes MUST refuse when `multi_tenant=True` and no tenant is bound — loud error, not silent fallback to a tenantless key.
- **Test strategy**: Tier 2 integration test with real PostgreSQL, real Redis, two tenants, same `db.express.list` call, assert tenant B gets a fresh query (not tenant A's cached row set). Tier 3 stress: parallel tenant switches under load, assert zero cross-tenant hits over 10k operations.

### C2 — `features/transactions.py` `TransactionManager` is a fake (no BEGIN/COMMIT/ROLLBACK)

- **File**: `packages/kailash-dataflow/src/dataflow/features/transactions.py:30-58`
- **Symptom**: `transaction()` stores a Python dict with `status = "committed"` and yields it. There is no connection acquisition, no `BEGIN`, no `COMMIT`, no `ROLLBACK`, no use of the configured `isolation_level`. On exception, it just flips the dict to `"rolled_back"` — the database is unchanged.
- **Consumer**: `core/model_registry.py:101` constructs a `TransactionManager` as `self._transaction_manager` and wraps every registration in `_transaction_context` at line 107. The model registry claims "Transaction safety" but has **zero transactional guarantees**.
- **Classification**: Dishonest stub + deceptive configuration + framework-first violation (`rules/zero-tolerance.md` Rule 2, `rules/dataflow-pool.md` Rule 3, `rules/framework-first.md` — DataFlow's own connection manager/savepoint primitives are not used).
- **Fix outline**: Replace the body with real driver-level transactions via `ConnectionManager` or `TransactionScopeNode`. `isolation_level` must be a real `SET TRANSACTION ISOLATION LEVEL` statement per dialect. Nested transactions use savepoints. `rollback_all` actually rolls back.
- **Test strategy**: Tier 2 integration test with real PostgreSQL: (a) `with db.transaction() as tx: db.express.create(...); raise ValueError` — assert the row is NOT persisted; (b) nested transactions use savepoints; (c) concurrent transactions observe the configured isolation level via `pg_stat_activity`.

### C3 — `core/multi_tenancy.py` RowLevelSecurity injects `tenant_id` via f-string (SQL injection)

- **File**: `packages/kailash-dataflow/src/dataflow/core/multi_tenancy.py:415, 420, 424, 427, 429, 452, 456, 477, 482, 490, 494, 367, 375`
- **Symptom**: `RowLevelSecurityStrategy.apply_isolation` builds the `WHERE tenant_id = '{tenant_id}' AND ...` clause via f-string. `create_tenant_policy` and `set_tenant_context` also f-string interpolate `tenant_id` into `CREATE POLICY`, `DROP POLICY`, `SET row_security.tenant_id = '{tenant_id}'`. `SchemaIsolationStrategy._create_schema_sql` at lines 367, 375 does the same for `CREATE TABLE {schema}.{table}` and `DROP SCHEMA IF EXISTS {schema} CASCADE`.
- **Impact**: A tenant_id of `' OR 1=1; DROP TABLE users;--` achieves full query rewrite and RCE through the security layer that was supposed to enforce isolation. Multi-tenant is the attack surface, not the defense.
- **Classification**: Security vulnerability, `rules/security.md` violation (parameterized queries), `rules/infrastructure-sql.md` violation.
- **Fix outline**: Strict regex-validate `tenant_id` at the top of `set_tenant_context` (alphanumeric + hyphens + underscores, max length 64). Use parameterized `SET row_security.tenant_id = $1`. For RLS policy DDL, use SQL identifier quoting (`psycopg2.sql.Identifier`) not string interpolation. Schema names get the same treatment. Add a regression test: attempting to register a tenant_id containing a quote raises at validation time.
- **Test strategy**: Tier 2 with real PostgreSQL: try tenant_ids containing `'`, `;`, `--`, `DROP`; assert every one is rejected at `register_tenant` time, and assert no injected SQL reaches the database (via pg query log inspection).

### C4 — `features/multi_tenant.py` `MultiTenantManager` is a fake (in-memory dict, doesn't touch the DB)

- **File**: `packages/kailash-dataflow/src/dataflow/features/multi_tenant.py:10-94`

- **Fix outline**: Delete the entire `features/multi_tenant.py` file. Route `MultiTenantManager` references in `core/engine.py` to the real `core/multi_tenancy.py` `TenantManager` (once C3 is fixed). Tenant persistence goes through a real DataFlow-managed `dataflow_tenants` table with a proper migration.
- **Test strategy**: Tier 2 with real PostgreSQL: create a tenant, restart the DataFlow instance, assert the tenant still exists (proves persistence). Delete a tenant, assert all its tenant-scoped rows are gone (proves cascade).

### C5 — `core/multi_tenancy.py:925-949` Tenant encryption is fake string concatenation

- **File**: `packages/kailash-dataflow/src/dataflow/core/multi_tenancy.py:925-949`
- **Symptom**: `encrypt_tenant_data(tenant_id, data)` returns `f"encrypted_{encryption_key}_{data}"` — literal string concatenation, not encryption. `_get_tenant_encryption_key` returns `"tenant_specific_key"` — a hardcoded constant. Line 930 has a bare `except:` swallowing tenant-not-found. The encryption "helper" is shipped as a security feature.
- **Classification**: Dishonest stub + bare-except (`rules/zero-tolerance.md` Rule 2 + Rule 3) + security lie. Any downstream user who writes `encrypt_tenant_data(tenant_id, credit_card_number)` thinks the data is protected and it is not.
- **Fix outline**: Delete `encrypt_tenant_data` and `_get_tenant_encryption_key` entirely; tenant-level encryption belongs in a real key-management subsystem (KMS/Vault), not in DataFlow core. If it stays, it MUST use a real symmetric cipher via `cryptography.fernet` with keys sourced from an environment-variable-referenced secrets backend. Never keys as hardcoded strings.
- **Test strategy**: Tier 2 integration test: encrypt a known plaintext, decrypt, assert round-trip. Additionally, assert the ciphertext is NOT a substring of the plaintext (catches concatenation fakes).

### C6 — `#352` sync `runtime.execute()` in async context (13 sites in `model_registry.py`)

- **File**: `packages/kailash-dataflow/src/dataflow/core/model_registry.py:246, 629, 705, 778, 822, 904, 1163, 1215, 1258, 1297, 1344, 1403, 1456`; reached via `engine.py:787 _sync_models_from_registry → model_registry.initialize() → _create_model_registry_table()`.
- **Symptom**: When `DataFlow(...)` is constructed inside a running event loop (FastAPI `lifespan`, pytest-async fixture, Jupyter), `engine.py:441-444` assigns `self.runtime = AsyncLocalRuntime()`. `ModelRegistry(self, runtime=self.runtime)` at `engine.py:538` accepts that runtime. All 13 `self.runtime.execute(workflow.build())` calls then hit `kailash/runtime/async_local.py:599-603` which explicitly **raises** `RuntimeError("AsyncLocalRuntime.execute() called from async context. Use execute_workflow_async instead.")`. The "fix" of detecting async context only swapped the deadlock for a raised exception. The behaviour `ModelRegistry` promises (registering the table, registering pending models) fails outright from the FastAPI startup path.
- **Context**: Engine.py:6049-6063 has the correct pattern — detect async loop, raise clear error, provide async variant at 6171. Model registry does not have an async variant. Every single sync call site is an infinite accumulation of the same bug.
- **Classification**: Confirmed #352 with deeper scope. Rules: `rules/zero-tolerance.md` Rule 1, `rules/patterns.md` (Docker/Nexus/async contexts require AsyncLocalRuntime + `execute_workflow_async`).
- **Fix outline**: Every `_xxx()` method in `model_registry.py` that calls `self.runtime.execute(...)` gets an `async def _xxx_async()` sibling that uses `await runtime.execute_workflow_async(...)`. The sync variants raise `RuntimeError` when invoked from an async context with a clear message. `initialize()` becomes `async def initialize_async()` with a sync shim via `async_safe_run()` that is only safe from sync callers.
- **Test strategy**: Tier 2 — stand up real PostgreSQL, construct `DataFlow` inside `pytest.mark.asyncio` test, register a `@db.model`, assert no exception, assert the `dataflow_model_registry` table exists. Tier 2 — run the same under a FastAPI `TestClient` with a lifespan.

Guard: until C1 and C7 are fixed, add a runtime warning that flags the combination as "production unsafe" — loud, in ERROR, at `DataFlow.__init__`.
