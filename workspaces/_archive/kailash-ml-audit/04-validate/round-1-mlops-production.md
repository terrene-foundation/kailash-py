# Round 1 — MLOps / Production Engineer Audit

**Persona:** MLOps / production engineer (kubeflow / mlflow / sagemaker / vertex / ray background).
**Scope:** kailash-ml 13 engines + serving + monitoring + registry — production readiness only.
**Date:** 2026-04-21
**Non-scope:** spec fidelity, DL internals, RL internals, newbie UX, cosmetic industry parity.

## TL;DR

kailash-ml is approximately **40% production-ready** from an MLOps-engineer perspective. The inner primitives (PSI/KS drift, ONNX export, LRU cache, scheduled monitoring, MLflow-compatible ExperimentTracker, point-in-time feature joins) are individually competent. **But the integration surface is catastrophically fragmented**, and every capability an MLOps engineer expects to find wired end-to-end is not:

1. **Two parallel registries** (`MLEngine._kml_engine_versions` tenant-aware scaffold vs `ModelRegistry._kml_model_versions` single-tenant production code) — the tenant-aware one is Phase-2-`NotImplementedError`.
2. **Two parallel trackers, two parallel databases** (`km.track()` → `~/.kailash_ml/ml.db` vs `ExperimentTracker` engine → `sqlite:///kailash-ml.db`) — already flagged in the audit brief, confirmed mechanically here.
3. **Zero tenant-isolation wiring in 13 of 13 production engines** (the `engines/*.py` set). `tenant_id` exists only in the new `_engine_sql.py` Phase-2 scaffold and the `tracking/` module; every engine an actual ML user touches today is single-tenant.
4. **Zero RBAC / authentication / authorization surface** on the entire package — no `actor_id` plumbed through `register_model`, `promote_model`, `predict`, `train`, `set_reference_data`. A DataFlow/Nexus user with zero PACT integration cannot answer "who promoted v7 to production?" for any model.
5. **Zero Prometheus / metrics endpoint / observability surface** for the `InferenceServer` hot path (only `time.perf_counter()` inside the request — nothing exported). No `/metrics` endpoint, no histogram, no counter.
6. **Zero shadow / canary / A-B test surface** on `InferenceServer`. The registry declares a `shadow` stage; the server has no code path that routes traffic between stages. `shadow` is a pure metadata label with no traffic-splitting consumer.
7. **Zero engine-to-tracker auto-wiring for 10 of 13 engines.** Only `TrainingPipeline`, `HyperparameterSearch`, `AutoMLEngine` accept a `tracker=` kwarg; the other 10 (FeatureStore, ModelRegistry, DriftMonitor, InferenceServer, ensemble, preprocessing, feature_engineer, model_explainer, data_explorer, model_visualizer) have zero tracker surface.
8. **Zero WARN/mode/source structured log fields** in inference hot path per `rules/observability.md` §3 (`mode=real/fake`) + Mandatory Log Points §1–3.

The brief's "known-real gap" list said `km.track()`-to-`MLDashboard` composition is broken. The audit confirms that **AND** extends the gap list by three orders of magnitude — every engine the MLOps engineer expects to be unified (tenant, RBAC, tracker, metrics, shadow traffic) is fragmented.

## Zero-Tolerance / Rule Violations Detected

1. **`rules/zero-tolerance.md` Rule 2 (No Stubs):**
   - `engine.py:617, 2431` raise `NotImplementedError("Phase 3 implements ...")`. Exported `MLEngine` symbol ships with stubbed `register`, `predict`, `serve`. Same failure mode as the Phase 5.11 `TrustAwareQueryExecutor` orphan (`rules/orphan-detection.md` §1) — the tenant-aware schema exists without a hot-path consumer.
   - `ALL_STAGES = {"staging", "shadow", "production", "archived"}` with no shadow-traffic consumer IS a Rule-2 "fake feature" per the Phase 5 wiring sweep examples (fake encryption, fake transaction, fake tenant isolation). `shadow` is a metadata label whose only side-effect is state-machine transitions; no prediction call uses the shadow version.

2. **`rules/tenant-isolation.md` MUST Rule 1 (Cache Keys Include tenant_id):** violated by every engine's internal caches (InferenceServer `_ModelCache`, DriftMonitor `_references`). Keys are model-name only. A multi-tenant deployment produces cross-tenant cache leaks the moment two tenants name a model identically.

3. **`rules/tenant-isolation.md` MUST Rule 2 (strict typed error on missing tenant_id):** violated in all 13 production engines. Most don't take a `tenant_id` arg at all — they don't error loud, they silently operate on a shared keyspace.

4. **`rules/observability.md` Mandatory Log Points §1-4:** `InferenceServer.predict` has zero entry/exit/error logs. `ModelRegistry.promote_model` has zero logs around the promotion. `DriftMonitor.check_drift` has one warning for missing column, zero for drift-detected — the most consequential event in the system doesn't emit a WARN or INFO.

5. **`rules/observability.md` Rule 3 (Data calls log `mode=real/fake`):** not present in any engine. `grep 'mode=' engines/*.py → 0 matches`.

6. **`rules/observability.md` Rule 2 (Correlation ID on every log line):** the `InferenceServer.register_endpoints` handler chain has no `request_id` binding per the structured-logger pattern.

7. **`rules/orphan-detection.md` §1 (facade without production call site):** `MLEngine` (exported in `__init__.py`?) is a facade whose `register`/`predict`/`serve` raise `NotImplementedError`. The `_kml_engine_*` tables exist; no production call site populates them. This is the exact structural hazard the rule prevents.

8. **`rules/facade-manager-detection.md` Rule 1 (manager-shape class Tier 2 test through facade):** 13 engines each expose as `kailash_ml.X` classes — no corresponding `test_*_wiring.py` audit has been run (out of scope here but flagged).
