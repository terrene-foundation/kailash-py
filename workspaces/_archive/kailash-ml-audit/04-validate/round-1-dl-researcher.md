# Round 1 — Deep Learning Researcher Audit

**Persona:** DL researcher (PyTorch / Lightning / HuggingFace / RLHF). Cares about gradient flow, activation stats, dead neurons, LR finder, loss curves, training dashboard, distributed training, mixed precision, checkpoint / resume, and live monitoring.

**Mandate:** Grep-verified only. No hand-waving. Every finding has a file:line citation.

**Workspace root:** `/Users/esperie/repos/loom/kailash-py`

**Scope boundary (non-scope):** spec-to-code fidelity, newbie UX, RL lifecycle (RL is absent and handled by the RL-researcher persona), MLOps production, industry benchmarking. This report focuses on DL-researcher pain.

---

## Executive Summary

kailash-ml has a **rich DL diagnostics surface** (~1,938 LOC in `diagnostics/dl.py` with 10 `plot_*` methods, 4 `track_*` installers, LR range test, Grad-CAM, `report()`) but it is **structurally disconnected from every other DL-relevant surface in the SDK**. The diagnostics class has zero tracker wiring, zero event sink, zero knowledge of `km.track()`, zero invocation from `TrainingPipeline._train_lightning`. `MLDashboard` reads from a different SQLite file than `km.track()` writes to. No DDP / FSDP / autocast / GradScaler / checkpoint-resume code exists anywhere. `transformers.Trainer` is not a first-class trainable.

The DL researcher can:

- Manually instantiate `DLDiagnostics(model)` and call `record_batch` / `record_epoch` inside a hand-rolled loop — then render 10 plotly figures inline.
- Run `km.train(Trainable, data)` which goes through Lightning — but DLDiagnostics does NOT fire, and no metrics flow into `km.track()`.
- Use `TrainingPipeline` with `framework="lightning"` and pass an engine-layer `ExperimentTracker` — this DOES log metrics, but NOT to the `km.track()` SQLite store, NOT with gradient/activation/dead-neuron diagnostics, and NOT to the MLDashboard's backing DB.

Three different "trackers", three different DBs, zero DL diagnostic events flowing into any of them. The researcher cannot observe a live training run without hand-wiring everything.

## Gaps flagged as out-of-scope for patch but worthy of a deferred-item ticket

- **DL-GAP-1**: Distributed training (DDP / FSDP) — zero code. Multi-GPU large-model training is not a supported use case in kailash-ml today. Document explicitly in `specs/ml-engines.md` under "Deferred Capabilities".
- **DL-GAP-2**: `transformers.Trainer` as first-class Trainable — today routed through `kailash-align` for LLMs only. Non-LLM HuggingFace models (vision, audio, tabular) have no path.
- **DL-GAP-3**: TensorBoard compat / ONNX Runtime profiler hooks — the `compat/` directory has `MlflowFormatReader/Writer` but no TB event-file writer.
