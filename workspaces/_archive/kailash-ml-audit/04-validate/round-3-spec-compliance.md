## Section C — Cross-Spec Drift Register

The narrow-scope failure mode of `specs-authority.md §5b` produces exactly the categories below. Each item is re-derivable via the grep commands in Section B.

### C.1 Single-Tenant Sentinel — `"_single"` vs `"global"` (HIGH, B.14a)

- `ml-tracking §7.2` line 712 + 759 + 1086: `"_single"`
- `ml-engines-v2 §5.1` line 672-678: `"global"`
- `ml-feature-store §12.1` line 499 + §13 line 639: `"global"`
- `ml-registry §2.2` line 65 + §6.2 line 336: `"global"`
- `ml-serving §1.3` line 36: `"global"`

**Impact:** At implementation time, a cache key or audit row written by `ml-serving` as `kailash_ml:v1:global:...` is invisible to a dashboard query that resolves `"_single"` from the `ml-tracking`-authored constant. Cross-tenant misrouting within the single-tenant dev path.

**Fix:** Pick one. Decision 3 does not explicitly name the literal — but Decision 14 says "mandatory tenant*id + actor_id + audit on every engine". Pin the value in `ml-tracking §7.2` as the authoritative name (it already IS authoritative per `§9.1` role for cross-domain re-exports) and sweep every sibling spec to that value. Recommend `"_single"` (the `*`-prefix signals "internal sentinel" and grep-distinguishes it from any legitimate tenant named `"global"`).

### C.2 `TrainingResult` Field Shape — three-string vs `DeviceReport` (HIGH, B.20)

- `ml-engines-v2 §4.1` declares: `device_used: str`, `accelerator: str`, `precision: str`
- `ml-rl-core §3.2` assumes: `device: DeviceReport` (one typed field)
- `ml-rl-core §3.3` reads `result.device` as an attribute
- `ml-rl-algorithms §2` (line 43) takes `device: DeviceReport` in adapter `__init__`
- `ml-backends §1` introduces `DeviceReport` type but doesn't show the mapping into `TrainingResult`

**Impact:** A consumer of `TrainingResult` cannot simultaneously satisfy both parent spec and child spec. Either `result.device.backend` raises AttributeError (ml-engines-v2's string field) OR `result.device_used == "cuda:0"` raises AttributeError (ml-rl-core's typed field).

**Fix:** `ml-engines-v2 §4.1` MUST be updated to `device: DeviceReport` as the canonical field AND the three string fields either (a) derived from `device` for backward compatibility, or (b) removed. Decision 14 lists `TrainingResult` as part of the breaking-change surface — correct disposition is (b) remove, with `device_used` / `accelerator` / `precision` read through `device.<attr>` properties.

### C.3 Error-Hierarchy Drift (HIGH, B.8a-d)

- B.8a **`ParamValueError`** — raised at `ml-tracking §4.1` line 304 but absent from `§9.1` canonical list. Add to `TrackingError` family, same pattern as `MetricValueError`.
- B.8b **`MultiTenantOpError`** — Decision 12 pinned at 1.0.0 for cross-tenant admin export. Zero matches across 15 specs. Must be added to `ml-tracking §9.1` under `TrackingError` (or a new `AdminError` family, but given §9.1 MUST NOT clause, simpler as TrackingError sub-type).
- B.8c **`UnsupportedTrainerError(MLError)`** — declared at `ml-engines-v2 §3.2` line 469 inheriting directly from `MLError`. `ml-tracking §9.1 MUST NOT` "MUST NOT declare a new top-level family". Either: (1) move under `BackendError` family (semantically: a backend-level unsupported-trainer condition), OR (2) declare a new `EngineError(MLError)` family and route through it.
- B.8d **`RLTenantRequiredError`** — `ml-rl-core §8.1` line 470 + §14 line 1014. Duplicates `TenantRequiredError` which is canonical at `ml-tracking §9.1` line 853 under TrackingError. Per `§9.1`, RL-path MUST re-export, not re-declare. Replace every `RLTenantRequiredError` with `TenantRequiredError` re-exported from `kailash_ml.errors`.
