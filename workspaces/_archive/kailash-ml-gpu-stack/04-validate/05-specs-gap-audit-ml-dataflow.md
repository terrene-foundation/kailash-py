## HIGH Findings (Summary)

1. **ml-engines.md §6.1 MUST 2 — ONNX `_export_torch` branch missing.** Spec mandates `torch` framework key has implemented branch via `torch.onnx.export`. Code has only sklearn/xgboost/lightgbm branches.
2. **ml-engines.md §6.1 MUST 2 — ONNX `_export_lightning` branch missing.** Spec mandates `lightning` via `to_torchscript` path.
3. **ml-engines.md §6.1 MUST 2 — ONNX `_export_catboost` branch missing.** Spec mandates `catboost` via `catboost.save_model(...)`.
4. **ml-engines.md §6.1 MUST 3 — No `test_onnx_roundtrip_{framework}.py` regression tests** for any of the 6 matrix entries. Spec mandates Tier 2 round-trip regression per framework key.
5. **ml-backends.md §7 — `km.doctor()` diagnostic absent.** Zero implementation of `python -m kailash_ml.doctor` / `km-doctor` console_script. Entire section is documented but not backed. §7.3 requires `cpu` CI lane to run `km-doctor --json` after install.
6. **ml-tracking.md §2.1 — `km.track()` is `NotImplementedError`.** `kailash_ml/__init__.py:187` raises with "Phase 6 (ExperimentTracker) will implement per specs/ml-tracking.md." Direct `ExperimentTracker` construction exists as a lower-level primitive, but the spec-sanctioned `async with km.track(...)` entry point is a stub.
7. **ml-tracking.md §2.4 (auto-capture fields) — 16 mandatory fields cannot be validated.** Blocked by #6; there is no code path that captures `host`, `python_version`, `kailash_ml_version`, `lightning_version`, `torch_version`, `cuda_version`, `git_sha`, `git_dirty`, `device_family`, `device_backend`, `device_fallback_reason`, `device_array_api`, `tenant_id` on a `km.track()` run.
8. **ml-tracking.md §2.2 — `COMPLETED/FAILED/KILLED` status auto-set not wired to `km.track`.** Same root cause as #6.

(NB: `ExperimentTracker.__aenter__`/`__aexit__` at engines/experiment_tracker.py:450 IS implemented — the HIGH is specifically about the `km.track()` convenience entry + the auto-capture fields contract.)

## MEDIUM Findings

1. **dataflow-express.md `bulk_*` classification** — `_apply_classification_mask_record` is called on single-record mutation returns (create/update/upsert). Bulk variants (`bulk_create`, `bulk_update`, `bulk_upsert`) need per-row verification. Not verified in this audit pass.
2. **dataflow-cache.md §6.2 + tenant-isolation.md §3a — keyspace `v*` wildcard invalidation** — Adapters `AsyncRedisCacheAdapter.invalidate_model` and `MemoryCacheAdapter.invalidate_model` exist; whether they match `dataflow:v*:` (wildcard) versus `dataflow:v2:` (pinned) was not grep-verified at the adapter method bodies this round.
3. **ml-tracking.md — `ExperimentTracker.open()` classmethod** — spec shows explicit-construction entry via `ExperimentTracker.open(name=, tenant_id=, store=)`. Not grep-verified in this audit.

## Surprises

1. **km.doctor totally absent.** Given `ml-backends.md` §7 is 30+ lines of spec (exit codes, JSON output, CI integration), zero implementation is a bigger gap than any single ML-engines §3 Trainable orphan would have been. It is the dispositioned public entry-point for "is my install healthy" and without it §7.3 "cpu CI lane MUST run km-doctor" cannot be satisfied.
2. **ONNX bridge is the v0.9.x bug-shape.** The spec §6.1 MUST 2 explicitly cites v0.9.x "xgboost=guaranteed but code raised NotImplementedError" as the pattern BLOCKED. The current code has xgboost/lightgbm/sklearn as real branches but **catboost/torch/lightning** are the new version of the same failure — advertised in the matrix, zero implementation. The rule against this pattern is embedded in the spec itself.
3. **km.track() stub is explicitly Phase-6-deferred in code but the spec was moved to authoritative `specs/` (not draft) in the prior cycle.** The spec should either have `km.track — Phase 6, not yet shipped` as an explicit deviation per specs-authority.md MUST Rule 6, or the code should ship the implementation. Current state: code says "Phase 6", spec says "MUST — `async with km.track(...)`", creating a spec-code contradiction that specs-authority.md §6 BLOCKS.
