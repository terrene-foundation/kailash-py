# Serving Infrastructure Analysis

## 1. Ollama's Model Import API

### GGUF Import Workflow

Ollama imports custom models via a `Modelfile` + `ollama create` CLI command:

```
# Modelfile
FROM ./my-model.gguf

# Optional system prompt
SYSTEM "You are a helpful assistant specialized in..."

# Optional parameters
PARAMETER temperature 0.7
PARAMETER num_predict 4096
```

```bash
ollama create my-fine-tuned-model -f Modelfile
```

### What `align.deploy(model, target="ollama")` Actually Needs To Do

1. **Ensure adapter is merged**: LoRA adapter + base model -> merged full model
2. **Convert merged model to GGUF**: via `convert_hf_to_gguf.py` from llama.cpp
3. **Quantize** (optional but recommended): via `llama-quantize`
4. **Write Modelfile**: `FROM /path/to/quantized.gguf` + optional system prompt
5. **Run `ollama create`**: subprocess call
6. **Verify**: `ollama show model_name` or HTTP `GET /api/show`

### Ollama REST API Endpoints (Relevant to kailash-align)

| Endpoint      | Method | Purpose                                                |
| ------------- | ------ | ------------------------------------------------------ |
| `/api/chat`   | POST   | Chat completions (already used by OllamaStreamAdapter) |
| `/api/tags`   | GET    | List locally available models                          |
| `/api/show`   | POST   | Show model metadata                                    |
| `/api/create` | POST   | Create model from Modelfile (alternative to CLI)       |

**Note**: The `/api/create` REST endpoint could replace the `subprocess.run(["ollama", "create", ...])` approach, giving better error handling and avoiding the dependency on the `ollama` CLI binary being on PATH.

### GGUF Format Requirements

- Ollama accepts GGUF files directly (no conversion from Ollama's side)
- Supported architectures: Llama, Mistral, Phi, Qwen, Gemma, StableLM, and most popular transformer variants
- Quantization levels: Q4_K_M (recommended balance), Q8_0 (higher quality), F16 (full precision)
- GGUF must include tokenizer data (most `convert_hf_to_gguf.py` conversions include this automatically)

## 2. vLLM's Model Loading and Configuration

### vLLM Model Loading

vLLM loads models directly from HuggingFace format (safetensors). No GGUF conversion needed:

```bash
python -m vllm.entrypoints.openai.api_server \
    --model /path/to/merged-model \
    --served-model-name my-fine-tuned-model \
    --host 0.0.0.0 --port 8000 \
    --tensor-parallel-size 1 \
    --max-model-len 4096 \
    --dtype bfloat16
```

### What `align.deploy(model, target="vllm")` Needs To Do

1. **Ensure adapter is merged**: Same as Ollama path
2. **Generate vLLM config**: JSON config + launch script
3. **NOT start vLLM**: User responsibility (correct architectural decision)

This is simpler than Ollama deployment because there is no format conversion step. The merged HuggingFace model is loaded directly.

### vLLM Configuration Generated by AlignmentServing

```json
{
  "model": "/path/to/merged-model",
  "host": "0.0.0.0",
  "port": 8000,
  "tensor-parallel-size": 1,
  "max-model-len": 4096,
  "dtype": "bfloat16",
  "served-model-name": "my-fine-tuned-model"
}
```

Plus a `launch_vllm.sh` convenience script. This is a simple code generation task.

## 3. GGUF Conversion: What Tools Exist

### Primary Tool: `convert_hf_to_gguf.py`

Located in `llama.cpp` repository. Converts HuggingFace safetensors to GGUF format.

**Requirements**:

- Python 3.10+
- `gguf` Python package (`pip install gguf`)
- Full model weights (NOT LoRA adapters directly)
- Input: HuggingFace model directory with `config.json`, `tokenizer.json`, `*.safetensors`
- Output: Single `.gguf` file

**Usage**:

```bash
python convert_hf_to_gguf.py /path/to/merged-model --outtype f16 --outfile model-f16.gguf
```

### Secondary Tool: `convert_lora_to_gguf.py`

Converts LoRA adapters to GGUF format separately (for Ollama's native LoRA loading). However, this approach is less reliable than merge-then-convert and has compatibility issues with newer architectures.

### Quantization Tool: `llama-quantize`

```bash
llama-quantize model-f16.gguf model-q4_k_m.gguf q4_k_m
```

This is a compiled C++ binary from llama.cpp. Must be built or obtained separately.

### Alternative: Unsloth's GGUF Export

Unsloth provides a higher-level API for GGUF conversion that handles merging + conversion + quantization in one step. However, this introduces a dependency on unsloth (large package, GPU-specific builds). Not recommended as a dependency for kailash-align.

## 4. Is Automated GGUF Conversion Reliable in 2026?

### Reliability Assessment: MEDIUM

**What works reliably**:

- Converting popular model architectures (Llama 2/3, Mistral, Phi-3, Qwen2.5) that have been well-tested in llama.cpp
- F16 conversion (no quantization, just format change) -- most reliable
- Q4_K_M quantization of models that converted successfully to F16

**What frequently breaks**:

- Newer architectures that llama.cpp hasn't added support for yet
- Models with custom tokenizers or non-standard config.json fields
- Models with custom rope_scaling configurations
- LoRA adapter direct conversion (merge first is required and more reliable)
- Models that require `trust_remote_code=True` in HuggingFace

**Specific Failure Modes**:

1. **Silent malformed GGUF**: Config differences (renamed tokenizer keys, missing fields) produce GGUF files that appear valid but crash at inference. No error during conversion.
2. **Architecture not supported**: `convert_hf_to_gguf.py` has a hardcoded list of supported architectures. New models require llama.cpp updates.
3. **Tokenizer mismatch**: Some tokenizers don't roundtrip correctly through GGUF format, producing garbled output.
4. **Quantization quality**: Aggressive quantization (Q2_K, Q3_K) can significantly degrade model quality, especially for smaller models (3B-7B).

### Practical Recommendation for kailash-align

1. **Always produce F16 GGUF first**, then quantize as a separate step. Never skip the F16 intermediate.
2. **Default to Q4_K_M** for deployment (good balance). Offer Q8_0 as a higher-quality option.
3. **Validate the GGUF output**: After conversion, run a simple inference test before registering as deployed.
4. **Pin llama.cpp version**: The conversion scripts change frequently. Pin to a known-working version.
5. **Provide clear error messages**: When conversion fails, explain the likely cause and suggest manual steps.
6. **Support "bring your own GGUF"**: Allow users to provide a pre-converted GGUF file, bypassing the conversion pipeline entirely.

### External Binary Dependency Risk

`convert_hf_to_gguf.py` is a Python script (can be pip-installed via `gguf` package), but `llama-quantize` is a compiled C++ binary. AlignmentServing must either:

- **Option A**: Require the user to install llama.cpp and set `LLAMA_CPP_DIR` env var
- **Option B**: Use the `llama-cpp-python` package which bundles compiled binaries (adds ~200MB, limited platform support)
- **Option C**: Skip quantization and only produce F16 GGUF (Ollama can quantize on import in some versions)

**Recommendation**: Option A is correct for v1. Document the llama.cpp requirement clearly. Do not bundle compiled binaries.

## 5. Summary

| Deployment Target | Complexity                                 | Reliability                      | External Deps            |
| ----------------- | ------------------------------------------ | -------------------------------- | ------------------------ |
| **Ollama**        | High (merge + convert + quantize + create) | Medium (GGUF conversion fragile) | llama.cpp, ollama binary |
| **vLLM**          | Low (merge + config generation)            | High (no format conversion)      | vLLM binary              |
| **local_hf**      | N/A (no serving step)                      | N/A                              | None                     |

Ollama serving is the highest-risk component in kailash-align due to the GGUF conversion fragility. vLLM serving is straightforward because it loads HuggingFace models natively.
