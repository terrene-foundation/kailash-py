# Issue #835: dataflow: TransactionManager._get_adapter selects daemon-thread pool whose _loop differs from caller loop

Filed: 2026-05-05T16:31:27Z
URL: https://github.com/terrene-foundation/kailash-py/issues/835
Labels: ['bug', 'cross-sdk', 'sdk-engine', 'type/bug', 'area/core']

---

## Summary

`db.transactions.transaction()` is unusable in any caller whose event loop differs from the daemon-thread persistent loop that backs `DataFlowExpressSync`. The asyncpg pool's `_loop.call_later(...)` (called by `pool.release` to schedule the inactive-callback) raises `RuntimeError: Event loop is closed` against an unrelated loop, breaking transaction commit/rollback in:

- pytest-asyncio function-scoped tests
- short-lived `asyncio.run(...)` invocations from CLI scripts
- any consumer whose primary event loop differs from the loop where the adapter pool was first initialized

The async hot path (`db.express.list/create`) works correctly because it uses a different pool — see "Two pools" below.

## Environment

- `kailash` 2.11.3
- `kailash-dataflow` 2.3.1
- Python 3.13.7 / pytest-asyncio 1.3.0 / asyncpg 0.31.0
- PostgreSQL 15

## Two pools

Empirically confirmed via `asyncpg.create_pool` monkeypatched stack-trace logging — DataFlow's runtime hosts **two distinct asyncpg pools**:

1. **AsyncSQLNode enterprise pool** (used by `db.express.*`) — created at first use via `kailash.nodes.data.async_sql.AsyncSQLNode.connect`, bound to the calling event loop. Works correctly across loop transitions because `_PoolWrapper(node._pool)` is the fallback in `TransactionManager._get_adapter()` only when no adapter is found via the connection-manager path.

2. **PostgreSQLAdapter pool** (used by `db.transactions.*`) — created via `dataflow.utils.connection.initialize_pool` running inside the daemon-thread persistent loop that backs `DataFlowExpressSync`. The pool's `_loop` reference is the daemon thread's loop, not the caller's.

`TransactionManager._get_adapter()` resolves first to `dataflow._connection_manager._adapter` — the daemon-thread pool — and only falls back to the AsyncSQLNode pool if the connection-manager path is empty. This is the wrong default for callers whose primary loop differs from the daemon-thread loop.

## Repro

```python
import asyncio, os
os.environ['DATABASE_URL'] = 'postgresql://user:pass@localhost:5432/test'

from dataflow import DataFlow
db = DataFlow(database_url=os.environ['DATABASE_URL'])

# A trivial @db.model
@db.model
class Probe:
    id: int
    name: str

async def main():
    loop = asyncio.get_running_loop()
    print('test loop id:', id(loop))

    # Express path: works (uses AsyncSQLNode pool, caller-loop bound)
    rows = await db.express.list('Probe', {'name': 'x'})
    print('express ok')

    # Inspect what TransactionManager will resolve to
    from dataflow.features.transactions import TransactionManager
    tm = TransactionManager(db)
    ad = tm._get_adapter()
    print('adapter pool._loop is test loop:', ad.connection_pool._loop is loop)
    print('adapter pool._loop closed:', ad.connection_pool._loop.is_closed())

    # Transaction path: RuntimeError: Event loop is closed
    async with db.transactions.transaction():
        pass

asyncio.run(main())
```

Output:
```
test loop id: 4582857696
express ok
adapter pool._loop is test loop: False
adapter pool._loop closed: True
RuntimeError: Event loop is closed
```

The adapter pool's `_loop` is bound to a closed loop **even within a single `asyncio.run` invocation** — the daemon-thread loop is not the caller's loop and was never the caller's loop.

## Workarounds tried (all failed)

1. **Module-level `pytestmark = pytest.mark.asyncio(loop_scope='session')`** on the test file — daemon-thread loop is invisible to pytest-asyncio's loop scoping.
2. **Removing the deprecated user-defined `event_loop` session fixture** from conftest — same outcome; the daemon-thread loop binding is independent of pytest-asyncio's session loop.
3. **Adding `loop_scope='session'` to `@pytest.mark.asyncio` per test** — same outcome.

## Proposed fix (one of)

**A. Re-order `_get_adapter()`** — prefer the AsyncSQLNode pool (caller-loop bound) over the connection-manager adapter (daemon-thread bound). Both speak asyncpg; the SQL contract is identical.

**B. Bind the adapter pool lazily on first use** — in `dataflow/utils/connection.py::initialize_pool`, defer `await asyncpg.create_pool(...)` until inside an `async` method invoked from the caller's loop, rather than executing it inside the daemon-thread loop.

**C. Make the daemon-thread loop optional** — gate `DataFlowExpressSync` initialization behind explicit opt-in (e.g., `df = DataFlow(..., enable_sync_api=True)`); when off, skip the daemon thread entirely so the only pool that exists is the caller-loop one.

A is the smallest fix and likely sufficient for most consumers; B is the most defensive; C is the cleanest if the sync API is rarely used.
