# Brief — Issue #354: Fabric product cache ignores `redis_url`

**Source**: terrene-foundation/kailash-py#354
**Reporter**: @esperie (Jack Hong)
**Impact site**: impact-verse (production, multi-replica)
**Severity**: CRITICAL — docstring promises a feature the implementation omits; every multi-replica deployment is affected
**Classification**: Zero-Tolerance Rule 2 (stub/partial implementation) + Rule 3 (silent fallback) + framework-first promise broken

## One-line summary

`PipelineExecutor` accepts `redis_url` and stores it on `self._redis_url`, but the cache itself is a per-process `OrderedDict`. Redis is plumbed through three layers (`DataFlow.__init__` → `FabricRuntime` → `PipelineExecutor`) yet never instantiated as a client for the product cache. The docstring at `pipeline.py:137` claims "Supports both in-memory (dev) and Redis (production) cache". The code does not.

### Problem (verified at source — kailash-dataflow 1.8.0)

`DataFlow(redis_url=...)` accepts a Redis URL parameter but `PipelineExecutor` does NOT use it for product caching. The product cache is stored in an in-memory `collections.OrderedDict`, per-process and per-replica. Switching to `DataFlowEngine` does not help (verified — see below).

### Verification — Redis IS used elsewhere in fabric, just not for the cache

| File                                  | Redis usage                         | Purpose                                                                              |
| ------------------------------------- | ----------------------------------- | ------------------------------------------------------------------------------------ |
| `webhooks.py:73` `_RedisNonceBackend` | `redis.sismember`, `sadd`, `expire` | Webhook nonce deduplication (cross-worker)                                           |
| `pipeline.py:582` `_RedisDebouncer`   | unclear from name                   | (reporter believed this existed; in current source, only `InMemoryDebouncer` exists) |
| `runtime.py:69-178`                   | Stores and forwards `redis_url`     | Passes to PipelineExecutor (which only uses it for the debouncer)                    |
| `pipeline.py` cache                   | **NONE**                            | Cache is `OrderedDict`, not Redis                                                    |

### DataFlowEngine does NOT add Redis caching

```
$ grep -nE "cache|Cache|redis|Redis" dataflow/engine.py
# (zero matches)
```

`DataFlowEngine.builder().build()` calls `dataflow = DataFlow(...)` — the same primitive. The Engine adds validation, classification, and query monitoring layers ON TOP of the primitive, but the underlying fabric cache is still `PipelineExecutor._cache_data` (in-memory OrderedDict).

### Impact on ImpactVerse production

1. **Each replica has its own cache** — no sharing, redundant DB queries, wasted memory
2. **Cache lost on every restart** — cold-start cost = full rebuild from sources
3. **`prewarm=True` blocks startup** — 26 products against 6,481 participants / 3,545 organizations / 14 clusters took 4-5+ minutes, exceeding Container Apps' default startup probe (~4 min). Container entered restart loop with 0% availability until `prewarm=False` was set.
4. **Wasted Redis bill** — Redis provisioned, but the cache that would benefit most is in-memory.
5. **Continuous refresh workaround still hits the database every cycle** — Redis caching would eliminate most of those queries.

### Suggested fix

Add `RedisBackedCache` in `dataflow/fabric/pipeline.py` with `get_cached`/`set_cached`/`invalidate` parity to the in-memory cache, using `redis.asyncio`:

```python
class RedisBackedCache:
    def __init__(self, redis_url: str, key_prefix: str = "dataflow:fabric:", ttl: int = 3600):
        import redis.asyncio as aioredis
        self._redis = aioredis.from_url(redis_url)
        self._prefix = key_prefix
        self._ttl = ttl

    async def get_cached(self, product_name, params=None): ...
    async def set_cached(self, product_name, data_bytes, content_hash, metadata, params=None): ...
    async def invalidate(self, product_name, params=None): ...
```

`PipelineExecutor.__init__` selects backend:

```python
if redis_url and not dev_mode:
    self._cache = RedisBackedCache(redis_url)
else:
    self._cache = InMemoryCache()
```

### Migration considerations

1. Async API change (in-memory is sync, Redis must be async) → entire cache layer becomes async
2. Serialization already handled (msgpack / json fallback)
3. Backward compat: `redis_url=None` or `dev_mode=True` keeps existing in-memory behavior
4. TTL strategy tied to product `staleness.max_age`
5. Key namespacing with DataFlow instance name

### Why this matters

The fabric design promises a shared, persistent cache layer between sources and consumers. With the current in-memory implementation, each replica is a sidecar with its own cache — defeating the purpose of a fabric in multi-replica deployments. The docstring at `pipeline.py:137` makes a promise the implementation does not keep.

For impact-verse, current workaround is a continuous refresh task re-fetching all 26 products every 60s. The DB still gets hit per replica; the only saving is user requests don't hit a cold cache.
