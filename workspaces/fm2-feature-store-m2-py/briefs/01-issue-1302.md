## Affected API

`kailash_ml.features` — ships `FeatureField`, `FeatureSchema`, `FeatureStore` (1.x read surface) but NOT the M2 consumer surfaces the integration spec describes.

## Missing surfaces (verified absent in source @ kailash-ml 2.0.1, 2026-06-12)

- `@feature` decorator (feature-computation pipeline consuming `dataflow.transform`)
- public `FeatureGroup` class (only the internal `SchemaFeatureGroup` read adapter exists — `features/_schema_feature_group.py`)
- `FeatureStore.materialize()` (canonical `features/store.py::FeatureStore` has no `materialize`)
- online-store adapter; M2 typed exceptions (`FeatureGroupNotFoundError`, `FeatureVersionNotFoundError`, `FeatureEvolutionError`, `OnlineStoreUnavailableError`, `CrossTenantReadError`, `FeatureVersionImmutableError`); FeatureStore-layer GDPR `erase_tenant`; feature evolution / version-immutability enforcement; DB-side windowed as-of

All catalogued as "Deferred to M2" in `specs/ml-feature-store.md` §11 and the `specs/dataflow-ml-integration.md` Wave-6.5 deferral note (§13).

## Expected vs actual

Expected (per `specs/dataflow-ml-integration.md` §2.5/§3): `kailash-ml` exposes `@feature` + `FeatureGroup` so those sections describe shipped behavior.
Actual: those surfaces are absent. The §2 `dataflow.ml_feature_source(...)` binding IS shipped and consumed via the internal `SchemaFeatureGroup` adapter (#1241); only the M2 author-side/online surfaces are unimplemented.

## Severity

MEDIUM — feature-store M2 capability gap. The 1.x read surface (`FeatureStore.get_features`, `ml_feature_source`) works; M2 authoring/online surfaces do not exist. Directly blocks the #693 spec graduation.
