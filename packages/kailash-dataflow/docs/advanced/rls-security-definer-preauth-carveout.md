# Row-Level Security & Pre-Auth Carve-Outs (`SECURITY DEFINER`)

DataFlow runs on PostgreSQL Row-Level Security (RLS) for multi-tenant
data isolation. RLS policies pivot on the `app.user_id` /
`row_security.tenant_id` GUC â€” a session-local setting installed by
the application's auth layer after a user logs in. Pre-auth read paths
(login lookup, password-reset lookup, invite-accept lookup) have a
chicken-and-egg problem: they NEED to read the `users` table to
authenticate, but the auth context that RLS pivots on isn't set yet.

The established mitigation is a narrow `SECURITY DEFINER` function
that runs as the function owner, bypasses the RLS policy on the user
table, and returns a minimum-disclosure row shape. DataFlow ships
`SecurityDefinerBuilder` to emit this pattern with all the load-bearing
invariants in one chain.

## RLS Posture On `@db.model`-Generated Tables

**Tables generated by `@db.model` ship WITHOUT RLS enabled by default.**
There is no implicit `USING(true)` policy. RLS is opt-in:

- A multi-tenant deployment configures
  `RowLevelSecurityProvider.create_tenant_policy()`, which emits
  `CREATE POLICY ... USING (tenant_id = current_setting(...))` â€”
  scoped to the bound tenant context, never `USING(true)`.
- A single-tenant deployment leaves RLS off and relies on application-
  level access controls (RBAC, query filters).

Operators who turn on RLS for cross-tenant isolation MUST also plan
the pre-auth read paths. That is what this guide covers.

## Why A Normal RLS Policy Doesn't Work For Login

Consider a `users` table with this RLS policy:

```sql
CREATE POLICY users_select_own_tenant ON users
  FOR SELECT
  USING (tenant_id = current_setting('row_security.tenant_id'));
```

When the login endpoint receives a `(email, password)` pair, it MUST
fetch the row to compare bcrypt hashes. But the session has not yet
issued `SET row_security.tenant_id = ...` because the user is not
authenticated. Postgres evaluates `current_setting('row_security.tenant_id')`
to NULL or empty, the policy returns 0 rows, and login fails for
every user.

The chicken-and-egg fix is a narrow `SECURITY DEFINER` function that:

1. Runs as the function's owner (a role with `BYPASSRLS` or that
   owns the table), bypassing the policy on the user table.
2. Accepts the lookup parameters (email, tenant_id) explicitly so the
   tenant scope is enforced inside the function body â€” not by RLS.
3. Returns the minimum row shape needed for bcrypt comparison â€”
   typically `id`, `email`, `password_hash`, `is_active` and nothing
   else.
4. `REVOKE ALL FROM PUBLIC` and `GRANT EXECUTE TO <authenticator_role>`
   so only the application role can call it.

## Four Load-Bearing Invariants

`SECURITY DEFINER` carve-outs have four invariants that are easy to
get wrong by hand:

1. **Pinned `search_path`** â€” defeats `CVE-2018-1058`-class function-
   resolution attacks where an attacker creates an operator in the
   `pg_temp` schema that shadows a built-in. The function MUST set
   `search_path = <app_schema>, pg_temp`.

2. **`REVOKE ALL FROM PUBLIC` + `GRANT EXECUTE TO <authenticator>`** â€”
   without this, every role inherits `EXECUTE` on the function. A
   compromised low-privilege role can then call the helper directly
   and read the password hash table.

3. **Timing-close dummy bcrypt compute when the user doesn't exist** â€”
   the caller's ~10-100ms bcrypt on a found row + 0ms on a missing
   row distinguishes "user exists" from "doesn't exist" via response
   latency. The helper's own 0-row response is constant-time; the
   CALLER is responsible for running a `bcrypt(candidate, DUMMY_HASH)`
   on the 0-row branch to close the timing side-channel. The emitted
   `COMMENT` reminds the operator.

4. **Multi-tenant filter inside the function body** â€” without
   `AND tenant_id = p_tenant_id` in the `WHERE`, a caller with a
   valid session in tenant A who passes
   `p_id = <user_id_from_tenant_B>` gets tenant B's row back,
   because `SECURITY DEFINER` bypasses RLS.

`SecurityDefinerBuilder` emits all four invariants together. Operators
cannot ship the carve-out without one of them silently missing.

## Usage

```python
from dataflow.migration import SecurityDefinerBuilder

stmts = (
    SecurityDefinerBuilder("resolve_user_by_email")
    .search_path("app")
    .authenticator_role("app_role")
    .user_table("users")
    .password_column("password_hash")
    .tenant_column("tenant_id")
    .active_column("is_active")
    .param("p_email", "text")
    .param("p_tenant_id", "bigint")
    .return_column("id", "bigint")
    .return_column("email", "text")
    .return_column("password_hash", "text")
    .return_column("is_active", "boolean")
    .build()
)

# stmts is a list of 4 strings â€” embed into your migration's upgrade()
for stmt in stmts:
    await dataflow.execute_raw(stmt)
```

The builder produces:

- `CREATE OR REPLACE FUNCTION app.resolve_user_by_email(p_email text, p_tenant_id bigint) RETURNS TABLE(...) LANGUAGE sql SECURITY DEFINER SET search_path = app, pg_temp STABLE STRICT AS $$ ... $$`
- `COMMENT ON FUNCTION app.resolve_user_by_email(text, bigint) IS '... CALLER MUST run a dummy bcrypt compare on 0-row results ...'`
- `REVOKE ALL ON FUNCTION app.resolve_user_by_email(text, bigint) FROM PUBLIC`
- `GRANT EXECUTE ON FUNCTION app.resolve_user_by_email(text, bigint) TO app_role`

## Caller Discipline (Timing-Close Compare)

The helper returns 0 rows in constant time whether the user exists or
not (single `SELECT ... LIMIT 1`). The application's bcrypt compare
on the 0-row branch is what closes the timing side-channel. The
canonical caller pattern:

```python
DUMMY_HASH = "$2b$12$dummyhashwithcorrectprefixlength......"  # known-bad bcrypt

async def login(email: str, tenant_id: int, candidate: str) -> User | None:
    rows = await db.execute_raw(
        "SELECT id, email, password_hash, is_active "
        "FROM app.resolve_user_by_email($1, $2)",
        email, tenant_id,
    )
    if not rows:
        # T7 â€” equalize timing by burning a bcrypt cycle anyway.
        bcrypt.checkpw(candidate.encode(), DUMMY_HASH.encode())
        return None
    row = rows[0]
    if not bcrypt.checkpw(candidate.encode(), row["password_hash"].encode()):
        return None
    return User(id=row["id"], email=row["email"])
```

## Required Setters

The builder fails fast at `build()` if any of these are unset:

- `function_name` â€” passed to the constructor or `function_name()`
- `search_path(schema)`
- `authenticator_role(role)`
- `user_table(table)`
- `password_column(column)`
- at least one `param(name, pg_type)`
- at least one `return_column(name, pg_type)`

If `tenant_column(column)` is set, `p_tenant_id` MUST also be declared
as a param â€” the builder validates the consistency and refuses to
build otherwise.

## Optional Setters

- `tenant_column(column)` â€” adds `AND <column> = p_tenant_id` to the
  WHERE; omit for single-tenant deployments.
- `active_column(column)` â€” adds `AND <column> = true` to the WHERE;
  omit when the schema has no boolean activity flag (e.g., text
  `status`, soft-delete `deleted_at IS NULL`).
- `primary_lookup_column(column)` â€” overrides the default derivation
  (strip `p_` from the first param name). Use when the param name and
  the schema column name diverge (e.g., `p_user_email` -> `email`).

## Allowed PostgreSQL Types

Param and return-column types MUST appear in `ALLOWED_PG_TYPES`:

```
bigint, int, integer, smallint, smallserial, serial, bigserial,
text, varchar, char, citext,
boolean, bool,
real, double precision, numeric, decimal,
timestamp, timestamptz, timestamp with time zone, timestamp without time zone,
date, time, interval,
uuid, inet, cidr, bytea, jsonb, json
```

This list is byte-shape identical with the kailash-rs Rust impl
(`crates/kailash-dataflow/src/migration/security_definer.rs`). User-
defined types and domain types are deliberately excluded â€” extend the
allowlist via a code review under
`rules/dataflow-identifier-safety.md`.

## Cross-SDK Byte-Shape Parity

The same builder chain produces byte-identical SQL across kailash-py
and kailash-rs. The shared fixture
`packages/kailash-dataflow/tests/fixtures/security_definer_vectors.json`
is the contract; both SDKs run regression tests against the fixture.

If you migrate a service from one SDK to the other, the migration's
emitted SQL is unchanged.

## Identifier Safety

Every user-supplied identifier (function name, schema, role, table,
column, parameter name) routes through
`PostgreSQLDialect.quote_identifier()` and is rejected if it fails
the `^[a-zA-Z_][a-zA-Z0-9_]*$` allowlist regex or exceeds
PostgreSQL's 63-char identifier limit.

Injection payloads like `'"; DROP TABLE users; --'` raise
`SecurityDefinerBuilderError` at `build()` time â€” the SQL is never
emitted. This is enforced by `rules/dataflow-identifier-safety.md`
MUST Rule 1.
