## 6. Cache Layer

### 6.1 Cache Backend Selection

Express uses a two-tier cache backend strategy:

1. **Redis** (preferred): When `redis_url` is provided (or `REDIS_URL` env var is set) and Redis is reachable, uses `AsyncRedisCacheAdapter`.
2. **In-Memory LRU** (fallback): When Redis is unavailable or not configured, uses `InMemoryCache` with bounded size.

Backend selection happens in `CacheBackend.auto_detect()` at Express initialization.

### 6.2 Cache Key Format

Generated by `CacheKeyGenerator`:

**For Express operations (no raw SQL):**

```
dataflow:v2:<model>:<operation>:<params_hash>
dataflow:v2:<tenant_id>:<model>:<operation>:<params_hash>   # multi-tenant
```

**For SQL queries:**

```
dataflow:<namespace>:<model>:v2:<query_hash>
```

**Keyspace version history:**

| Version | Shipped                          | Change                                                                                           |
| ------- | -------------------------------- | ------------------------------------------------------------------------------------------------ |
| `v1`    | ≤ kailash-dataflow 2.0.10        | Initial shape.                                                                                   |
| `v2`    | kailash-dataflow 2.0.11 (BP-049) | Classified PK values hashed BEFORE JSON serialization; cross-SDK parity with kailash-rs v3.19.0. |

Invalidation (`AsyncRedisCacheAdapter.invalidate_model`, `MemoryCacheAdapter.invalidate_model`) matches the version segment as a wildcard (`dataflow:v*:...`) so v1 legacy entries and current v2 entries are swept in one call.

The `params_hash` is an 8-character MD5 hex digest of the JSON-serialized parameters, where classified PKs are pre-hashed via `format_record_id_for_event`. The `query_hash` is a 16-character SHA-256 hex digest.

### 6.3 Cache TTL

- Default TTL: 300 seconds (5 minutes) for Express operations
- Configurable per-DataFlow instance via `cache_ttl` constructor parameter
- Overridable per-operation via `cache_ttl` parameter on `read()`, `list()`, `find_one()`, `count()`
- TTL of `0` disables caching entirely

### 6.4 Cache Invalidation

Write operations (`create`, `update`, `delete`, `upsert`, `bulk_*`) automatically invalidate all cache entries for the affected model. Invalidation is **model-scoped** -- all cached queries for that model are cleared.

In multi-tenant mode, invalidation is **tenant-scoped**: a write from tenant A only clears tenant A's cache entries for that model.

```python
await db.express.clear_cache()                # Clear all cache entries
await db.express.clear_cache(model="User")    # Clear only User cache entries
```

### 6.5 Cache Statistics

```python
stats = db.express.get_cache_stats()     # Sync: {"hits": int, "misses": int, "backend": str}
stats = await db.express.cache_stats()    # Async: adds "size" field
```

## 9. Dialect System

### 9.1 Architecture

```
SQLDialect (ABC)
├── PostgreSQLDialect
├── MySQLDialect
└── SQLiteDialect
```

Access via `DialectManager.get_dialect("postgresql")`.

### 9.2 `quote_identifier`

Every dialect validates and quotes SQL identifiers. The contract:

1. **Validates** against `^[a-zA-Z_][a-zA-Z0-9_]*$`
2. **Rejects** invalid input with `InvalidIdentifierError`
3. **Quotes** with dialect-appropriate character: `"` for PostgreSQL/SQLite, `` ` `` for MySQL
4. **Does not escape** embedded quote characters -- invalid inputs are rejected, not escaped

```python
dialect = DialectManager.get_dialect("postgresql")
safe = dialect.quote_identifier("users")     # -> '"users"'
dialect.quote_identifier("users; DROP")      # -> raises InvalidIdentifierError
```

### 9.3 Parameter Placeholders

| Dialect    | Placeholder         |
| ---------- | ------------------- |
| PostgreSQL | `$1`, `$2`, `$3`... |
| MySQL      | `%s`                |
| SQLite     | `?`                 |

### 9.4 Type Mapping Differences

| Generic     | PostgreSQL  | SQLite    | MySQL       |
| ----------- | ----------- | --------- | ----------- |
| `boolean`   | `BOOLEAN`   | `INTEGER` | `BOOLEAN`   |
| `blob`      | `BYTEA`     | `BLOB`    | `BLOB`      |
| `timestamp` | `TIMESTAMP` | `TEXT`    | `TIMESTAMP` |
| `json`      | `JSONB`     | `TEXT`    | `JSON`      |
| `bigint`    | `BIGINT`    | `INTEGER` | `BIGINT`    |

### 9.5 Feature Support Matrix

| Feature          | PostgreSQL | SQLite      | MySQL      |
| ---------------- | ---------- | ----------- | ---------- |
| JSON             | Yes        | Yes (3.38+) | Yes (5.7+) |
| Arrays           | Yes        | No          | No         |
| Window functions | Yes        | Yes (3.25+) | Yes (8.0+) |
| CTE              | Yes        | Yes         | Yes (8.0+) |
| Upsert           | Yes        | Yes         | Yes        |
| RETURNING        | Yes        | Yes (3.35+) | No         |
| Full outer join  | Yes        | No          | No         |
| Lateral join     | Yes        | No          | Yes (8.0+) |
| Fulltext search  | Yes        | Yes         | Yes        |

### 9.6 Upsert SQL Generation

- **PostgreSQL/SQLite**: `ON CONFLICT (cols) DO UPDATE SET col = EXCLUDED.col`
- **MySQL**: `ON DUPLICATE KEY UPDATE col = VALUES(col)`

### 9.7 Dialect Parameter Conversion

```python
query, params = DialectManager.convert_query_parameters(
    "SELECT * FROM users WHERE id = $1",
    [42],
    source_dialect="postgresql",
    target_dialect="sqlite",
)
# -> ("SELECT * FROM users WHERE id = ?", [42])
```

## 10. Record ID Coercion

Express API methods (`read`, `update`, `delete`) accept `Union[str, int]` for the `id` parameter. DataFlow coerces the value to match the model's declared primary key type.

### 10.1 Coercion Rules

The `_coerce_record_id` function in `core/nodes.py`:

1. If `id_value` is `None`, returns `None`
2. Extracts the `id` field type from the model's field registry
3. Normalizes the type annotation (strips `Optional[...]`, `Union[..., None]`)
4. If PK type is `int`: converts string IDs to `int` (e.g., `"42"` -> `42`). Falls back to original value on `ValueError`.
5. If PK type is `str`: converts non-string IDs to `str` (e.g., `42` -> `"42"`).
6. For other types (UUID, custom): preserves the value as-is.

### 10.2 Where Coercion Happens

Coercion is applied in the generated `ReadNode`, `UpdateNode`, and `DeleteNode` classes before SQL parameter binding. This ensures integer IDs work correctly on PostgreSQL (which rejects string parameters for integer columns).

## 12. Transaction Support

### 12.1 Transaction Manager

Accessed via `db.transactions`:

```python
async with db.transactions.begin() as txn:
    await db.express.create("User", {"id": "u1", "name": "Alice"})
    await db.express.create("Profile", {"user_id": "u1", "bio": "..."})
    # Both committed atomically on clean exit
```

### 12.2 Isolation Levels

```python
async with db.transactions.begin(isolation_level="SERIALIZABLE") as txn:
    ...
```

Supported levels:

- `"READ COMMITTED"` (default)
- `"REPEATABLE READ"`
- `"SERIALIZABLE"`

### 12.3 Nested Transactions (SAVEPOINTs)

Nested `begin()` calls automatically create SAVEPOINTs:

```python
async with db.transactions.begin() as outer:
    await db.express.create("User", {"id": "u1", "name": "Alice"})

    try:
        async with db.transactions.begin() as inner:
            # This is a SAVEPOINT
            await db.express.create("Profile", {"user_id": "u1"})
            raise ValueError("oops")
            # SAVEPOINT rolled back
    except ValueError:
        pass
    # Outer transaction continues -- User is committed, Profile is not
```

### 12.4 Automatic Rollback

Exceptions within the transaction body trigger automatic rollback. The exception is re-raised after rollback.

### 12.5 Transaction Statistics

```python
stats = db.transactions.get_stats()
# {"total_started": int, "total_committed": int, "total_rolled_back": int}
```

## 13. Connection Pooling

### 13.1 Pool Configuration

Pool size is resolved through `DatabaseConfig.get_pool_size()`. There is a single source of truth -- no competing defaults.

| Parameter                            | Default                  | Purpose                               |
| ------------------------------------ | ------------------------ | ------------------------------------- |
| `pool_size`                          | Via `get_pool_size()`    | Base connections in pool              |
| `pool_max_overflow` / `max_overflow` | `max(2, pool_size // 2)` | Maximum overflow connections          |
| `pool_recycle`                       | `3600`                   | Connection recycle interval (seconds) |

### 13.2 Read/Write Splitting (TSG-105)

```python
db = DataFlow(
    "postgresql://primary:5432/app",
    read_url="postgresql://replica:5432/app",
    read_pool_size=10,
)
```

Read operations use the read replica by default. Pass `use_primary=True` to force reads from the primary.

### 13.3 SQLite Connection Management

SQLite connections are managed through `AsyncSQLitePool`:

- Uses WAL mode, busy_timeout, synchronous, cache_size, foreign_keys PRAGMAs
- URI shared-cache for `:memory:` databases: `file:memdb_NAME?mode=memory&cache=shared`
- Separate read/write connection acquisition: `acquire_read()` / `acquire_write()`
- Bounded concurrency via `max_read_connections`
