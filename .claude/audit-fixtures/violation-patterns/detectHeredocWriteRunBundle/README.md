# detectHeredocWriteRunBundle audit fixtures

Per `rules/cc-artifacts.md` Rule 9 + `rules/hook-output-discipline.md` MUST-4. Each fixture pins one scope-restriction predicate `detectHeredocWriteRunBundle(command, pathRx)` relies on. Inputs are bash command strings (paired with a path regex the caller passes — the trust-posture `STATE_PATH_RX`); expected outputs are the JSON the detector returns — `null` (no flag) or the structural detection object `{ "layer": 1, "kind": "heredoc-write-run-bundle" }`.

## What the detector is (#764 item 3)

`splitShellSegments` is NOT heredoc-aware, so a bundled `cat > s.cjs <<EOF … fs.writeFileSync("<protected>") … EOF && node s.cjs` has its heredoc body's internal `;` fracture the interpreter-lead (`node s.cjs`) from the protected-path write — no per-segment `detectStateFileMutation` call sees both. `detectHeredocWriteRunBundle` runs on the WHOLE command and flags ONLY the STRUCTURAL write→execute conjunction:

- **(a)** a heredoc whose **redirect-target** script file's BODY contains a protected **path LITERAL** anywhere (matched by `pathRx.test(body)` — NOT a write-verb allowlist, NOT a prefix-anchored write-call regex), AND
- **(b)** that SAME written script is SUBSEQUENTLY EXECUTED by an interpreter in the positive allowlist `RUN_INTERPRETER_RX` (`node`/`nodejs`/`python`/`ruby`/`perl`/`bash`/`sh`/`zsh`/`deno`/`bun`/`php`/`tsx`/`Rscript`/`lua`/…) via `<interp> <a-written-path>` (token-identity match on any collected write target, `./`-normalized; leading `VAR=val` prefixes stripped).

`severity:block` is justified per `hook-output-discipline.md` MUST-2 because the signal is STRUCTURAL (a real write-a-script-touching-a-protected-path AND run-it dataflow — one indirection deeper than the Layer-1 heredoc-to-protected-path redirect the guard already blocks), NOT a lexical form. Surface rewrites of the **script name, path prefix, quoting, write verb, delimiter shape** do not evade the detector's coverage: (b) matches the written-path TOKEN identity; (a) matches the protected-path SUBSTRING in the body. The **write-surface and interpreter allowlists are POSITIVE** (per `cc-artifacts.md` Rule 10) — the write surface covers `>`/`>>` + `tee`/`sponge`/`cp`/`install` stdin-sinks + `dd of=`, and the interpreter set covers the standard shells + common script interpreters (`node`/`python`/`ruby`/`perl`/`deno`/`bun`/`php`/`tsx`/`Rscript`/`lua`/…); a write-verb or interpreter OUTSIDE either allowlist (a `patch` write, an exotic interpreter, a `mv`-rename dataflow hop) is a documented residual, forever-defended by the signed-fold / fail-closed-to-L1 layer, not this interceptor. The delimiter is parsed with bash quote-removal + escape (numeric `<<9`, quoted `<<"9"`, hyphenated `<<'a-b'`, partially-quoted `<<E"O"F`, ANSI-C/locale `<<$'EOF'` / `<<$"EOF"` — all close on their dequoted terminator), while the `<<<` here-string form is STRUCTURALLY excluded (a char-scan, not a regex lookahead a shifted re-match could slip past). A heredoc opener is committed ONLY when its close line exists downstream, so a decoy `<<WORD` (an arithmetic `1<<4`, a quoted/commented `<<X`) with no close is IGNORED rather than swallowing the RUN line into a phantom body — an unclosed heredoc can only ADD to the scanned surface, never hide from it (fail-toward-more-scanning). `tee` collects EVERY file operand (`tee a b`, `|& tee`, `|tee`, backslash-in-quote args) so executing ANY written file is caught. A fail-closed **BACKSTOP** covers any residual divergence between this hand-written parser's terminator/close derivation and bash's (an ANSI-C `<<$'EO\x46'` escape, a same-line arithmetic `1<<4` opener with a seeded close, a `\r`-seeded early close): such a divergence can only SPILL the real body + RUN line into `structural`, so if the protected path appears on a structural line AND a structural-written script is executed on a structural line, it flags — the per-body check's `pathRx.test(hd.body)` gate runs before the structural scan and cannot see a truncated body, but the backstop can. **Two residual classes remain** (SAME class as `state-file-write-guard.md` Rule 5 residuals (a)/(d) — not closable here without in-hook shell expansion, which `hook-output-discipline.md` MUST-3 forbids): **(i)** VAR-INDIRECT exec where the write and run use DIFFERENT tokens resolving to the same file (`cat >/tmp/s.cjs <<E…E; T=/tmp/s.cjs; node "$T"` — a SHARED var token still matches, since identity holds pre-expansion); **(ii)** a RUN segment NOT recognized as interpreter-led after `VAR=` stripping — one root cause spanning a non-`VAR=` command prefix (`sudo`/`env`/`nice`/subshell), direct shebang/executable-bit invocation (`chmod +x s && ./s`), and shell sourcing (`source s` / `. s`). The forever-defense for both is the signed-fold / fail-closed-to-L1 integrity layer, not this interceptor. Doc/rule/test authoring WRITES a file but does NOT execute it, so (b) fails structurally — no false-block (the failure attempt-1's lexical heredoc-body write-call regex produced).

The git-commit exception needs no special skip: a commit-MESSAGE heredoc has no redirect-target script (its body is git's STDIN) OR its target file is consumed by `git`, never an interpreter — so (b) fails; a heredoc chained AFTER `git commit` is analyzed on its own write→exec merits (closes attempt-1's per-line HIGH-2 evasion). Delimiter-close is STRUCTURAL: plain `<<EOF` closes ONLY on an exactly-`EOF` line (no leading whitespace); `<<-EOF` strips leading TABS only (never spaces).
